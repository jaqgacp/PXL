# Collection Monitoring

## Module Overview
The Collection Monitoring module provides a dedicated workspace for Credit and Collection teams. Unlike the AR Aging report which is a static financial snapshot, this module is operational. It tracks Promises to Pay (PTP), Post-Dated Checks (PDC), and follow-up activities against outstanding Sales Invoices. It helps businesses proactively manage their receivables to avoid cash flow bottlenecks.

### Document Flow & Data Inheritance
* **Source:** Links directly to open (unpaid) **Sales Invoices**.
* **Inheritance:** Does not inherit lines, but attaches activity records (notes, calls, dates) to specific invoice headers.

## Dashboard UI
An interactive, task-oriented list of overdue accounts needing follow-up.

### The Action Bar
* **Search Bar:** Search by Customer or Invoice Number.
* **Filter:** Dropdown for Follow-up Status (Pending Call, PTP Secured, PDC Received, Escalated).
* **Date Filter:** Filter by Next Action Date.
* **Export Button:** Download collection task list to Excel.
* **Log Activity Button:** Record a call or update a PTP date.

### The Data List
Default columns displayed:
* Customer Name
* Invoice Number
* Days Overdue
* Outstanding Balance
* Last Contact Date
* Promise to Pay (PTP) Date
* Next Action Date
* Status

## Data Fields (Collection Activity Entry)
When a user clicks "Log Activity" on an invoice, a modal appears to capture the follow-up details.

| Field Name | Component Type | Required? | Behavior |
| :--- | :--- | :--- | :--- |
| Invoice | Read-only | Yes | The invoice being followed up on. |
| Contact Person | Text Input | Yes | Who the collector spoke to. |
| Activity Type | Dropdown | Yes | Call, Email, Field Visit. |
| Result/Status | Dropdown | Yes | PTP Secured, PDC Received, Unreachable, Disputed. |
| PTP Date | Date Picker | Conditional | Date the customer promised to pay. |
| Next Action Date | Date Picker | Yes | When the collector should follow up again. |
| Notes | Text Area | Yes | Detailed summary of the conversation. |

## Supabase Database Architecture


**Critical Database Rule for Lovable:**
- Every transactional table MUST include a `company_id` to ensure tenant isolation.
- Composite unique constraints MUST be established (e.g., `company_id` + document number) to prevent duplicate documents within the same company.

To track this without cluttering the main `sales_invoices` table, we use a separate activity log table.

### Table 1: `collection_activities`
| Column Name | Data Type | Rules | What it stores |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System ID for the activity log. |
| `company_id` | UUID | Required, Foreign Key | Links to `companies.id`. |
| `invoice_id` | UUID | Required, Foreign Key | Links to `sales_invoices.id`. |
| `customer_id` | UUID | Required, Foreign Key | Links to `customers.id`. |
| `contact_person` | Text | Required | Name of customer representative. |
| `activity_type` | Text | Required | `call`, `email`, `visit`. |
| `result_status` | Text | Required | `ptp`, `pdc`, `unreachable`, `disputed`. |
| `ptp_date` | Date | Nullable | Expected date of payment. |
| `next_action_date`| Date | Required | Reminder date for collector. |
| `notes` | Text | Required | Conversation details. |
| `created_by` | UUID | Required, Foreign Key | Links to `users.id` (The Collector). |
| `updated_by` | UUID | Nullable, Foreign Key | Links to `users.id`. |
| `created_at` | Timestamp | Auto | Date logged. |
| `updated_at` | Timestamp | Auto | Date record was last edited. |
