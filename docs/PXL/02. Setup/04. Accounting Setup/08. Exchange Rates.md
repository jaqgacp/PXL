# PXL ERP Blueprint: Exchange Rates (Multi-Currency Framework)

## Module Overview
The Exchange Rates module provides the daily foreign exchange rate data required for multi-currency transaction processing. PXL ERP operates with a **functional currency** of PHP (Philippine Peso) — all financial statements are presented in PHP. When transactions occur in a foreign currency (USD, EUR, JPY, etc.), the system records both the **transaction currency amount** and the **functional currency equivalent** (using the exchange rate on the transaction date).

**Activation Condition:** Visible and required only when `sys_feature_enablement.multi_currency = true`.

### Key Concepts

| Term | Definition |
| :--- | :--- |
| **Functional Currency** | Always PHP. The currency in which financial statements are prepared. |
| **Transaction Currency** | The currency in which a specific transaction occurred (USD, EUR, etc.). |
| **Exchange Rate** | The rate used to convert transaction currency to functional currency on the transaction date. |
| **Forex Gain/Loss** | The difference between the exchange rate on the transaction date vs. the settlement date (or period-end revaluation date). |
| **Month-End Revaluation** | At period close, open foreign-currency balances (AR, AP, bank balances) are revalued to the closing exchange rate. The difference is posted to a Forex Gain/Loss account. |

### Table 1: `exchange_rates`

**Critical Database Rules:**
1. `UNIQUE(company_id, rate_date, currency_code, rate_type)` — one rate per currency per type per day per company.
2. Rates are immutable once posted transactions reference them. The system blocks deletion of rates that have been used in posted documents.
3. If no rate exists for a transaction date, the system falls back to the most recent rate before that date (never leaves the transaction without a rate).

| Column Name | Data Type | Rules | What it stores |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System ID. |
| `company_id` | UUID | Required, Foreign Key | Links to `companies.id`. |
| `rate_date` | Date | Required | The date this rate is effective. |
| `currency_code` | Text | Required | ISO 4217 code (e.g., `USD`). |
| `rate_type` | Text | Required | ENUM: `bsp_reference`, `buy`, `sell`. |
| `rate` | Numeric(15,6) | Required | PHP per 1 unit of foreign currency (e.g., 57.250000 for USD). |
| `source` | Text | Required | ENUM: `manual`, `bsp_import`, `api_feed`. |
| `created_by` | UUID | Required, Foreign Key | Links to `users.id`. |
| `created_at` | Timestamp | Auto | Creation timestamp. |
| `updated_at` | Timestamp | Auto | Last update timestamp. |

**Composite Unique Constraint:** `UNIQUE(company_id, rate_date, currency_code, rate_type)`

## How Exchange Rates Are Used in Transactions

When a Sales Invoice, Purchase Invoice, or Journal Entry is created in a foreign currency:

1. **User selects the transaction currency** (e.g., USD).
2. The system **auto-fetches the exchange rate** for the transaction date from `exchange_rates` (using `rate_type = 'bsp_reference'` as default).
3. The document stores both:
   - `transaction_currency = 'USD'`
   - `transaction_amount = 10,000.00` (the foreign currency amount)
   - `exchange_rate = 57.25`
   - `functional_amount = 572,500.00` (PHP equivalent posted to GL)
4. The GL entry stores `functional_debit` or `functional_credit` in PHP.

**Required additions to transactional tables (Sales Invoice, Purchase Invoice, Journal Entries):**

| Column | Data Type | What it stores |
| :--- | :--- | :--- |
| `transaction_currency` | Text, Default `PHP` | ISO 4217 code of the transaction currency. |
| `exchange_rate` | Numeric(15,6), Default 1.000000 | Rate on the document date. `1.0` if transaction currency is PHP. |
| `functional_currency` | Text, Always `PHP` | The reporting currency. Constant for all Philippine entities. |

## Month-End Forex Revaluation

At the end of each accounting period, open foreign-currency AR, AP, and bank balances must be revalued to the period-closing exchange rate. The difference between the **original transaction rate** and the **closing rate** is a Forex Gain or Loss.

### Revaluation Procedure

1. **Select period-end date** — the system fetches the closing exchange rate from `exchange_rates`.
2. **Query all open foreign-currency balances:**
   - Open Sales Invoices in foreign currency (AR)
   - Open Purchase Invoices in foreign currency (AP)
   - Bank account balances in foreign currency
3. **Compute revaluation difference:**
   - `Revaluation Amount = Open Balance (foreign) × (Closing Rate - Original Rate)`
4. **Post revaluation journal entry:**
   - If Closing Rate > Original Rate on AR: Dr. AR Account / Cr. Forex Gain
   - If Closing Rate < Original Rate on AR: Dr. Forex Loss / Cr. AR Account
   - Opposite logic for AP
5. **Reversing entry** is auto-created for the first day of the next period (revaluation is temporary).

### Revaluation Journal Entry Specification

```
DR / CR  | Account               | Source
---------|-----------------------|----------------------------
Dr       | AR / AP / Bank GL     | Revalued balance (if gain)
Cr       | Forex Gain/Loss       | Difference amount
source_document_type = 'forex_revaluation'
source_document_id  = [revaluation_run_id]
```

## Row-Level Security

```sql
-- Tenant isolation on exchange rates
CREATE POLICY "tenant_isolation_exchange_rates"
ON exchange_rates
FOR ALL
USING (company_id = (auth.jwt() ->> 'company_id')::UUID);

-- Exchange rates referenced by posted documents cannot be deleted
CREATE POLICY "block_delete_used_rates"
ON exchange_rates
FOR DELETE
USING (false);
```
