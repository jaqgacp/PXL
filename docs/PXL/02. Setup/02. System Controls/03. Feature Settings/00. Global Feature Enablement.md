# PXL ERP Blueprint: Global Feature Enablement

## Module Overview
The Global Feature Enablement module controls **progressive UI disclosure** — the principle that modules, screens, and features are only visible when they have been explicitly activated for a company. A new company that has not enabled the Fixed Assets module should not see the Fixed Assets menu. A company that is not VAT-registered should not see VAT Returns.

This is the master on/off switch for every major module in PXL ERP. The app reads the `sys_feature_enablement` table on app load and hides or shows entire menu sections accordingly. This prevents UI clutter, prevents users from accidentally entering data in modules they haven't configured, and allows the SaaS platform to enforce module-based billing tiers.

**Principle 13 (Low Maintenance):** Never hardcode module visibility in frontend code. All module visibility is driven by `sys_feature_enablement` records. When a new module is added to PXL ERP, it gets a row in the pre-seeded `ref_feature_definitions` table, and the frontend automatically respects the toggle.

## Dashboard UI

This module is only visible to users with the `system_admin` or `super_admin` role.

### The Action Bar
* **Search Bar:** Find a feature by name or module.
* **Filter:** By Module (Sales, Purchasing, Accounting, etc.), Status (Enabled/Disabled).
* **Export Button:** Export feature enablement report for support purposes.

### The Data List
Displays all available features with their current status for this company.

* Feature Name
* Module Category
* Description
* Status Toggle (Enabled / Disabled)
* Enabled By (User Name)
* Enabled At (Timestamp)

**Pagination Mandate:** Server-side pagination via Supabase `.range(from, to)`. Default page size: 50.

| `petty_cash` | Banking & Treasury | Petty Cash Funds, Vouchers, Replenishments. |
| `banking_module` | Banking & Treasury | Bank Accounts, Reconciliation, Fund Transfers. |
| `vat_compliance` | Compliance | VAT Returns (2550M/2550Q), SLSP, RELIEF, Input VAT tracking. |
| `ewt_compliance` | Compliance | EWT Returns (0619-E/1601EQ), 2307 Certificates, QAP. |
| `fwt_compliance` | Compliance | FWT Returns (0619-F/1601FQ), 2306 Certificates. |

## Supabase Database Architecture

### Reference Table: `ref_feature_definitions`
Pre-seeded master list of all available features. Never user-edited.

| Column Name | Data Type | Rules | What it stores |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System ID. |
| `feature_key` | Text | Required, Unique | Machine-readable key used in frontend conditionals. |
| `feature_name` | Text | Required | Human-readable feature name. |
| `module_category` | Text | Required | Module grouping for display (e.g., `Sales`, `Compliance`). |
| `description` | Text | Required | What this feature enables. |
| `always_enabled` | Boolean | Default: `false` | If `true`, cannot be disabled (e.g., `core_accounting`). |
| `is_active` | Boolean | Default: `true` | Whether this feature exists in the current product version. |
| `sort_order` | Integer | Required | Display order in the feature list. |
| `created_at` | Timestamp | Auto | Creation timestamp. |

---

### Table 1: `sys_feature_enablement`
One row per company per feature. Controls what each company can see and use.

**Critical Database Rules:**
1. `UNIQUE(company_id, feature_definition_id)` — one enablement record per feature per company.
2. Features with `ref_feature_definitions.always_enabled = true` have `is_enabled = true` and are protected from UPDATE by RLS.
3. On company creation, the system seeds one `sys_feature_enablement` row per feature definition, all set to `is_enabled = false` (except `always_enabled` features).

| Column Name | Data Type | Rules | What it stores |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System ID. |
| `company_id` | UUID | Required, Foreign Key | Links to `companies.id`. |
| `feature_definition_id` | UUID | Required, Foreign Key | Links to `ref_feature_definitions.id`. |
| `is_enabled` | Boolean | Required, Default: `false` | Whether this feature is active for the company. |
| `enabled_by` | UUID | Nullable, Foreign Key | Links to `users.id`. Who activated this feature. |
| `enabled_at` | Timestamp | Nullable | When this feature was activated. |
| `disabled_by` | UUID | Nullable, Foreign Key | Links to `users.id`. Who deactivated this feature. |
| `disabled_at` | Timestamp | Nullable | When this feature was deactivated. |
| `created_at` | Timestamp | Auto | Creation timestamp. |
| `updated_at` | Timestamp | Auto | Last update timestamp. |

**Composite Unique Constraint:** `UNIQUE(company_id, feature_definition_id)`

## Frontend Usage Pattern

```typescript
// On app load, fetch enabled features for current company
const { data: features } = await supabase
  .from('sys_feature_enablement')
  .select('feature_key:ref_feature_definitions(feature_key), is_enabled')
  .eq('company_id', currentCompanyId)
  .eq('is_enabled', true);

const enabledKeys = new Set(features.map(f => f.feature_key));

// In navigation component — only show if enabled:
// enabledKeys.has('fixed_assets') ? <FixedAssetsMenu /> : null
// enabledKeys.has('vat_compliance') ? <VATMenu /> : null
```

## Row-Level Security

```sql
-- Always-enabled features cannot be toggled off
CREATE POLICY "protect_always_enabled_features"
ON sys_feature_enablement
FOR UPDATE
USING (
  NOT EXISTS (
    SELECT 1 FROM ref_feature_definitions rfd
    WHERE rfd.id = feature_definition_id
      AND rfd.always_enabled = true
  )
);

-- Only system_admin or super_admin can modify feature toggles
CREATE POLICY "admin_only_feature_enablement"
ON sys_feature_enablement
FOR UPDATE
USING (
  company_id = (auth.jwt() ->> 'company_id')::UUID
  AND (auth.jwt() ->> 'role') IN ('system_admin', 'super_admin')
);

-- Tenant isolation on reads
CREATE POLICY "tenant_isolation_feature_enablement"
ON sys_feature_enablement
FOR SELECT
USING (company_id = (auth.jwt() ->> 'company_id')::UUID);
```
