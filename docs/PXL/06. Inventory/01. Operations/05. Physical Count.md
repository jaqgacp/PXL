# PXL ERP Blueprint: Physical Count

## Module Overview
The Physical Count module facilitates the periodic auditing of actual warehouse stock against system records. In the Philippines, year-end physical inventory counts are standard accounting practices required for accurate financial statements and BIR compliance. This module captures the "freeze" of system quantities, records the counted actuals, and upon approval, automatically generates a `Stock Adjustment` transaction to true-up the ledger.

## What You Will See on the Screen (The Dashboard)

### The Action Bar (Top of the page):
* **Search Bar:** Search by Count Reference Number.
* **Filter:** Filter by Status (Draft, In Progress, Completed, Voided).
* **Customize View:** Choose columns to display.
* **Export Button:** Download list to Excel or CSV.
* **Create New Physical Count Button:** Opens the form to initialize a count.

### The Data List (The main table - must be customizable):
Columns:
* Count Reference Number
* Date
* Warehouse
* Total Items Counted
* Variance Value
* Status

## The Supabase Database Architecture (For Lovable)

### Table 1: `physical_counts` (Header)
| Column Name | Data Type | Rules | What it stores |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System ID. |
| `count_number` | Text | Required, Unique | E.g., "PC-2026-001". |
| `count_date` | Date | Required | Date of the count. |
| `warehouse_id` | UUID | Required, Foreign Key | Links to `warehouses.id`. |
| `counter_name` | Text | Required | Person conducting count. |
| `remarks` | Text | Nullable | Explanatory notes. |
| `status` | Text | Required | `draft`, `in_progress`, `completed`, `voided`. |
| `created_by` | UUID | Required, Foreign Key | Links to `users.id`. |
| `updated_by` | UUID | Nullable, Foreign Key | Links to `users.id`. |
| `created_at` | Timestamp | Auto | Date record was created. |
| `updated_at` | Timestamp | Auto | Date record was last edited. |

### Table 2: `physical_count_items` (Line Items)
| Column Name | Data Type | Rules | What it stores |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System ID. |
| `count_id` | UUID | Required, Foreign Key | Links to `physical_counts.id`. |
| `item_id` | UUID | Required, Foreign Key | Links to `items.id`. |
| `system_qty` | Numeric(15,4) | Required | Snapshot of stock at count time. |
| `counted_qty`| Numeric(15,4) | Required | Actual counted quantity. |
| `variance_qty`| Numeric(15,4) | Required | `counted_qty - system_qty`. |
| `created_by` | UUID | Required, Foreign Key | Links to `users.id`. |
| `updated_by` | UUID | Nullable, Foreign Key | Links to `users.id`. |
| `created_at` | Timestamp | Auto | Date record was created. |
| `updated_at` | Timestamp | Auto | Date record was last edited. |

**Crucial Logic:** Upon completion (Status = `completed`), the system MUST automatically generate a `Stock Adjustment` record for any items with a `variance_qty` != 0.

**Critical UX/GL Rule:** The system MUST auto-fetch the Item's default Revenue, Expense, COGS, and Inventory GL accounts in the background to auto-generate the Journal Entry. Encoders MUST NEVER manually select a GL account for standard items.
