# PXL ERP Blueprint: Inventory Dashboard

## Module Overview
The Inventory Dashboard provides a bird's-eye view of the organization's current stock health. In the Philippine business context, maintaining optimal inventory levels prevents capital tie-up and stockouts. This dashboard aggregates data from the `inventory_movements` ledger, presenting real-time valuations, low-stock warnings, and warehouse distribution without requiring manual report generation.

## What You Will See on the Screen (The Dashboard)

### The Action Bar (Top of the page):
* **Global Date Filter:** Date Picker to filter metrics up to a specific 'As Of' date.
* **Warehouse Filter:** Dropdown (Links to Warehouses) to filter metrics by a specific branch/warehouse.
* **Export Button:** Download the summary widgets data to PDF or Excel.

### The Widgets & Data Visualizations:
1. **Total Inventory Value:** A large KPI card showing the aggregate monetary value of all items on hand (Quantity × Unit Cost).
2. **Low Stock Alerts:** A list or table highlighting items that have fallen below their configured `reorder_level` in the Items master.
3. **Top Moving Items:** A bar chart showing items with the highest outbound volume (derived from Sales Delivery transactions).
4. **Slow Moving Inventory:** A list identifying items that haven't had an outbound movement in the last 90+ days.
5. **Stock Distribution by Warehouse:** A pie chart showing the percentage of total inventory value held in each warehouse.

---

## What Information We Need to Capture (The Data Fields)
*Note: The Dashboard is purely for data visualization. It does not capture new data. It queries existing tables.*

### Inherited Data Sources:
* **Item Master (`items`):** Provides the Item Name, Category, and Reorder Level.
* **Warehouse Master (`warehouses`):** Provides the location contexts.
* **Inventory Movements (`inventory_movements`):** Provides real-time aggregation of quantities (SUM of `quantity`) and valuations to derive exact balances at any point in time.

---

## The Supabase Database Architecture
*No distinct tables are created for the Dashboard itself.*
The system will rely on SQL Views or RPC (Remote Procedure Call) functions in Supabase to calculate these metrics efficiently in real-time, relying strictly on the `inventory_movements`, `items`, and `warehouses` tables. Audit trails (`created_by`, `updated_by`, etc.) are inherited from those underlying source tables.


**Critical UX/GL Rule:** The system MUST auto-fetch the Item's default Revenue, Expense, COGS, and Inventory GL accounts in the background to auto-generate the Journal Entry. Encoders MUST NEVER manually select a GL account for standard items.
