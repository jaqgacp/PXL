# PXL ERP Blueprint: Control Account Reconciliation

## Module Overview
The Control Account Reconciliation is a critical internal audit and period-end closing tool. The balance of a GL Control Account (e.g., Accounts Receivable — `1200`) MUST perfectly match the sum of all its underlying Subsidiary Ledger entries (the sum of balances for Customer A, Customer B, Customer C, etc.). Discrepancies occur when manual journal entries are posted directly to the control account without tagging a specific `contact_id`, violating subledger integrity. This module automatically compares the GL balance against the aggregated subledger balances and flags variances for immediate correction.

**This is a computed view, not a data-entry module.** It stores no user input — it queries live data and presents the comparison.

---

## Dashboard UI

A high-level dashboard used by the Chief Accountant or Controller during month-end close.

### The Action Bar
* **As-Of Date Picker:** Define the cutoff date for the reconciliation.
* **Refresh Button:** Recalculates all balances immediately (triggers the reconciliation queries).
* **Export Button:** Export the reconciliation summary to Excel as audit evidence.

### The Data List
| Column | What It Shows |
| :--- | :--- |
| Control Account | The GL account name (e.g., Trade Receivables, Trade Payables, Inventory) |
| GL Balance | The sum of all debit/credit postings to this `account_id` in `general_ledger_entries` |
| Subledger Total | The aggregate of individual contact/item balances linked to this account |
| Variance | `GL Balance - Subledger Total` |
| Status | **Matched** (Green) when Variance = 0. **Variance** (Red) when there is a discrepancy. |
| Action | "View Details" — opens the drill-down view |

### Drill-Down View (Variance Investigation)
When "View Details" is clicked on a row with a variance, the system shows **Orphan Entries** — transactions posted to the Control Account that are missing a `contact_id` or `item_id`. This pinpoints exactly which manual JE caused the discrepancy.

---

## Computed Fields

This module presents computed results. No user-entered data is stored.

| Computed Field | Calculation |
| :--- | :--- |
| GL Balance | `SUM(debit - credit) FROM general_ledger_entries WHERE account_id = [control_account_id] AND posting_date <= :as_of_date` |
| Subledger Total | `SUM(balance) FROM ar_subledger WHERE account_id = [control_account_id] AND date <= :as_of_date` (or AP/Inventory equivalent) |
| Variance | `GL Balance - Subledger Total` |

## Database Architecture

### No New Tables Required
This module uses aggregate queries on existing tables. The `chart_of_accounts` table must have one additional flag:

**Addition to `chart_of_accounts`:**
| Column | Data Type | Purpose |
| :--- | :--- | :--- |
| `is_control_account` | Boolean, Default `false` | Marks this GL account as a control account that requires subledger matching |
| `subledger_type` | Text, Nullable | ENUM: `ar`, `ap`, `inventory`, `employee_advances`, `other`. Determines which subledger to compare against. |

### Posting Validation Rule (Enforced at GL Level)
If `chart_of_accounts.is_control_account = true`, the system MUST reject any direct posting to `general_ledger_entries` where `contact_id IS NULL`. This is enforced via a PostgreSQL trigger on `general_ledger_entries`:

```sql
CREATE OR REPLACE FUNCTION fn_validate_control_account_posting()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
DECLARE
  v_is_control BOOLEAN;
BEGIN
  SELECT is_control_account INTO v_is_control
  FROM chart_of_accounts
  WHERE id = NEW.account_id;

  IF v_is_control = true AND NEW.contact_id IS NULL THEN
    RAISE EXCEPTION 'Direct posting to control account % requires a contact_id. Use source document posting instead.', NEW.account_id;
  END IF;

  RETURN NEW;
END;
$$;

CREATE TRIGGER trg_validate_control_account
BEFORE INSERT ON general_ledger_entries
FOR EACH ROW EXECUTE FUNCTION fn_validate_control_account_posting();
```

### Reconciliation Query (Safety Net)
Even with the trigger in place, run this query during month-end to confirm zero orphans:

```sql
-- Find orphan entries causing control account variances
SELECT
  coa.account_code,
  coa.account_name,
  gle.id,
  gle.posting_date,
  gle.debit,
  gle.credit,
  gle.description
FROM general_ledger_entries gle
JOIN chart_of_accounts coa ON coa.id = gle.account_id
WHERE coa.is_control_account = true
  AND gle.contact_id IS NULL
  AND gle.company_id = :company_id
  AND gle.posting_date <= :as_of_date
ORDER BY gle.posting_date DESC;
```

### Control Account Balance vs Subledger Comparison Query

```sql
-- AR Control Account Reconciliation
SELECT
  coa.account_name,
  SUM(gle.debit - gle.credit) AS gl_balance,
  (
    SELECT SUM(si.total_amount_due - si.total_paid)
    FROM sales_invoices si
    WHERE si.ar_account_id = gle.account_id
      AND si.company_id = :company_id
      AND si.date <= :as_of_date
  ) AS subledger_total,
  SUM(gle.debit - gle.credit) -
  (
    SELECT COALESCE(SUM(si.total_amount_due - si.total_paid), 0)
    FROM sales_invoices si
    WHERE si.ar_account_id = gle.account_id
      AND si.company_id = :company_id
      AND si.date <= :as_of_date
  ) AS variance
FROM general_ledger_entries gle
JOIN chart_of_accounts coa ON coa.id = gle.account_id
WHERE coa.is_control_account = true
  AND coa.subledger_type = 'ar'
  AND gle.company_id = :company_id
  AND gle.posting_date <= :as_of_date
GROUP BY coa.account_name, gle.account_id;
```
