# Document Void Register

## Purpose

The Document Void Register is the immutable CAS evidence of a document entering a terminal lifecycle state such as voided, cancelled, bounced, or reversed. It is not rebuilt from the document's current status in the browser.

## Authoritative Source

`cas_document_void_events` is populated by database lifecycle triggers in the same transaction as the terminal status change. Each event retains:

- company and branch context;
- source table/ID, document code/number, and original document date;
- terminal status, event actor, and exact event timestamp;
- required reason code/text;
- party ID/type/name/TIN and document amount frozen at the transition;
- original and reversal journal-entry links;
- the source-row JSON snapshot used as event evidence.

The table is company-scoped by RLS and read-only to application roles. Direct insert, update, delete, and truncate privileges are revoked; security-definer lifecycle code is the only writer. A unique source/status key prevents duplicate evidence for one transition.

## User Interface

`CASDocumentVoidRegisterPage` reads this table directly, filters by the event timestamp, and displays both the event and original document dates. CSV export is a convenience rendering of the governed rows; immutable/exported-byte provenance for downloadable CAS artifacts remains under PXL-DA-019.

## Control Rules

- A terminal transition must carry a nonblank reason; SI/VB use governed reason codes and other cancellation RPCs carry the required memo through a transaction-local database setting.
- Posted source documents retain their original journal. Reversal evidence links to a separate exact-opposite journal entry.
- Voiding never rolls a number-series counter backward and never makes the number reusable.
- Historical rows that predate the control are backfilled from surviving source/audit evidence and remain distinguishable by their captured snapshot/timestamps.
