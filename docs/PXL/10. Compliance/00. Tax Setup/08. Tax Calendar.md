## Module Overview
The Tax Calendar provides a complete schedule of BIR and LGU filing deadlines for the company's active compliance obligations. Missing BIR deadlines results in 25% surcharge + 12% annual interest + compromise penalties — a catastrophic cost for any enterprise. **The calendar is auto-generated from the company's Compliance Profile** — there is no manual entry of deadlines. When a Compliance Profile is saved or updated (e.g., `vat_registered` toggled to `true`), the system automatically generates all corresponding deadline events for the current and next fiscal year.

**Principle 11 (Low Maintenance):** Developers must NEVER hardcode BIR deadlines in application logic. All deadline computation is driven by `compliance_profiles` flags and eFPS group rules stored in `ref_compliance_forms`.

## Auto-Generation Logic (Triggered on Compliance Profile Save)

When `compliance_profiles` is created or updated, the backend runs the `fn_generate_tax_calendar(company_id, fiscal_year)` function. This function creates `tax_calendar_events` records based on the active flags:

| Compliance Profile Flag | Forms Generated | Frequency |
| :--- | :--- | :--- |
| `vat_registered = true` AND `vat_filing_frequency = 'monthly'` | 2550M | 12 events per year (one per month) |
| `vat_registered = true` | 2550Q | 4 events per year (one per quarter) |
| `ewt_registered = true` AND `files_0619e = true` | 0619-E | 8 events per year (months 1 & 2 of each quarter) |
| `ewt_registered = true` | 1601EQ | 4 events per year (one per quarter) |
| `ewt_registered = true` AND `qap_required = true` | QAP | 4 events per year (attached to 1601EQ) |
| `ewt_registered = true` AND `requires_1604e = true` | 1604-E | 1 event per year (annual) |
| `fwt_registered = true` AND `files_0619f = true` | 0619-F | 8 events per year |
| `fwt_registered = true` | 1601FQ | 4 events per year |
| `slsp_required = true` | SLSP | 4 events per year (quarterly) |
| `relief_required = true` | RELIEF | 4 events per year (quarterly) |
| `sawt_required = true` | SAWT | 4 events per year (quarterly) |
| Always | 1702Q (Income Tax) | 3 events per year (Q1, Q2, Q3 quarters) |
| Always | 1702 (Annual ITR) | 1 event per year |
| Always | Mayor's Permit Renewal | 1 event per year (January 20) |

### eFPS Group Deadline Adjustment

If `compliance_profiles.efps_enrolled = true`, all BIR deadlines are adjusted based on `compliance_profiles.efps_group` per BIR Revenue Regulations:

| eFPS Group | Adjustment |
| :--- | :--- |
| A (Large Taxpayers) | Statutory deadline + 5 banking days |
| B | Statutory deadline + 4 banking days |
| C | Statutory deadline + 3 banking days |
| D | Statutory deadline + 2 banking days |
| E | Statutory deadline + 1 banking day |
| (Not enrolled) | Statutory deadline (no adjustment) |

The `fn_generate_tax_calendar` function applies this adjustment to every generated event's `efps_adjusted_deadline` column.

### Table 1: `tax_calendar_events`

**Critical Database Rules:**
1. `UNIQUE(company_id, form_code_id, coverage_period_start)` — prevents duplicate events for the same form and period.
2. Events are generated (not manually created). The UI allows updating `status`, `date_filed`, and `efps_reference_no` only.
3. `efps_adjusted_deadline` is computed on generation based on eFPS group rules. Never manually editable.

| Column Name | Data Type | Rules | What it stores |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System ID. |
| `company_id` | UUID | Required, Foreign Key | Links to `companies.id`. |
| `compliance_form_id` | UUID | Required, Foreign Key | Links to `ref_compliance_forms.id`. NOT free text. |
| `coverage_period_start` | Date | Required | First day of the period covered (e.g., 2026-01-01 for January 2550M). |
| `coverage_period_end` | Date | Required | Last day of the period covered (e.g., 2026-01-31). |
| `statutory_deadline` | Date | Required | The base BIR deadline per revenue regulations. |
| `efps_adjusted_deadline` | Date | Nullable | Deadline adjusted for eFPS group. Null if not eFPS enrolled. |
| `effective_deadline` | Date | Required | The deadline the system alerts on: `efps_adjusted_deadline` if not null, else `statutory_deadline`. |
| `status` | Text | Required, Default `pending` | ENUM: `pending`, `filed`, `late`. `late` is auto-set by the system when `effective_deadline < NOW()` and `status = 'pending'`. |
| `date_filed` | Date | Nullable | Actual filing/submission date. |
| `efps_reference_no` | Text | Nullable | eFPS confirmation number or eBIR reference. |
| `assigned_to_user_id` | UUID | Nullable, Foreign Key | Links to `users.id`. User responsible for this filing. |
| `created_by` | UUID | Required, Foreign Key | Links to `users.id`. System user that generated the event. |
| `updated_by` | UUID | Nullable, Foreign Key | Links to `users.id`. |
| `created_at` | Timestamp | Auto | Creation timestamp. |
| `updated_at` | Timestamp | Auto | Last update timestamp. |

**Composite Unique Constraint:** `UNIQUE(company_id, compliance_form_id, coverage_period_start)`

## Auto-Generation Function Specification

```sql
-- Called when compliance_profiles is created or updated
CREATE OR REPLACE FUNCTION fn_generate_tax_calendar(
  p_company_id UUID,
  p_fiscal_year INTEGER
)
RETURNS void
LANGUAGE plpgsql
AS $$
DECLARE
  v_profile compliance_profiles%ROWTYPE;
  v_efps_days INTEGER;
BEGIN
  -- Fetch the company's compliance profile
  SELECT * INTO v_profile
  FROM compliance_profiles
  WHERE company_id = p_company_id AND is_active = true;

  -- Compute eFPS adjustment days
  v_efps_days := CASE
    WHEN v_profile.efps_enrolled = false THEN 0
    WHEN v_profile.efps_group = 'A' THEN 5
    WHEN v_profile.efps_group = 'B' THEN 4
    WHEN v_profile.efps_group = 'C' THEN 3
    WHEN v_profile.efps_group = 'D' THEN 2
    WHEN v_profile.efps_group = 'E' THEN 1
    ELSE 0
  END;

  -- Delete existing PENDING events for this year before regenerating
  -- (preserves FILED events)
  DELETE FROM tax_calendar_events
```

## Row-Level Security

```sql
-- Filed events are immutable (only pending events can be updated)
CREATE POLICY "block_update_on_filed_calendar"
ON tax_calendar_events
FOR UPDATE
USING (status = 'pending');

-- No deletes on calendar events
CREATE POLICY "block_delete_calendar"
ON tax_calendar_events
FOR DELETE
USING (false);

-- Tenant isolation
CREATE POLICY "tenant_isolation_calendar"
ON tax_calendar_events
FOR ALL
USING (company_id = (auth.jwt() ->> 'company_id')::UUID);
```
