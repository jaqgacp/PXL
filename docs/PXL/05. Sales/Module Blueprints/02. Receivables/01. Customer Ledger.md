## Module Overview
The Customer Ledger (Subsidiary Ledger) provides a comprehensive, chronological view of all financial transactions related to a specific customer. In Philippine accounting, it is essential for reconciling balances, tracing the history of invoices, payments (receipts), credit memos, and debit memos. It acts as the definitive source of truth for a customer's running balance.

*Note: This is a reporting/register module, not a data entry transaction. No inheritance rules apply as it merely aggregates posted data.*

## Dashboard UI
A read-only report view that aggregates financial data.

### The Action Bar
* **Customer Selector:** A required dropdown to choose which Customer's ledger to view.
* **Date Range Filter:** "From" and "To" date pickers to narrow the statement period.
* **Export Button:** Export the ledger to Excel or PDF (often sent as a Statement of Account).
* **Print Button:** Print the ledger directly.

### The Data List
Once a customer and date range are selected, the grid displays the chronological transaction history.

Default columns displayed:
* Date
* Document Type (e.g., SI, OR, CM, DM)
* Document Number
* Description / Remarks
* Debit (Increases AR)
* Credit (Decreases AR)
* Running Balance

## Supabase Database Architecture


**Critical Database Rule:**
- Every transactional table MUST include a `company_id` to ensure tenant isolation.
- Composite unique constraints MUST be established (e.g., `company_id` + document number) to prevent duplicate documents within the same company.

Because the Customer Ledger is a consolidated report of multiple transactional tables, it does not have its own standalone table. Instead, we use a PostgreSQL Database View to union the data for easy querying by the frontend.

### View Definition: `vw_customer_ledger`
This view combines data from `sales_invoices`, `receipts`, `credit_memos`, and `debit_memos`.

| Column Name | Data Type | Source | What it stores |
| :--- | :--- | :--- | :--- |
| `company_id` | UUID | Required | Links to `companies.id` / for tenant filtering. |
| `customer_id` | UUID | All Tables | To filter the ledger. |
| `transaction_date`| Date | `invoice_date`, `receipt_date`, etc. | Chronological sorting. |
| `doc_type` | Text | Hardcoded per SELECT | 'SI', 'OR', 'CM', 'DM'. |
| `doc_number` | Text | `si_number`, `receipt_number`, etc. | Reference number. |
| `description` | Text | `remarks` | Context of the transaction. |
| `debit_amount` | Numeric | SI and DM total amounts | Increases the receivable. |
| `credit_amount` | Numeric | OR and CM total amounts | Decreases the receivable. |
| `created_at` | Timestamp | Source `created_at` | For strict chronological sorting if dates match. |
| `created_by` | UUID | Required, Foreign Key | Links to `users.id`. |
| `updated_by` | UUID | Nullable, Foreign Key | Links to `users.id`. |
| `updated_at` | Timestamp | Auto | Date record was last edited. |

*Note: The frontend or a Supabase RPC (Remote Procedure Call) will calculate the `Running Balance` dynamically by summing `debit_amount` and subtracting `credit_amount` chronologically.*
