## Module Overview
The Petty Cash Settings page defines the global rules and financial controls that govern how the Petty Cash module operates across the company. Petty cash is used for small, routine disbursements that are impractical to process through the normal check-based payment workflow. These settings establish monetary limits, replenishment thresholds, GL account defaults, and approval requirements to ensure that petty cash funds are properly controlled and auditable.

**Critical Philippine Internal Control Note:** Under standard Philippine accounting practice and BIR audit readiness guidelines, petty cash disbursements are subject to strict monetary limits. Disbursements exceeding the company-defined limit per Petty Cash Voucher (PCV) are not allowed — they must be processed as a formal check payment or payment voucher instead. The system enforces this limit as a hard block to prevent policy violations. Additionally, all petty cash replenishments are subject to the same expense documentation and approval standards as regular disbursements.

## What You Will See on the Screen (The Dashboard)
When you open the "Petty Cash Settings" page, you will see a single, organized settings form — not a list. This page has one configuration record per company.

### The Action Bar (Top of the page):
* **Company Selector:** A dropdown to select which company's petty cash settings you are configuring.
* **Save Settings Button:** A prominent button to save all changes on the page.
* **Reset to Defaults Button:** A secondary button to revert all settings to the system defaults.

## What Information We Need to Capture (The Data Fields)
The settings form is divided into logical sections.

### Section 1: Disbursement Limits & Controls
| Field Name | Description | Required? |
| :--- | :--- | :--- |
| Company | Dropdown linked to active companies. These settings apply to the selected entity. | Yes |
| Maximum Amount Per Petty Cash Voucher (PCV) | The maximum amount (in PHP) that can be disbursed on a single Petty Cash Voucher. Disbursements exceeding this amount must be processed through the regular payment voucher workflow. The system applies this as a **hard block** — the PCV cannot be saved if the amount exceeds this limit. Common policy limits in the Philippines are PHP 1,000 to PHP 5,000 per voucher. | Yes |
| Maximum Single Fund Size | The maximum total amount (in PHP) that a single Petty Cash Fund can hold at any time. This applies when setting up a new fund or increasing an existing fund. | Yes |
| Allow Splitting of Expenses Across Multiple PCVs | Toggle (Yes / No). If NO, a single purchase or expense cannot be covered by multiple PCVs (prevents circumvention of the per-voucher limit). | Yes |

### Section 2: Replenishment Rules
| Field Name | Description | Required? |
| :--- | :--- | :--- |
| Replenishment Trigger | Dropdown: **Manual** (Fund custodian initiates replenishment at any time), **Threshold-Based** (System triggers a replenishment alert when the fund balance falls below the threshold defined below). | Yes |
| Low Balance Threshold (%) | The percentage of the fund's original value at which a replenishment alert is triggered (e.g., 20% means an alert fires when less than 20% of the fund remains). Visible only when Replenishment Trigger = Threshold-Based. | Conditional |
| Require Approval for Replenishment | Toggle (Yes / No). If YES, a Petty Cash Replenishment request must be approved through the Approval Matrix before the replenishment check or payment is released. | Yes |
| Require Supporting Documents for Replenishment | Toggle (Yes / No). If YES, all Petty Cash Vouchers used in a replenishment batch must have an attached receipt or supporting document before the replenishment can be processed. This is critical for BIR audit readiness. | Yes |

### Section 3: GL Account Defaults
| Field Name | Description | Required? |
| :--- | :--- | :--- |
| Default Petty Cash Asset Account | Dropdown linked to the Chart of Accounts. The GL account that represents the petty cash fund as a current asset (e.g., "1005-000 Petty Cash Fund"). | Yes |
| Default Cash Short / Over Account | Dropdown linked to the Chart of Accounts. The GL account used to record variances discovered during a Cash Count (e.g., "7200-000 Cash Short and Over"). | Yes |

### Section 4: Cash Count Controls
| Field Name | Description | Required? |
| :--- | :--- | :--- |
| Require Cash Count Before Replenishment | Toggle (Yes / No). If YES, a Cash Count Sheet must be completed and reconciled before a replenishment request can be submitted. This ensures the physical cash on hand matches the system balance. | Yes |
| Allow Cash Count Variance | Toggle (Yes / No). If YES, small variances discovered during a Cash Count are automatically posted to the Cash Short/Over account. If NO, the fund custodian must resolve any variance before the count is closed. | Yes |
| Maximum Allowable Cash Count Variance (PHP) | The maximum peso amount of variance that the system will automatically accept and post to the Cash Short/Over account. Variances exceeding this amount require supervisor approval. | Yes |

## The Supabase Database Architecture
This module requires one settings table. It stores one record per company.

### Table 1: `petty_cash_settings` (The Main Table)
**Critical Database Rule:** Apply a UNIQUE constraint on `company_id` to ensure only one settings record exists per company.

**Application Logic:** When saving a new Petty Cash Voucher, the system must validate the entered amount against `max_amount_per_pcv`. If the amount exceeds the limit, the save action must be blocked with a clear error message: *"Disbursement amount exceeds the maximum allowed per Petty Cash Voucher (PHP [limit]). Please process this as a Payment Voucher instead."*

| Column Name | Data Type | Rules | What it stores |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System ID. |
| `company_id` | UUID | Required, Foreign Key, Unique | Links to `companies.id`. |
| `max_amount_per_pcv` | Numeric(18,2) | Required, Default: `2000.00` | Maximum peso amount allowed per single Petty Cash Voucher. |
| `max_fund_size` | Numeric(18,2) | Required, Default: `50000.00` | Maximum total amount a single petty cash fund can hold. |
| `allow_splitting_across_pcv` | Boolean | Default: `false` | Whether one expense can be split across multiple PCVs. |
| `replenishment_trigger` | Text | Required, Default: `manual` | `manual` or `threshold_based`. |
| `low_balance_threshold_pct` | Numeric(5,2) | Nullable, Default: `20.00` | Fund balance percentage that triggers a replenishment alert. |
| `require_replenishment_approval` | Boolean | Default: `true` | Whether replenishment requests require approval. |
| `require_supporting_docs` | Boolean | Default: `true` | Whether receipts must be attached before replenishment. |
| `default_petty_cash_gl_id` | UUID | Nullable, Foreign Key | Links to `gl_accounts.id`. Petty cash asset account. |
| `default_cash_short_over_gl_id` | UUID | Nullable, Foreign Key | Links to `gl_accounts.id`. Cash short/over account. |
| `require_cash_count_before_replenishment` | Boolean | Default: `true` | Whether a cash count is required before replenishment. |
| `allow_cash_count_variance` | Boolean | Default: `true` | Whether small cash count variances are auto-posted. |
| `max_allowable_variance` | Numeric(18,2) | Default: `50.00` | Maximum variance (in PHP) auto-posted to cash short/over. |
| `created_by` | UUID | Required, Foreign Key | Links to `users.id`. User who created this record. |
| `updated_by` | UUID | Nullable, Foreign Key | Links to `users.id`. User who last edited this record. |
| `created_at` | Timestamp | Auto | Date record was created. |
| `updated_at` | Timestamp | Auto | Date record was last edited. |
| `updated_at` | Timestamp | Auto | Date record was last edited. |
