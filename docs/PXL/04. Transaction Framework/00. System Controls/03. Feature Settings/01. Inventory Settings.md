## Module Overview
The Inventory Settings page defines the global behavioral rules that govern how the Inventory module operates across the entire company. These settings affect costing, stock controls, warehouse management, and how inventory transactions post to the General Ledger. They are configured once at the company level and applied system-wide to all inventory-related transactions.

**Critical Philippine Tax Compliance Note:** The Bureau of Internal Revenue (BIR) explicitly prescribes acceptable inventory costing methods under Revenue Regulations and the Philippine Financial Reporting Standards (PFRS). The **LIFO (Last-In, First-Out)** method is strictly **NOT permitted** in the Philippines. PXL ERP enforces this by removing LIFO as an option entirely. Only **FIFO** and **Moving/Weighted Average** are available.

## What You Will See on the Screen (The Dashboard)
When you open the "Inventory Settings" page, you will see a single, organized settings form — not a list. This page has one configuration record per company.

### The Action Bar (Top of the page):
* **Company Selector:** A dropdown to select which company's inventory settings you are configuring (for multi-company setups).
* **Save Settings Button:** A prominent button to save all changes on the page.
* **Reset to Defaults Button:** A secondary button to revert all settings to the system defaults.

## What Information We Need to Capture (The Data Fields)
The settings form is divided into logical sections.

### Section 1: Costing & Valuation
| Field Name | Description | Required? |
| :--- | :--- | :--- |
| Company | Dropdown linked to active companies. These settings apply to the selected entity. | Yes |
| Inventory Costing Method | Dropdown: **FIFO (First-In, First-Out)** or **Moving Weighted Average**. This is a foundational setting — it determines how Cost of Goods Sold (COGS) and ending inventory values are computed. **LIFO is not available** (BIR-prohibited). | Yes |
| Costing Method Lock | Toggle (Yes / No). If YES, the costing method is locked and cannot be changed after the first inventory transaction has been recorded. Prevents mid-period valuation inconsistencies. | Yes |
| Default Inventory Valuation Account | Dropdown linked to the Chart of Accounts. The default GL account for inventory asset balances (e.g., "1300-000 Merchandise Inventory"). | Yes |
| Default COGS Account | Dropdown linked to the Chart of Accounts. The default GL account for cost of goods sold (e.g., "5000-000 Cost of Sales"). | Yes |

### Section 2: Stock Controls
| Field Name | Description | Required? |
| :--- | :--- | :--- |
| Prevent Negative Inventory | Toggle (Yes / No). **Recommended: YES.** If enabled, the system will block any stock outflow transaction (Sales Invoice, Delivery Receipt, Goods Issue) that would result in an item's quantity going below zero. This prevents phantom inventory and incorrect COGS calculations. | Yes |
| Negative Inventory Warning Behavior | Dropdown (Visible only if Prevent Negative Inventory = YES): **Hard Block** (transaction cannot be saved), **Soft Warning** (shows an alert but allows the user to proceed with override). | Conditional |
| Auto-Reserve Stock on Sales Order | Toggle (Yes / No). If YES, confirming a Sales Order will automatically reserve the ordered quantity, reducing the "Available" count even before a Delivery Receipt is created. | Yes |
| Require Physical Count Approval | Toggle (Yes / No). If YES, Physical Count adjustments must go through an approval workflow before they are posted to the GL. | Yes |

### Section 3: Warehouse & Tracking
| Field Name | Description | Required? |
| :--- | :--- | :--- |
| Enable Multi-Warehouse Tracking | Toggle (Yes / No). If YES, inventory quantities are tracked per warehouse location. Stock transfers between warehouses are recorded as formal transactions. If NO, the system treats inventory as a single pool. | Yes |
| Default Warehouse | Dropdown linked to active warehouses. The warehouse that is auto-selected when creating new stock receipts and delivery transactions. | Yes |
| Enable Lot / Batch Tracking | Toggle (Yes / No). If YES, items can be assigned lot or batch numbers for traceability (useful for perishables, pharmaceuticals, or items with expiry dates). | Yes |
| Enable Expiry Date Tracking | Toggle (Yes / No, visible if Lot Tracking = YES). If YES, the system tracks expiry dates per lot and can trigger alerts for near-expiry stock. | Conditional |

### Section 4: Purchasing & Receiving Integration
| Field Name | Description | Required? |
| :--- | :--- | :--- |
| Require Receiving Report Before Vendor Bill | Toggle (Yes / No). If YES, a Vendor Bill for inventory items cannot be posted unless a corresponding Receiving Report (RR) has been created and confirmed. Enforces the 3-way matching control (PO → RR → Bill). | Yes |
| Allow Partial Receiving | Toggle (Yes / No). If YES, a single Purchase Order can have multiple Receiving Reports (i.e., goods can be received in batches). | Yes |

## The Supabase Database Architecture
This module requires one settings table. It stores one record per company.

### Table 1: `inventory_settings` (The Main Table)
**Critical Database Rule:** Apply a UNIQUE constraint on `company_id` to ensure only one settings record exists per company.

| Column Name | Data Type | Rules | What it stores |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System ID. |
| `company_id` | UUID | Required, Foreign Key, Unique | Links to `companies.id`. |
| `costing_method` | Text | Required, Default: `moving_average` | `fifo` or `moving_average`. LIFO is excluded. |
| `costing_method_locked` | Boolean | Default: `false` | Whether the costing method can be changed. |
| `default_inventory_gl_id` | UUID | Nullable, Foreign Key | Links to `gl_accounts.id`. Default inventory asset account. |
| `default_cogs_gl_id` | UUID | Nullable, Foreign Key | Links to `gl_accounts.id`. Default COGS account. |
| `prevent_negative_inventory` | Boolean | Default: `true` | Whether negative stock is blocked. |
| `negative_inventory_behavior` | Text | Default: `hard_block` | `hard_block` or `soft_warning`. |
| `auto_reserve_on_sales_order` | Boolean | Default: `false` | Whether SO confirmation reserves stock. |
| `require_physical_count_approval` | Boolean | Default: `true` | Whether stock adjustments require approval. |
| `enable_multi_warehouse` | Boolean | Default: `false` | Whether per-warehouse tracking is active. |
| `default_warehouse_id` | UUID | Nullable, Foreign Key | Links to `warehouses.id`. |
| `enable_lot_tracking` | Boolean | Default: `false` | Whether lot/batch tracking is active. |
| `enable_expiry_tracking` | Boolean | Default: `false` | Whether expiry date tracking is active. |
| `require_rr_before_bill` | Boolean | Default: `true` | Whether 3-way matching is enforced. |
| `allow_partial_receiving` | Boolean | Default: `true` | Whether partial PO receiving is allowed. |
| `created_by` | UUID | Required, Foreign Key | Links to `users.id`. User who created this record. |
| `updated_by` | UUID | Nullable, Foreign Key | Links to `users.id`. User who last edited this record. |
| `created_at` | Timestamp | Auto | Date record was created. |
| `updated_at` | Timestamp | Auto | Date record was last edited. |
