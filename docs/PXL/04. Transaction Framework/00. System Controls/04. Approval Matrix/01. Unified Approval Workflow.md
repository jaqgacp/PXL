## Current Implementation

PXL has one approval architecture. MDP-14 extends the existing `approval_workflows`, `approval_workflow_steps`, and `approval_instances` tables and adds `approval_requests` as the request header. No module-specific approval tables or parallel role/permission system may be introduced.

Implementation evidence:

- Migration: `20260722000009_mdp14_approval_matrix_integration.sql`
- Focused test: `074_mdp14_approval_matrix_integration_test.sql` (61/61)
- Full regression: 74 files / 1,568 assertions
- Existing compatibility tests: 011, 014, 050, 071, 072, 073

The current bounded consumer is a configured MDP-15 master-data import commit. The generic APIs are reusable by future transaction consumers, but MDP-14 does not claim transaction-wide rollout.

## Configuration Model

`approval_workflows` stores one company rule and supports:

- optional branch
- module and record/document type
- action type
- amount threshold
- currency context
- requester user or existing membership role code
- explicit priority
- effective-from and effective-to timestamps
- active/inactive status
- requester/approver separation

`approval_workflow_steps` stores ordered approval levels. An active step assigns either an existing company membership role code or a specific user. The existing `user_company_memberships` role is authoritative; `approver_role_code` does not define a second role model.

No rules are seeded. New and provisioned companies therefore remain usable until an administrator deliberately activates a valid route.

## Deterministic Precedence

Only active, effective rules for the company, module, record, action, requester, amount, currency, and optional branch may match. The first rule is selected in this exact order:

1. Highest number of constrained optional criteria.
2. Branch-specific before company fallback.
3. Record/document-specific before blank fallback.
4. Exact action before `*` fallback.
5. Requester-user-specific.
6. Requester-role-specific.
7. Currency-specific.
8. Non-`always` condition.
9. Highest matching amount threshold.
10. Highest explicit priority.
11. Newest effective start.
12. Earliest creation timestamp, then UUID.

Specificity is evaluated before explicit priority, so a broad high-priority rule cannot unexpectedly override a more specific valid rule.

## Runtime Model

`approval_requests` stores the matched rule, company/branch context, requester and role, source identity, immutable source version, safe snapshot metadata, current level, lifecycle state, and final consumption evidence.

`approval_instances` remains the per-step runtime record. MDP-14 links each instance to its request and adds role-code snapshots plus `waiting`, `pending`, `approved`, `rejected`, `cancelled`, and `superseded` states while preserving legacy states.

Allowed request transitions:

| Current | Operation | Result |
| --- | --- | --- |
| not submitted | submit with matching valid rule | pending |
| pending | approve with later levels | partially approved |
| pending / partially approved | final approve | approved |
| pending / partially approved | reject | rejected |
| pending / partially approved | requester/admin withdraw | withdrawn |
| pending / partially approved | changed source version | superseded |
| approved | supported consumer executes exact action | approved and consumed |

Rejected, withdrawn, superseded, consumed, and fully approved requests cannot be approved again. A changed version must be resubmitted.

## Server Interfaces

| Interface | Purpose |
| --- | --- |
| `fn_resolve_approval_rule` | Return the deterministic matching rule and precedence evidence. |
| `fn_get_approval_decision` | Return required/not-required, route health, and final-actionable state before submission. |
| `fn_submit_approval_request` | Validate permission, branch, route, candidates, source version, and create ordered instances. |
| `fn_approve_approval_request` | Lock and approve only the current step after permission, route, SOD, branch, and version checks. |
| `fn_reject_approval_request` | Reject the current step with a required reason and cancel later waiting steps. |
| `fn_withdraw_approval_request` | Let the requester or company administrator close an actionable request. |
| `fn_get_approval_request_status` | Return current state and ordered history. |
| `fn_approval_inbox` | Return only current pending steps the caller is eligible to decide. |

The legacy `fn_required_approval_workflow(UUID, TEXT, TEXT, NUMERIC)` signature remains as a DEC-010 compatibility adapter.

## Security and SOD

- Company membership and optional branch scope are checked server-side.
- Master Data submission/decision permissions use MDP-03 `fn_can_master_data_permission`.
- Master Data conflict enforcement uses `master_data_sod_conflicts`; advisory conflicts remain advisory and `enforced` conflicts block the decision.
- A configured route can require requester and approver separation.
- Specific-user and role candidates must be current company members and active, non-banned auth users.
- A route with no valid approver fails closed during submission.
- Authenticated users cannot directly insert/update/delete approval instances or requests.
- Workflow and step configuration remains company owner/admin maintained through RLS.

The authenticated UI is an additional visibility/ergonomics layer, not the authorization boundary.

## Integrity and Concurrency

- A partial unique index allows only one actionable request for the same company/module/action/source.
- Same-version submission is idempotent.
- Changed-version submission supersedes the prior actionable request.
- Approval/rejection locks the request and current instance before transition.
- Only the current pending sequence can be acted on.
- Version mismatch supersedes the stale request without changing the source record.
- Repeated and invalid lifecycle actions are rejected.
- Rule, request, instance, and decision changes reuse `fn_audit_trigger` and `sys_audit_logs`.

## MDP-15 Integration

MDP-15 preview remains unchanged. When no active `master_data` / `import` rule matches, commit remains unchanged. When a rule matches:

1. The requester previews a valid payload.
2. The request is anchored to the preview batch and input hash.
3. All configured levels approve the exact hash.
4. Commit supplies `approval_request_id` in `p_options`.
5. The wrapper re-resolves the rule, locks the request, verifies requester/company/master/hash/status, calls the frozen MDP-15 core, and records consumption.
6. The same idempotency key may replay; another commit cannot reuse the approval.

The internal MDP-15 core is not executable by authenticated clients, preventing direct bypass.

## UI Scope

The existing Approval Workflow page exposes rule criteria, role/user steps, owner/admin-aware maintenance, clear server validation messages, and a server-filtered inbox with approve/reject actions. It preserves existing navigation and styling and does not claim unsupported transaction coverage.
