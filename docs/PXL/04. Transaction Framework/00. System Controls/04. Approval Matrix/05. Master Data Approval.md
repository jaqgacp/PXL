**Status:** Implemented foundation with bounded import enforcement
**Authority:** Tier 2 Implementation Blueprint; migrations and tests prevail
**Owner / Domain:** Setup, Master Data, and Transaction Framework
**Applies To:** Configured sensitive Master Data actions
**Read When:** Adding or validating a Master Data approval consumer

**Last Reviewed:** 2026-07-22 after MDP-14

## Architecture

Master Data uses the unified approval architecture. It has no module-specific workflow, role, permission, SOD, or audit tables.

- Rules: `approval_workflows` with `module_type = 'master_data'`
- Steps: `approval_workflow_steps`
- Request header: `approval_requests`
- Runtime levels/history: `approval_instances`
- Permissions: MDP-03 `master_data_permissions` and `master_data_role_permissions`
- SOD metadata: MDP-03 `master_data_sod_conflicts`
- Audit: MDP-02 `fn_audit_trigger` and `sys_audit_logs`

Rule `document_type` uses the current MDP-15 `master_key` (for example, `departments`, `items`, or `chart_of_accounts`). Rule `action_type` identifies the sensitive operation such as `create`, `edit`, `delete`, `activate`, `deactivate`, or `import`.

## Current Enforced Consumer

MDP-14 enforces configured `import` rules on MDP-15 commit operations. Import preview remains available for validation and supplies the immutable batch/input hash used by the approval request. An approved request authorizes only the exact company, master key, requester, workflow, payload hash, and commit consumption.

No rules are seeded. If no active rule matches, the completed MDP-15 behavior is unchanged. MDP-08 provisioning also remains unchanged and does not create a default approver dependency.

## Permission Mapping

Submission uses the operation's MDP-03 permission:

| Approval action | MDP-03 permission |
| --- | --- |
| create | create |
| edit / activate / deactivate | edit |
| delete / archive | delete |
| import | import |
| export | export |

Approval and rejection require the master key's `approve` permission plus eligibility for the current workflow step. Existing custom membership role codes are supported through `master_data_role_permissions`.

## SOD and Route Controls

- Requester/approver separation is enabled by default for new rules.
- Company and branch scope must match.
- Specific-user and role approvers must be active valid company users.
- A missing valid approver prevents submission.
- MDP-03 advisory conflicts remain visible but do not block; conflicts explicitly changed to `enforced` block the decision.
- Direct request/instance mutation by authenticated users is denied.
- Stale, repeated, wrong-level, wrong-role, rejected, withdrawn, and superseded actions are denied.

## Consumer Contract

Future sensitive Master Data consumers must:

1. Call `fn_get_approval_decision` before deciding whether to submit.
2. Supply a stable source UUID and deterministic current record version.
3. Use `fn_submit_approval_request`; never insert request/instance rows directly.
4. Require final `approved` status before executing the governed action.
5. Revalidate the same current version inside the action transaction.
6. Record consumption or equivalent final-action evidence atomically.
7. Preserve no-rule behavior unless the product requirement deliberately configures a rule.

MDP-14 does not automatically gate every ordinary master edit. Additional consumers should be selected only for genuinely sensitive operations to avoid unnecessary workflow friction.
