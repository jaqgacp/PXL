# PXL Sales Invoice Form UX Standard

Status: Approved  
Date: 2026-07-15
Scope: Canonical Sales Invoice creation and draft-edit UX standard  
Document type: Product and UX standard only  
Implementation boundary: This document does not implement code, edit database schemas, change accounting logic, change tax logic, change posting logic, change routes or components, create migrations, invent missing master data, or start a refactor.

## 1. Executive Summary

The Sales Invoice Form workspace is the canonical pilot for PXL transaction capture and draft-edit experiences. It governs Sales Invoice creation, draft editing, auto-population, line entry, controlled overrides, live totals, preview, and readiness without becoming a basic CRUD form, a government-style data-entry form, or a dashboard of disconnected cards.

The primary experience principle is:

> The user enters each business fact once. Everything that can be derived, defaulted, computed, validated, or retrieved from authoritative setup must be handled by PXL.

PXL should ask only for information the system cannot determine reliably.

This standard preserves the existing document's strengths while correcting duplication risks, terminology issues, and architecture inconsistencies. It aligns the Sales Invoice form pilot with the official PXL transaction-workspace direction and prepares the capture/edit pattern for future use across Sales, Purchasing, Accounting, Inventory, Banking, Fixed Assets, Payroll, and Compliance.

Existing saved-invoice review, approval review, posted-document view, collection monitoring, void/reversal review, drilldown, and audit UX are governed by `PXL_SALES_INVOICE_VIEW_UX_STANDARD.md`.

The goal is not to imitate NetSuite visually. The goal is to adopt strong enterprise ERP principles while making PXL faster, clearer, less overwhelming, and more appropriate for Philippine accounting and tax workflows.

## 2. Authority And Precedence

This UX standard is subordinate to:

1. Governed accounting rules
2. Governed tax rules
3. Transaction lifecycle rules
4. Database constraints and authoritative RPCs
5. Permission and segregation-of-duties rules
6. Posted-document immutability
7. Existing canonical PXL transaction-workspace architecture
8. `PXL_SALES_INVOICE_VIEW_UX_STANDARD.md` for existing saved-document view behavior

This UX standard defines:

- Page structure
- Information hierarchy
- Field ownership
- User interaction
- Presentation
- Progressive disclosure
- Role-based visibility
- Reusable experience patterns

It must not redefine accounting or tax behavior. If accounting, tax, master-data, or authoritative data source coverage is incomplete, the interface must identify the gap truthfully instead of hiding it with attractive UI.

## 3. Product Experience Objective

The Sales Invoice Form must feel like one intelligent enterprise transaction-capture workspace, not:

- A basic CRUD form
- A government-style data-entry form
- A dashboard made of disconnected cards
- A page that repeats the same customer, financial, tax, or audit data in several places

The user must be able to:

- Create an invoice quickly
- Understand what information is required
- Avoid entering information already available in master data
- See totals update immediately
- Review expected GL and tax impact
- Know whether the document is ready for saving, submission, approval, or posting
- Understand the document lifecycle
- Review related documents
- Inspect the customer profile when needed
- Trace every accounting and audit consequence after posting

The design must remain powerful for accountants and auditors without overwhelming encoders and normal business users.

## 4. Core UX Principles

### 4.1 Enter Once, Derive Everything Possible

Every field belongs to one of these categories:

1. User-entered business fact
2. Default from master data
3. Derived from another selected record
4. Computed by the system
5. Generated by configuration or number series
6. System or audit metadata

Only Category 1 should normally require manual user entry.

### 4.2 Repetition Is Allowed Only By Level And Purpose

Repetition is allowed only when the information serves a different level and purpose.

Customer hierarchy:

| Location | Allowed purpose |
| --- | --- |
| Header | Customer name and link only, for immediate document context |
| Customer Information card | Compact transaction identity and tax snapshot, to confirm the customer facts applied to the invoice |
| Related Party tab | Full current operational customer profile for credit, contacts, addresses, aging, recent transactions, and current master information |

Financial hierarchy:

| Location | Allowed purpose |
| --- | --- |
| Header | Invoice Total, Collected, Balance Due |
| Compact live entry summary | Immediate editable-document totals only |
| Financial tab | Complete financial and collection interpretation |

Status hierarchy:

| Location | Allowed purpose |
| --- | --- |
| Header | Current status chips only |
| Workflow and Approval | Full lifecycle and authorization details |

Do not repeat full customer profiles, full financial breakdowns, GL impact, tax impact, audit history, workflow history, or technical metadata across several surfaces.

### 4.3 Progressive Disclosure

Show information when it becomes relevant.

Examples:

- Expected CWT appears only when the customer or transaction is subject to withholding.
- Warehouse fields appear only for inventory items.
- Project or cost-center fields appear only when configured or required.
- Foreign currency details appear only for foreign currency transactions.
- Advanced accounting overrides appear only to authorized accounting roles.
- System metadata remains inside the System tab.

### 4.4 Table-First Enterprise Presentation

Use professional tables for structured records:

- Invoice lines
- Financial breakdown
- GL impact
- Tax impact
- Validation results
- Workflow history
- Approval history
- Audit history
- Related documents
- Related party sections
- Recent transactions
- Attachments
- Activity
- Notes
- System information

Cards are allowed only for the three compact primary information groups near the top of the workspace.

### 4.5 No Permanent Duplicate Sidebar

Do not create a permanent right sidebar containing duplicate information such as Financial Summary, Customer Snapshot, Posting Validation, Quick Actions, Audit Summary, or Transaction Health.

The right sidebar consumes space, forces duplication, and reduces usable invoice-line grid width. Document actions belong in the header toolbar. Detailed information belongs in the corresponding tab.

### 4.6 Truthful Unavailable States

Never fabricate placeholder business data.

Use clear values:

- Not assigned
- Not configured
- Not applicable
- Not linked
- Not yet posted
- Not recorded
- No workflow configured
- No attachments linked

### 4.7 Hide Technical Implementation Details From Normal Users

The primary transaction workspace must use business terminology, not implementation terminology.

Do not show technical values to normal users or normal accountants, including:

- Database identifiers
- Raw UUIDs
- RPC names
- Configuration keys
- Internal rule identifiers
- Internal source-table names
- Implementation metadata

Use business-friendly labels instead:

| Technical value | Business label |
| --- | --- |
| `company_accounting_config.ar_account_id` | Default Accounts Receivable Account |
| `company_accounting_config.vat_payable_account_id` | Default VAT Payable Account |
| `document_line_account` | Revenue Account from Item |

If implementation tracing is required, place it in expandable row details or the System tab for support, diagnostics, administrators, auditors, or developers. It must not clutter the normal transaction workspace.

## 5. Current Design Problems

Current Sales Invoice implementation already contains important foundations, including a saved-document workspace, GL impact visibility, audit evidence, line editing, expected CWT calculation, and server-side validation. The standard must correct the remaining UX and architecture issues before the Sales Invoice becomes the canonical pilot.

Current problems to address:

- `/sales-invoices` mixes register, create, and draft edit behavior in one page.
- `/sales-invoices/:id` is a saved-document workspace, so create/edit and review do not yet fully share one canonical shape.
- `/sales-invoices/new` and `/sales-invoices/:id/edit` are not yet canonical route experiences.
- The prior draft placed Customer under both Document Information and Customer Information.
- The create/edit form uses a broad header section instead of the required three-card information band.
- Customer lookup is a basic select instead of a useful enterprise lookup.
- Customer address can appear as an ordinary editable invoice field, which risks master-data re-entry.
- GL impact and audit evidence can appear inline rather than only in standard tabs.
- Line grid behavior is not yet the full enterprise worksheet target.
- Workflow, approval, validation, related-party, related-document, notes, attachments, activity, system, GL, and tax details are not yet consistently separated by standard responsibility.

These are target gaps for future implementation, not instructions to change code in this document update.

## 6. Canonical Route And Workspace Model

Use one canonical Sales Invoice workspace with mode-based behavior.

| Route | Purpose |
| --- | --- |
| `/sales-invoices` | Sales Invoice register or list |
| `/sales-invoices/new` | Create a new Sales Invoice |
| `/sales-invoices/:id` | Review an existing invoice, governed by `PXL_SALES_INVOICE_VIEW_UX_STANDARD.md` |
| `/sales-invoices/:id/edit` | Edit an editable draft invoice |

Do not create separate competing View, Open, Details, and Review page designs.

The same workspace architecture must support the following modes across the form and view standards:

- Create mode
- Edit mode
- Read-only review mode
- Approval mode
- Posted document mode
- Voided or reversed document mode

The layout remains familiar across modes. Fields, actions, and information may change based on status and permissions, but the page shape must not change. This form standard governs `/sales-invoices/new` and `/sales-invoices/:id/edit`; saved-document view details are governed by `PXL_SALES_INVOICE_VIEW_UX_STANDARD.md`.

## 7. Canonical Page Anatomy

The workspace hierarchy is:

1. Compact document header
2. Three-card information band
3. Compact one-line tab bar
4. Active tab content
5. Optional compact footer metadata

The invoice line grid must become visible quickly and remain the dominant work area. Avoid excessive vertical space above the line grid.

## 8. Header Standard

Header content for an existing Sales Invoice:

- Sales Invoice
- Document Number
- Posting Status
- Collection Status
- Lock Status
- Customer Name and Link
- Invoice Total
- Collected
- Balance Due
- Header Actions

Header content for a new invoice:

- Sales Invoice
- NEW
- Draft Status
- Current Readiness State
- Header Actions

Recommended status chips:

- Posting: Draft, Approved, Posted, Voided, Reversed
- Collection: Open, Partially Paid, Paid, Overpaid
- Lock: Editable, Frozen, Locked

Header metrics:

- Invoice Total and Expected Net Collectible should be visually stronger than secondary labels.
- Do not abbreviate Expected Net Collectible as Expected Net.
- Use weight, spacing, and alignment to create hierarchy.
- Do not significantly increase type size.
- Monetary values must use tabular numbers.

Sales Invoice inherits the system-wide transaction workspace coloring standard:

- Header uses the Sales family subtle light-blue tint.
- Tab navigation uses the same color family.
- Active tabs must be visibly stronger than inactive tabs.
- Use light tint, spacing, a bottom border, and subtle elevation to separate the header, cards, tabs, and line workspace.
- Do not use bright colors or status colors as navigation background.

### PXL Transaction Workspace Design System v1

The Sales Invoice Transaction Workspace is the approved reference implementation for the PXL Transaction Workspace Design System. Future transaction forms and views must inherit the shared standards in:

- `PXL_DESIGN_SYSTEM.md`
- `PXL_TRANSACTION_WORKSPACE_DESIGN_STANDARD.md`
- `PXL_TYPOGRAPHY_STANDARD.md`
- `PXL_COLOR_SYSTEM.md`
- `PXL_COMPONENT_LIBRARY.md`
- `PXL_TABLE_STANDARD.md`
- `PXL_CARD_STANDARD.md`
- `PXL_BUTTON_STANDARD.md`
- `PXL_FORM_STANDARD.md`
- `PXL_TAB_STANDARD.md`

Principles:

- Use Oracle NetSuite as visual inspiration for weight, contrast, readability, and hierarchy, without copying its layout or pixels.
- Preserve the approved PXL transaction architecture.
- Improve visual maturity through contrast, spacing, borders, typography, and restrained color, not decoration.
- Implement reusable tokens and components, not page-specific CSS.

Sales Invoice reference palette:

- Header: deep desaturated blue-gray tint.
- Tab Container: light blue-gray with stronger borders.
- Borders: medium gray-blue.
- Primary Text: `#1F2937`.
- Secondary Text: `#4B5563`.
- Muted Text: `#6B7280`.
- Success: professional green.
- Warning: professional amber.
- Error: professional red.

Header rules:

- Header background must be darker than the generic transaction tint while remaining clean and unsaturated.
- Header must have a stronger bottom border and subtle shadow.
- Header labels, metadata, metrics, and customer link must remain readable.
- Action button hierarchy remains unchanged.

Tab rules:

- The tab container uses a darker neutral blue-gray background.
- Active tab uses white surface, dark charcoal text, semi-bold weight, and a clear top or bottom emphasis.
- Inactive tabs remain readable with dark gray text.
- Hover state uses a subtle blue-gray fill.
- Avoid white text in tabs.

Typography rules:

- Use one enterprise UI font stack: Inter, Segoe UI, Roboto, Helvetica Neue, Arial, sans-serif.
- Workspace Title / Document Number: 20px, semi-bold.
- Customer Name: 16px, semi-bold, brand-colored, clickable.
- Section Headings: 14px, semi-bold, uppercase.
- Tab Labels: 13px, medium.
- Field Labels: 12px, medium.
- Body and table cells: 13px, regular.
- Captions: 11px, regular.
- Totals: 18px, bold, tabular numbers.
- Buttons and status badges: medium.
- Use consistent line heights and no negative letter spacing.

Reusable implementation:

- Tokens live in `src/index.css`.
- Workspace helpers live in `src/lib/transactionWorkspace.ts`.
- Shared document primitives live in `src/components/document`.
- Shared UI atoms live in `src/components/ui`.
- Sales Invoice must not carry its own private visual standard.

Table rules:

- Line table header row is slightly darker than the card background.
- Table borders are stronger but still subtle.
- Row hover uses a restrained blue-gray highlight.
- Text columns align left.
- Numeric columns align right and use tabular numbers.
- Totals use a stronger background and semi-bold values.

The header must not permanently include:

- Full workflow path
- Full approval history
- Full customer profile
- Created by
- Modified by
- Approved by
- Posted by
- GL lines
- Tax rows
- Related-document chain
- Attachments
- Notes
- Technical IDs
- RPC names
- Engine versions

Those belong in their dedicated tabs.

## 9. Toolbar And Action Hierarchy

The toolbar is the only permanent source of document actions. Do not create a duplicate Quick Actions card.

Typical actions:

| Mode or status | Actions |
| --- | --- |
| Create | Cancel, Save Draft, Save and New, Submit for Approval or Post, More |
| Draft | Cancel, Save Changes, Submit for Approval or Post, More |
| Approved | Post, Return to Draft when governed, Print, Email, More |
| Posted | Create Receipt, Print, Email, More |

Possible More actions:

- Duplicate
- Create Credit Memo
- Create Debit Memo
- View Customer
- View Customer Ledger
- Open Journal Entry
- Open Full Accounting Trace
- Void
- Reverse
- Cancel
- Export
- Generate E-Invoice, only when configured

Rules:

- Show no more than three prominent actions plus More.
- Actions are status-aware.
- Actions are permission-aware.
- Visual emphasis must match business importance:
  - Primary: Post
  - Secondary: Submit
  - Neutral: Save Draft
  - Text: Cancel
- Destructive actions require confirmation and a reason where applicable.
- Posted and frozen documents do not expose normal edit actions.
- Disabled actions explain why they are unavailable.
- Save and Post prevent duplicate submissions.

## 10. Three-Card Information Band

Use exactly three compact cards:

1. Document Information
2. Customer Information
3. Sales Context

Do not add a fourth card and do not create a permanent sidebar summary. All three cards must use the same spacing, label style, alignment, and density. They must not become mini dashboards.

Card visual guidance:

- Document Information uses a neutral surface.
- Customer Information may use a very subtle brand tint.
- Sales Context may use a very subtle neutral accent.
- Avoid strong colors and decorative graphics.
- Use whitespace rather than borders to create separation.
- Use comfortable enterprise spacing: 16-20px internal padding, consistent field spacing, and consistent vertical rhythm.
- Avoid compressed cards.

### Card 1: Document Information

Fields:

- Invoice Date
- Due Date
- Branch
- Currency
- Payment Terms
- External Reference
- VAT Price Basis

Behavior:

- Invoice date defaults from current date or company policy.
- Branch defaults from active context.
- Currency defaults from customer or company.
- Payment terms default from customer.
- Due date computes from invoice date and terms.
- External Reference captures a customer, vendor, logistics, contract, or other outside document reference.
- External Reference placeholder examples include `Customer PO`, `Supplier Invoice`, `Delivery Receipt`, `Contract`, and `Other External Reference`.
- VAT Price Basis allows prices to be encoded as VAT Exclusive or VAT Inclusive.
- VAT Price Basis should default from company, customer, transaction type, or document policy where authoritative.
- Changing VAT Price Basis recomputes Net Amount, VAT, Gross Amount, and Expected Net Collectible without changing the entered commercial price.
- VAT Price Basis is part of the document and should later be reused by Quotation, Sales Order, Purchase Order, Vendor Bill, Purchase Invoice, Credit Memo, Debit Memo, and purchase workflows.
- Posting period derives from invoice date and fiscal calendar.
- Closed periods produce blocking validation.
- Number series remains hidden unless the governed exception applies.

Do not place Customer lookup in Document Information.

### Card 2: Customer Information

Fields:

- Customer Lookup
- Customer Code
- TIN
- TIN Branch
- VAT Classification
- Business Style, only when useful

Behavior:

- Customer lookup is the only primary manual customer action.
- All other displayed values are derived or snapshot values.
- Customer Code, TIN, TIN Branch, and VAT Classification should use a balanced two-column grid with equal column widths.
- TIN displays the full PXL Philippine TIN format, such as `203-455-789-00000`.
- TIN Branch, when shown separately, displays the 5-digit branch code, such as `00000`.
- Never display or persist 3-digit or 4-digit branch identifiers.
- Follow `PXL_PHILIPPINE_TIN_STANDARD.md` for UI, search, storage, import/export, API, reporting, and future BIR modules.
- Use consistent spacing and vertical rhythm across readonly customer facts.
- Readonly fields must look readonly without looking disabled.
- Customer master changes occur through the Customer master, not through normal invoice entry.
- The card remains concise.
- Full current master information belongs in Related Party.

### Card 3: Sales Context

Purpose:

Sales Context identifies internal operational ownership and reporting dimensions for the transaction. It answers who inside the organization owns or is responsible for the transaction.

It must not duplicate customer information, tax information, payment information, branch identity, accounting impact, or audit data.

Supported and future dimensions may include:

- Account Owner
- Salesperson
- Department
- Project
- Cost Center
- Business Unit
- Location
- Campaign, future CRM
- Lead Source, future CRM

Behavior:

- Account Owner is the preferred operational owner field. It identifies the internal employee responsible for the customer relationship and should be selected from Employee Master. It is not necessarily the encoder, creator, or approver.
- Future implementations may inherit Account Owner automatically from Customer Master when that relationship is governed.
- Default configured dimensions from customer, employee, user, branch, project, campaign, lead source, or company setup where authoritative.
- Header defaults flow to lines.
- Line overrides require appropriate permissions or business rules.
- Override indicators appear when a line differs from the header.
- Show only configured dimensions.
- When dimensions are enabled for the company and document type, display editable selectors.
- When dimensions are disabled, hide them completely.
- Hide dimensions that do not exist or are not enabled.
- Do not display placeholder rows.
- Do not display instructional paragraphs inside the card.
- If no operational dimensions are assigned, show: `No operational dimensions assigned.`

Sales Context is the Sales Invoice presentation of the reusable Dimension Context (`DimensionContext`) component. The component stays standardized; only the available dimensions change.

Examples:

- Sales Invoice
- Purchase Invoice
- Vendor Bill
- Customer Payment
- Journal Entry
- Inventory Adjustment
- Expense
- Cash Receipt

Do not include in Sales Context:

- Customer Name
- Customer Code
- TIN
- VAT Classification
- Payment Terms
- Branch Code
- Customer Address
- Business Style
- Credit Limit
- Outstanding AR
- Customer Group
- Tax Profile
- Sales Order or Delivery Receipt links
- Payment links
- Customer contacts
- Audit data
- Accounting information
- Hardcoded or unsupported CRM fields
- Unsupported master-data values

Source Sales Orders, Delivery Receipts, and payment links belong in Related Docs.

### Number Series Behavior

Number Series must not normally appear as a user-editable field in the Sales Invoice create form.

Default behavior:

- PXL determines the correct series from company, branch, transaction type, date, and configuration.
- The document number is generated through the authoritative number-series process.
- The resulting number appears after save or when officially reserved, depending on system design.

Show a Number Series selector only when all of the following are true:

1. Multiple valid number series are configured.
2. The series are applicable to the selected branch and transaction type.
3. The user is authorized to select among them.
4. Series selection is a legitimate business requirement.

Otherwise:

- Hide the field from normal users.
- Show series metadata in System when operationally useful.
- Do not create manual numbering behavior outside governed controls.

## 11. Create-Mode Field Inventory

Primary design target:

> A standard Sales Invoice should normally require no more than five intentional header-level inputs, excluding line item selection, quantity, and necessary controlled overrides.

Typical intentional header inputs:

1. Customer
2. Invoice date, only when different from default
3. Branch, only when different from default
4. External Reference, when applicable
5. Optional VAT Price Basis or Dimension Context field not safely defaulted, such as an explicit VAT Price Basis selection, Account Owner, Department, Project, or Cost Center

Line-level business inputs typically include:

- Item or service
- Quantity
- Price only when no valid default exists or an authorized override is required
- Discount only when applicable
- Required dimensions only when not defaulted

Maximum complexity guardrail:

> No normal Sales Invoice should expose more than 15 intentional header-level inputs without progressive disclosure or a documented transaction-specific justification.

Five is the normal target. Fifteen is the maximum guardrail, not the target.

## 14. Invoice Snapshot Vs Current Customer Master

The standard must explicitly separate Invoice Snapshot from Current Customer Master.

This distinction is mandatory for audit integrity, invoice reprinting, tax support, and future e-invoicing.

### Invoice Snapshot

Invoice Snapshot fields preserve facts used when the transaction was created, saved, approved, or posted, as governed by the data model.

Typical snapshot fields include:

- Customer registered name used
- Trade or business name used
- Customer code
- TIN
- TIN Branch
- VAT classification
- Billing address used
- Delivery address used, when applicable
- Payment terms used
- Currency used
- VAT Price Basis used
- Tax treatment used

Relevant UI labels must use `Invoice Snapshot` where historical facts are being shown.

A posted invoice must not silently change its historical presentation because the customer master was edited later. Posted invoice review, printable documents, and tax/e-invoice outputs must use the authoritative transaction snapshot.

### Current Customer Master

Current Customer Master fields may include:

- Current credit limit
- Current outstanding AR
- Current available credit
- Current contacts
- Current addresses
- Current account status
- Current customer group
- Current salesperson
- Current price level
- Current recent transactions
- Current aging
- Current recent payments

The Related Party tab may show live current information, but it must label the source as `Current Customer Master` where relevant.

If the current schema does not preserve a required snapshot field, identify it as a data-model gap. Do not assume it already exists.

## 18. Customer Lookup And Customer-Change Policy

### Customer Lookup

The lookup must allow search by:

- Customer name
- Customer code
- TIN
- Trade name
- Contact
- Address, where practical

Enterprise lookup behavior:

- Clicking inside the Customer field immediately opens the searchable customer list.
- Typing filters the list in real time.
- Arrow keys navigate results.
- Enter selects the highlighted result.
- Escape closes the list.
- Clicking the dropdown icon opens the list.
- A clear affordance must make the field obviously searchable.

The lookup should show concise columns:

- Customer Code
- Customer Name
- TIN
- TIN Branch
- VAT Classification
- Status
- Outstanding Balance

After customer selection:

1. Refresh Customer Snapshot.
2. Refresh TIN.
3. Refresh TIN Branch.

1. A standard invoice normally requires no more than five intentional header-level inputs, excluding line item selection, quantity, and necessary controlled overrides.
2. No normal Sales Invoice exposes more than 15 intentional header-level inputs without progressive disclosure and documented justification.
3. Customer is selected in only one place.
4. Customer master data is not manually re-entered.
5. Invoice Snapshot and Current Customer Master are clearly distinguished.
6. Item, tax, account, pricing, and dimension defaults are retrieved automatically where authoritative.
7. Existing manual overrides are never silently replaced.
8. Changing the customer uses a controlled refresh policy.
9. The line grid remains the dominant work area.
10. No permanent right sidebar duplicates information.
11. Document actions appear only in the header toolbar.
12. The compact live summary does not duplicate the full Financial tab.
13. The readiness indicator remains visible during entry.
14. The Validation tab provides full explanations and resolutions.
15. Expected CWT and actual CWT are clearly distinguished.
16. Draft GL and tax information is labeled as preview.
17. Posted GL and tax information comes from authoritative records.
18. Customer details are consolidated in Related Party without becoming visually overwhelming.
19. Related documents use a canonical relationship table.
20. Audit history exists only in Audit.
21. Technical information exists only in System.
22. System metadata is displayed only when it already exists and is operationally useful.
23. Existing PXL shared transaction components are reused and extended.
24. The form remains responsive with at least 100 invoice lines.
25. Customer and item lookups begin showing results within 500 milliseconds under normal conditions.
26. Save and Post actions prevent duplicate submissions.
27. Unsaved changes are protected.
28. Failed saves preserve user-entered work.
29. Posted documents remain read-only except for governed downstream actions.
30. Technical identifiers, configuration keys, RPC names, and implementation metadata do not appear in the normal primary workspace.
31. GL Impact defaults to business-readable columns, with technical provenance moved to row detail or System.
32. VAT Price Basis is treated as a standard document field and uses VAT Exclusive or VAT Inclusive terminology.
33. Financial tab content is grouped into Revenue, Taxes, Invoice Summary, Collections, Expected Withholding, and Balance where applicable.
34. Readonly business fields remain readable and do not look like disabled controls.
35. Sales Context contains only operational ownership and reporting dimensions, not duplicated customer, tax, payment, branch, accounting, or audit information.
36. Sales Context hides unconfigured dimensions, shows no placeholder rows, and uses a concise empty state.
37. Account Owner is treated as business ownership from Employee Master or a governed customer ownership source.
38. Semantic status colors, typography hierarchy, and transaction design tokens are consistent.
39. The same architecture, including Dimension Context, can be reused across future transaction families.
40. Sales Invoice uses the system-wide Philippine TIN standard: `XXX-XXX-XXX-XXXXX`.

## 44. Rollout Rules For Future Transaction Forms

All future transaction workspaces must begin from this Sales Invoice architecture.

Rules:

- Reuse the same compact document header model.
- Reuse the same three-card information band.
- Reuse the same Dimension Context component; only the available dimensions should change by transaction type.
- Keep actions in the header toolbar only.
- Keep tabs one-line and responsibility-based.
- Keep line or detail records table-first.
- Keep GL impact in GL Impact.
- Keep tax impact in Tax Impact.
- Keep validation in Validation with compact readiness indicator support.
- Keep lifecycle in Workflow.
- Keep authorization in Approval or combined Workflow & Approval only when justified.
- Keep related entities in Related Party or the equivalent related-entity tab.
- Keep document relationships in Related Docs.
- Keep immutable audit facts in Audit.
- Keep technical metadata in System.
- Do not create permanent sidebars that duplicate content.
- Do not ask users to re-enter master data.
- Do not hide incomplete accounting or tax rules behind UI.
- Inherit the shared transaction design tokens and semantic color roles.
- Use concise enterprise empty states instead of long instructional paragraphs.
- Show truthful unavailable states for unsupported or unconfigured features.
- Reuse and extend existing PXL shared transaction components before proposing new ones.

Transaction-specific adaptations are allowed only for business-specific fields, line types, actions, validations, and related-document relationships. The overall workspace model must remain consistent across PXL.

Recommendation: Approve this revised standard as the canonical Sales Invoice Create and Draft-Edit UX model for PXL, paired with `PXL_SALES_INVOICE_VIEW_UX_STANDARD.md` for saved-document review, approval, posting, collection, void/reversal, drilldown, and audit UX, subject to confirmation of the listed accounting, tax, master-data, and authoritative-source gaps.
