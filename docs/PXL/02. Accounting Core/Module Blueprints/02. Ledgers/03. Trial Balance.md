## Module Overview
The Trial Balance is a foundational accounting report that lists the ending balances of all General Ledger accounts at a specific point in time. In the Philippine accounting and audit context, it is the absolute prerequisite for generating the Income Statement and Balance Sheet. Its primary purpose is to mathematically prove that Total Debits equal Total Credits across the entire ledger, ensuring that the double-entry system has been strictly maintained. It serves as the primary working paper for external auditors.

## Dashboard UI
The Trial Balance is a dynamic, generation-based report view rather than a data entry interface.

### The Action Bar
* **As-Of Date Picker:** A date selector to define the cutoff date for the balances (e.g., December 31, 2026).
* **View Type Toggle:** 
  - **Summary:** Shows only accounts with non-zero balances.
  - **Detailed:** Shows all accounts, including zero-balance accounts.
  - **Working Trial Balance:** Includes columns for Adjusting Journal Entries (AJEs) (Unadjusted Balance -> Adjustments -> Adjusted Balance).
* **Export Button:** Exports to Excel/CSV. Critical feature: The Excel export must maintain the hierarchical structure (Assets -> Current Assets -> Cash) for easy formatting.
* **Print Button:** Generates a PDF.
* **Recalculate Button:** Refreshes the view to include the latest posted transactions.

### The Data List
The data is typically presented in a strict Chart of Accounts hierarchical order (Assets, Liabilities, Equity, Revenue, Expenses).
* **Account Code:** The numeric GL code.
* **Account Name:** The name of the account.
* **Debit:** The total debit balance (if the net balance is debit).
* **Credit:** The total credit balance (if the net balance is credit).

At the very bottom, a **Grand Total** row strictly validates that `Total Debit = Total Credit`. If there is a variance, a red alert must be displayed.

## Supabase Database Architecture

Like the Account Detail Ledger, the Trial Balance does not have its own persistent data table. It is a highly optimized query built on top of the `general_ledger_entries` and `chart_of_accounts`.

### Data Flow & Logic
To generate the Trial Balance efficiently:
1.  **Join COA & GL:** The system queries all active accounts from the `chart_of_accounts`.
2.  **Aggregate:** It performs a `SUM(debit) - SUM(credit)` from `general_ledger_entries` for each `account_id` where the `posting_date <= As-Of Date`.
3.  **Determine Column:** Based on the account's normal balance (e.g., Asset = Debit, Liability = Credit) and the resulting algebraic sign, the absolute value is placed in either the Debit or Credit presentation column.
4.  **Tree Aggregation:** Balances from child accounts (postable) are rolled up to their parent accounts (header accounts) for hierarchical display.

```sql
-- Conceptual View for Trial Balance Generation
CREATE OR REPLACE VIEW trial_balance_view AS
SELECT 
    coa.account_code,
    coa.account_name,
    coa.account_type,
    SUM(gle.debit) as total_debit,
    SUM(gle.credit) as total_credit,
    SUM(gle.debit) - SUM(gle.credit) as net_balance
FROM chart_of_accounts coa
LEFT JOIN general_ledger_entries gle 
    ON coa.id = gle.account_id 
    AND gle.posting_date <= CURRENT_DATE -- Parameterized in actual app
GROUP BY coa.account_code, coa.account_name, coa.account_type;
```
