## Module Overview
The Amortization Schedules module automates the gradual expensing of prepaid assets (rent, insurance, software subscriptions) over their useful life. Recording a massive prepayment as a single-month expense distorts the Income Statement. This module tracks the original prepayment, the number of periods, and automatically generates the monthly journal entry that shifts value from the Prepaid Asset account (Balance Sheet) to the Expense account (Income Statement), enforcing strict matching principle compliance.

On each period-end run, the system posts a JE: Dr. Expense Account / Cr. Prepaid Asset Account — automatically, without manual input.

## Dashboard UI

The interface is a registry of all active, completed, and paused amortization schedules.

### The Action Bar
* **Search Bar:** Search by Schedule Name, Source Document, or Reference.
* **Filter:** By Status (Active, Completed, Paused).
* **Export Button:** Export the schedule list for audit working papers.
* **Create Schedule Button:** Manually create a new schedule (can also be auto-created from AP Bills).

### The Data List
* Schedule ID
* Name / Description
* Asset Account (the Prepaid GL account)
* Expense Account (target expense GL)
* Total Amount
* Amortized Amount to Date
* Remaining Balance
* Start Date / End Date
* Status (Active / Completed / Paused)

**Pagination Mandate:** Server-side pagination via Supabase `.range(from, to)`. Default page size: 50.

## Data Fields

### Header Level

| Field Name | Description | Required? |
| :--- | :--- | :--- |
| Schedule Name | Text describing the amortization (e.g., "Annual Fire Insurance 2026"). | Yes |
| Source Document Type | The originating document (AP Bill, Journal Entry, Manual). | Yes |
| Source Document ID | FK to the originating document. | Conditional |
| Prepaid Asset Account | GL account holding the prepaid balance (must be an Asset account). | Yes |
| Expense Account | Target GL expense account (must be an Expense account). | Yes |
| Branch ID | The branch this schedule belongs to. | Yes |
| Total Amount | The principal amount to be amortized over the full period. | Yes |
| Start Date | The first period to recognize expense (e.g., 2026-01-01). | Yes |
| End Date | The last period to recognize expense (e.g., 2026-12-31). | Yes |
| Number of Periods | Auto-calculated from Start Date and End Date. | Auto |
| Monthly Amount | `Total Amount ÷ Number of Periods`. Last period may vary by rounding. | Auto |
| Amortized Amount | Running total of all posted period amounts. | Auto |
| Remaining Balance | `Total Amount - Amortized Amount`. | Auto |
| Status | ENUM: `active`, `completed`, `paused`. | Auto |

### Line Items (The Schedule)
When the header is saved, the system automatically generates one line per period.

| Field Name | Description | Required? |
| :--- | :--- | :--- |
| Period Date | The period-end date the JE should post (e.g., 2026-01-31). | Auto |
| Amount | The pro-rata expense for this period. | Auto |
| Journal Entry ID | FK to `journal_entries.id` after the period has been run. | Auto |
| Status | ENUM: `scheduled`, `posted`, `skipped`. | Auto |

## Supabase Database Architecture

### Table 1: `amortization_schedule_headers`

**Critical Database Rules:**
1. `UNIQUE(company_id, schedule_no)` — sequential numbering per tenant.
2. Once `status = 'completed'`, no modifications are allowed (RLS block).
3. `source_document_id` enables drill-back from an amortization schedule to the AP Bill that created it (Principle 7).

| Column Name | Data Type | Rules | What it stores |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System ID. |
| `company_id` | UUID | Required, Foreign Key | Links to `companies.id`. |
| `branch_id` | UUID | Required, Foreign Key | Links to `branches.id`. |
| `schedule_no` | Text | Required | `UNIQUE(company_id, schedule_no)`. |
| `schedule_name` | Text | Required | Descriptive name. |
| `source_document_type` | Text | Nullable | ENUM: `ap_bill`, `journal_entry`, `manual`. |
| `source_document_id` | UUID | Nullable | FK to the originating document. |
| `prepaid_asset_account_id` | UUID | Required, Foreign Key | Links to `chart_of_accounts.id`. Must be Asset type. |
| `expense_account_id` | UUID | Required, Foreign Key | Links to `chart_of_accounts.id`. Must be Expense type. |
| `total_amount` | Numeric(15,2) | Required | Full amount to be amortized. |
| `amortized_amount` | Numeric(15,2) | Required, Default 0 | Running total of posted amounts. |
| `remaining_balance` | Numeric(15,2) | Required | `total_amount - amortized_amount`. |
| `start_date` | Date | Required | First amortization period. |
| `end_date` | Date | Required | Last amortization period. |
| `number_of_periods` | Integer | Required | Auto-calculated total periods. |
| `monthly_amount` | Numeric(15,2) | Required | `total_amount / number_of_periods`. |
| `status` | Text | Required, Default `active` | ENUM: `active`, `completed`, `paused`. |
| `created_by` | UUID | Required, Foreign Key | Links to `users.id`. |
| `updated_by` | UUID | Nullable, Foreign Key | Links to `users.id`. |
| `created_at` | Timestamp | Auto | Creation timestamp. |
| `updated_at` | Timestamp | Auto | Last update timestamp. |

## Row-Level Security

```sql
-- Block modifications on completed schedules
CREATE POLICY "block_update_on_completed_amortization"
ON amortization_schedule_headers
FOR UPDATE
USING (status IN ('active', 'paused'));

-- No deletes
CREATE POLICY "block_delete_amortization"
ON amortization_schedule_headers
FOR DELETE
USING (false);

-- Tenant isolation
CREATE POLICY "tenant_isolation_amortization"
ON amortization_schedule_headers
FOR ALL
USING (company_id = (auth.jwt() ->> 'company_id')::UUID);
```
