## Module Overview
The Revenue Recognition Run is the sister module to the Amortization Run. Used during the month-end close, this batch-processing engine scans all active `revenue_recognition_schedules`. It identifies the unearned revenue slated for recognition in the current period, automatically generates the necessary Journal Entries, and shifts the balances from the Deferred Revenue (Liability) account to the Service Income (Revenue) account. This ensures strict adherence to PFRS 15 matching principles without manual data entry for every customer contract.

## Dashboard UI
A wizard-like interface identical in flow to the Amortization Run, designed for month-end batch execution.

### The Action Bar
* **Target Period:** A Month/Year selector (e.g., "January 2026").
* **Preview Run Button:** Generates a draft list of the revenue to be recognized.
* **Execute Run Button:** Commits the batch to the General Ledger.

### The Data List (Preview Mode)
* **Contract/Customer:** The source schedule and tagged customer.
* **Deferred Account:** The GL Liability account to be debited (decreased).
* **Revenue Account:** The GL Income account to be credited (increased).
* **Amount:** The calculated revenue to recognize for this month.
* **Status:** "Ready to Post" or "Error".

## Supabase Database Architecture

The execution logic directly mirrors the Amortization Run, operating on the revenue tables.

### Data Flow & Logic
1.  **Identify Targets:** The system queries `revenue_recognition_lines` where `period_date` falls within the Target Period AND `status = 'pending'`.
2.  **Generate JEs:** For each valid line, the system constructs a Journal Entry:
    *   **Header:** Date = `Posting Date`, Description = "Rev Rec for [Description] - [Period]".
    *   **Debit Line:** Account = `deferred_account_id` from the schedule header, Amount = `amount` (Reduces Liability).
    *   **Credit Line:** Account = `revenue_account_id` from the schedule header, Amount = `amount` (Increases Income).
    *   *Validation Check:* Total Debit = Total Credit.
3.  **Commit:** The system inserts the JEs, posts them to `general_ledger_entries`.
4.  **Update Schedule:** The `revenue_recognition_lines` record is updated: `status = 'posted'`, and linked to the `journal_entry_id`.
5.  **Audit Log:** Records the execution in the `batch_job_logs` table (Job Type = `rev_rec_run`).
