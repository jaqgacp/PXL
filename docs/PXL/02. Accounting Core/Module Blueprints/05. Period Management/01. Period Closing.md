## Module Overview
The Period Closing module governs the workflow for finalizing financial periods (monthly, quarterly, or annually). Once financial statements are issued to management or the BIR, the underlying data must not change. This module acts as a strict gateway — enforcing a checklist of prerequisites before a Chief Accountant or Controller can officially close a period.

**Architecture Decision — Single Source of Truth for Period Locks:**
`accounting_periods` is the **master layer** for period locking. It is checked first and overrides all other locks. `fiscal_locks` provides granular module-level locks within open periods (e.g., AP locked on the 3rd, AR locked on the 5th). The hierarchy is:
1. `accounting_periods.is_closed = true` → **Hard Block (whole month, all modules, all roles).**
2. `fiscal_locks.lock_date` → **Module Block (specific transaction types, within an open period).**
3. `period_controls.status` → **Soft advisory status used for UI display only.** Does not block transactions independently.

Any code that validates a posting date must check these two layers in this exact order.

## Supabase Database Architecture

### Table 1: `accounting_periods`

**This is the master period lock table. All posting modules query this table first.**

**Critical Database Rules:**
1. Composite unique constraint on `(company_id, period_year, period_month)` — one record per month per company.
2. `is_closed` defaults to `false`. Once set to `true`, only a `super_admin` role with a logged justification can revert it.
3. `company_id` is mandatory — this is a multi-tenant system. Period status is per-company.

| Column Name | Data Type | Rules | Description |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System identifier. |
| `company_id` | UUID | Required, Foreign Key | Links to `companies.id`. Tenant isolation anchor. |
| `fiscal_year_id` | UUID | Required, Foreign Key | Links to `fiscal_years.id`. |
| `period_name` | Text | Required | Display name, e.g., "January 2026". |
| `period_year` | Integer | Required | E.g., `2026`. |
| `period_month` | Integer | Required | 1–12. |
| `start_date` | Date | Required | First day of the period. |
| `end_date` | Date | Required | Last day of the period. |
| `status` | Text | Required, Default `open` | Advisory display status: `open`, `pending_review`, `closed`. |
| `is_closed` | Boolean | Required, Default `false` | **MASTER LOCK.** If `true`, no transaction with a date in this period can be posted by any module. |
| `closed_at` | Timestamp | Nullable | Exact timestamp the period was locked. Set by system. |
| `closed_by` | UUID | Nullable, Foreign Key | Links to `users.id`. Set by system on close action. |
| `reopen_justification` | Text | Nullable | Required text when a super-admin reopens a closed period. |
| `created_by` | UUID | Required, Foreign Key | Links to `users.id`. |
| `updated_by` | UUID | Nullable, Foreign Key | Links to `users.id`. |
| `created_at` | Timestamp | Auto | Creation timestamp. |
| `updated_at` | Timestamp | Auto | Last update timestamp. |

**Composite Unique Constraint:** `UNIQUE(company_id, period_year, period_month)`

### Global Posting Rule — Application Logic (Master Layer)

Every module capable of posting to `general_ledger_entries` (Invoices, Bills, JEs, Payments) must execute this pre-flight check **first**, before any other validation:

```sql
-- STEP 1: Check the master period lock (accounting_periods).
-- This is the first and highest-priority check.
SELECT is_closed
FROM accounting_periods
WHERE company_id = :company_id
  AND :transaction_date BETWEEN start_date AND end_date;

-- If is_closed = true:
--   ABORT. Return error: "Cannot post to a closed accounting period. Contact your Controller."
--   No further checks are needed. No exceptions, no role overrides for is_closed.

-- STEP 2: If period is open, check module-specific fiscal lock (fiscal_locks).
SELECT lock_date
FROM fiscal_locks
WHERE company_id = :company_id
  AND module_name = :posting_module;

-- If lock_date IS NOT NULL AND :transaction_date <= lock_date:
--   ABORT. Return error: "The [module] is locked for dates on or before [lock_date]."

-- STEP 3: If both checks pass, allow the posting to proceed.
```

**Precedence Summary:**
| Layer | Table | Wins Against |
| :--- | :--- | :--- |
| 1st (Hardest) | `accounting_periods.is_closed` | Everything. No override. |
| 2nd | `fiscal_locks.lock_date` | Module-level. Overridable by super-admin only. |
| 3rd (Softest) | `period_controls.status` | UI advisory only. Does not block transactions. |
