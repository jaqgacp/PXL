## Module Overview
The Inventory Movements module acts as the immutable general ledger for all stock. **CRITICAL ARCHITECTURAL RULE:** Stock levels are NEVER directly updated. Every change in stock quantity must insert a record here. This table is tightly tied to Purchasing (Purchase Receipts) and Sales (Delivery Receipts / Sales Invoices), ensuring absolute integrity between financial transactions and physical stock balances. This is vital for Philippine BIR audits, allowing full traceability from physical stock back to source VAT documents.

## What You Will See on the Screen (The Dashboard)
When you open "Inventory Movements", you will see a read-only, chronological ledger of all stock ins and outs.

### The Action Bar (Top of the page):
* **Search Bar:** Search by Item Code, Reference Number (e.g., SI-001, PR-002).
* **Filter:** Filter by Date Range, Warehouse, Item Category, and Transaction Type (Sales, Purchase, Transfer, Adjustment).
* **Customize View:** Select data columns.
* **Export Button:** Download the filtered ledger to Excel/CSV for audit trailing.
* **Import Button:** Not applicable (Movements cannot be manually imported directly; they are system-generated).

### The Data List (The main table):
Columns:
* Date
* Transaction Type (Purchase Receipt, Sales Delivery, Adjustment, Transfer)
* Reference Number (Clickable link to source document)
* Warehouse
* Item Code & Name
* Quantity Change (+ in, - out)
* Running Balance

## What Information We Need to Capture (The Data Fields)
*Note: This is a system-generated table. No manual data entry form exists for this module. Data is inherited directly from approved Sales, Purchasing, and Inventory transactions.*

| Field Name | Description | Source / Inheritance |
| :--- | :--- | :--- |
| Posting Date | Date the physical movement happened. | Auto-filled from Source Document Date |
| Transaction Type | What caused the movement? | Auto-filled from System Context (e.g., `purchase_receipt`) |
| Source Document ID| The UUID of the originating document. | Auto-filled from Source Header ID |
| Item | The item being moved. | Auto-filled from Source Line Item |
| Warehouse | Location the item was moved to/from. | Auto-filled from Source Header or Line Item |
| Quantity | Amount. Positive for IN, Negative for OUT. | Auto-filled from Source Line Item |
| Unit Cost | Valuation at the time of movement. | Auto-filled based on Costing Method computation |

## The Supabase Database Architecture

### Table: `inventory_movements` (The Stock Ledger)
This is the master table for all quantitative stock tracking.

| Column Name | Data Type | Rules | What it stores |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System ID for the movement. |
| `posting_date` | Date | Required | Date the inventory change is recognized. |
| `transaction_type` | Text | Required | e.g., `purchase_receipt`, `sales_delivery`, `stock_adjustment`, `stock_transfer`, `goods_issue`. |
| `reference_id` | UUID | Required | The ID of the document that caused this (e.g., `sales_deliveries.id`, `purchase_receipts.id`). |
| `reference_number`| Text | Required | Human-readable document number (e.g., "DR-2026-001"). |
| `item_id` | UUID | Required, Foreign Key | Links to `items.id`. |
| `warehouse_id` | UUID | Required, Foreign Key | Links to `warehouses.id`. |
| `quantity` | Numeric(15,4) | Required | Positive for stock IN, Negative for stock OUT. |
| `uom` | Text | Required | Unit of Measure at time of transaction. |
| `unit_cost` | Numeric(15,4) | Required | Cost per unit at time of transaction (Crucial for COGS calculation). |
| `total_cost` | Numeric(15,4) | Required | `quantity * unit_cost`. |
| `created_by` | UUID | Required, Foreign Key | User who approved the source document. |
| `updated_by` | UUID | Nullable, Foreign Key | Automatically filled if source is amended (though usually immutable). |
| `created_at` | Timestamp | Auto | Date record was created. |
| `updated_at` | Timestamp | Auto | Date record was last edited. |


**Critical UX/GL Rule:** The system MUST auto-fetch the Item's default Revenue, Expense, COGS, and Inventory GL accounts in the background to auto-generate the Journal Entry. Encoders MUST NEVER manually select a GL account for standard items.
