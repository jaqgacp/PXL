## Module Overview
The Stock Adjustment module allows authorized users to correct inventory levels when there are discrepancies between system records and actual physical stock. This is crucial for documenting write-offs, damages, spoilage, or administrative data entry errors. For Philippine accounting, undocumented stock discrepancies can trigger BIR penalties during audits; thus, every adjustment must capture a specific reason code and immediately update the `inventory_movements` ledger to ensure exact traceability.

## What Information We Need to Capture (The Data Fields)

### Header Information
| Field Name | Description | Required? | UI Component / Inheritance |
| :--- | :--- | :--- | :--- |
| Adjustment Number | Auto-generated ID (e.g., SA-2026-001). | System | Auto-computed |
| Posting Date | Date the adjustment takes effect. | Yes | Date Picker |
| Warehouse | The specific warehouse being adjusted. | Yes | Dropdown (Links to Warehouses) |
| Reason Code | Why is the adjustment being made? (e.g., Damage, Spoilage, Found). | Yes | Dropdown linked to Reason Codes |
| Remarks | Detailed explanation for the auditor. | Optional | Text Area |

### Line Items
| Field Name | Description | Required? | UI Component / Inheritance |
| :--- | :--- | :--- | :--- |
| Item Code | Select item. | Yes | Dropdown linked to Items. UPON SELECTION, instantly auto-fills Description, UOM, Unit Price/Cost, and Tax Code. |
| Current Quantity | System's current quantity. | System | Auto-filled from `inventory_movements` |
| New Quantity | The actual physical count. | Yes | Number Input |
| Adjusted Qty | Difference (New - Current). Positive or Negative. | System | Auto-computed |
| Unit Cost | Value of the adjusted stock. | System | Auto-filled (FIFO/Weighted Avg) |

## The Supabase Database Architecture

### Table 1: `stock_adjustments` (Header)
| Column Name | Data Type | Rules | What it stores |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System ID. |
| `adjustment_number`| Text | Required, Unique | E.g., "SA-2026-001". |
| `posting_date` | Date | Required | Date the adjustment is recorded. |
| `warehouse_id` | UUID | Required, Foreign Key | Links to `warehouses.id`. |
| `reason_code` | Text | Required | E.g., `damage`, `spoilage`, `shrinkage`, `found_stock`. |
| `remarks` | Text | Nullable | Explanatory notes. |
| `status` | Text | Required | `draft`, `approved`, `cancelled`. |
| `created_by` | UUID | Required, Foreign Key | Links to `users.id`. |
| `updated_by` | UUID | Nullable, Foreign Key | Links to `users.id`. |
| `created_at` | Timestamp | Auto | Date record was created. |
| `updated_at` | Timestamp | Auto | Date record was last edited. |

### Table 2: `stock_adjustment_items` (Line Items)
| Column Name | Data Type | Rules | What it stores |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System ID. |
| `adjustment_id` | UUID | Required, Foreign Key | Links to `stock_adjustments.id`. |
| `item_id` | UUID | Required, Foreign Key | Links to `items.id`. |
| `system_qty` | Numeric(15,4) | Required | Snapshot of quantity before adjustment. |
| `actual_qty` | Numeric(15,4) | Required | The inputted physical quantity. |
| `adjusted_qty` | Numeric(15,4) | Required | `actual_qty - system_qty`. (Sent to `inventory_movements`). |
| `unit_cost` | Numeric(15,4) | Required | Cost assigned to the variance. |
| `created_by` | UUID | Required, Foreign Key | Links to `users.id`. |
| `updated_by` | UUID | Nullable, Foreign Key | Links to `users.id`. |
| `created_at` | Timestamp | Auto | Date record was created. |
| `updated_at` | Timestamp | Auto | Date record was last edited. |

**Crucial Logic:** Upon approval (Status = `approved`), the system MUST automatically insert records into `inventory_movements` using the `adjusted_qty` for each line item.

**Critical UX/GL Rule:** The system MUST auto-fetch the Item's default Revenue, Expense, COGS, and Inventory GL accounts in the background to auto-generate the Journal Entry. Encoders MUST NEVER manually select a GL account for standard items.
