# PXL ERP Blueprint: Receiving Reports

## Module Overview
The Receiving Report (RR) module documents the physical receipt of goods from suppliers. In Philippine business processes, this relies heavily on the Delivery Receipt (DR) provided by the supplier. The RR serves as proof that items have been delivered and accepted by the warehouse or receiving department. It updates inventory quantities and acts as the crucial matching document against the Purchase Order and the forthcoming Vendor Bill (3-way matching) to ensure the company only pays for what it actually received.

## What You Will See on the Screen (The Dashboard)
When you open the "Receiving Reports" page, you will see a clean workspace.

### The Action Bar (Top of the page):
* **Search Bar:** Quickly find an RR by RR Number, Supplier Name, or Supplier DR Number.
* **Filter:** Dropdowns to filter by Date Range and Status (Draft, Received, Billed, Cancelled).
* **Customize View:** Choose which columns to show or hide.
* **Export Button:** Downloads the list to Excel or CSV.
* **Print Button:** Generates a print-ready view of the RR list.
* **Create New RR Button:** Prominent button to log a new receipt of goods.
* **Import Button:** Bulk upload Receiving Reports from an Excel/CSV file.

### The Data List (The main table):
* RR Date
* RR Number
* Supplier Name
* Reference PO Number
* Supplier DR Number
* Status (Draft, Received, Billed, Cancelled)

## What Information We Need to Capture (The Data Fields)
When creating or editing an RR, the form is divided into Header and Line Items.

**Data Inheritance:** A Receiving Report is almost always generated by pulling from an approved Purchase Order (PO). The system will auto-populate the Supplier and expected items/quantities. The user only needs to verify and adjust the actual received quantities.

### Section 1: Header Details
| Field Name | Description | UI Component | Required? | Inheritance |
| :--- | :--- | :--- | :--- | :--- |
| RR Number | Unique, auto-generated system number. | Auto-computed | Yes | None |
| RR Date | Date the goods were physically received. | Date Picker | Yes | None |
| Reference PO | The Purchase Order this delivery fulfills. | Search/Dropdown | Yes | None |
| Supplier | The vendor delivering the goods. | Dropdown linked to Master Data. UPON SELECTION, system MUST instantly auto-fill: Address, TIN, Credit Terms, and Tax Type. (Auto-filled if from PO) | Yes | Auto-filled from Reference PO |
| Supplier DR No. | Delivery Receipt number from the supplier's paperwork. | Text Input | Yes | None |
| Receiving Location | Warehouse or branch receiving the goods. | Dropdown | Yes | Default to user's location / PO delivery address |
| Status | Current state of the RR. | Auto-computed | Yes | None |
| Remarks | Notes on delivery condition (e.g., partial delivery). | Text Area | Optional | None |

### Section 2: Line Items
| Field Name | Description | UI Component | Required? | Inheritance |
| :--- | :--- | :--- | :--- | :--- |
| Item/Service | The product being received. | Dropdown linked to Items. UPON SELECTION, instantly auto-fills Description, UOM, Unit Price/Cost, and Tax Code. | Yes | Auto-filled from PO |
| Description | Detailed description of the item. | Auto-computed | Optional | Auto-filled from PO |
| Ordered Qty | The original amount ordered. | Read-only | Yes | Auto-filled from PO |
| Previously Received| Quantity already received in past RRs. | Read-only | Yes | Auto-computed by system |
| Received Qty | Actual quantity received in this delivery. | Number Input | Yes | User inputs, defaults to remaining balance |
| UOM | Unit of Measure. | Dropdown linked to UOM master | Yes | Auto-filled from PO |
| Reject Qty | Items rejected due to damage or wrong spec. | Number Input | Optional | None |

## The Supabase Database Architecture

**Critical Database Rule:**
- Ensure a composite unique constraint on `company_id` and `rr_number` if a `company_id` is used, otherwise `rr_number` must be unique.

**Critical UX/GL Rule:** The system MUST auto-fetch the Item's default Revenue, Expense, COGS, and Inventory GL accounts in the background to auto-generate the Journal Entry. Encoders MUST NEVER manually select a GL account for standard items.

### Table 1: `receiving_reports` (Header Table)
| Column Name | Data Type | Rules | What it stores |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System ID for the RR. |
| `rr_number` | Text | Required, Unique | Auto-generated number (e.g., "RR-2023-0001"). |
| `rr_date` | Date | Required | Date goods were received. |
| `po_id` | UUID | Required, Foreign Key | Links to `purchase_orders.id`. |
| `supplier_id` | UUID | Required, Foreign Key | Links to `suppliers.id`. |
| `supplier_dr_no` | Text | Required | Supplier's Delivery Receipt number. |
| `location_id` | UUID | Required, Foreign Key | Links to `locations.id` (warehouse). |
| `status` | Text | Required | `draft`, `received`, `billed`, `cancelled`. |
| `remarks` | Text | Nullable | Observations on the delivery. |
| `created_by` | UUID | Required, Foreign Key | Links to `users.id`. |
| `updated_by` | UUID | Nullable, Foreign Key | Links to `users.id`. |
| `created_at` | Timestamp | Auto | Record creation timestamp. |
| `updated_at` | Timestamp | Auto | Record update timestamp. |

### Table 2: `receiving_report_lines` (Line Items Table)
| Column Name | Data Type | Rules | What it stores |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System ID for the line item. |
| `rr_id` | UUID | Required, Foreign Key | Links to `receiving_reports.id`. |
| `po_line_id` | UUID | Required, Foreign Key | Links to `purchase_order_lines.id` for exact matching. |
| `item_id` | UUID | Required, Foreign Key | Links to `items.id`. |
| `received_qty` | Numeric | Required | Quantity accepted. |
| `reject_qty` | Numeric | Default: 0 | Quantity rejected. |
| `uom_id` | UUID | Required, Foreign Key | Links to `uoms.id`. |
| `created_by` | UUID | Required, Foreign Key | Links to `users.id`. |
| `updated_by` | UUID | Nullable, Foreign Key | Links to `users.id`. |
| `created_at` | Timestamp | Auto | Record creation timestamp. |
| `updated_at` | Timestamp | Auto | Record update timestamp. |
