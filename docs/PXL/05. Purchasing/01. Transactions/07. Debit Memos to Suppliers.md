# PXL ERP Blueprint: Debit Memos to Suppliers

## Module Overview
The Debit Memo to Suppliers module is used when the company (the buyer) notifies a supplier of a deduction in the amount owed to them. This often occurs due to damaged goods, shortages in delivery, pricing errors, or penalties. Unlike a Vendor Credit (which is issued by the supplier), the Debit Memo is initiated internally. In Philippine practice, issuing a Debit Memo typically prompts the supplier to issue a corresponding Credit Memo to formally adjust the Accounts Payable and Input VAT records.

## What You Will See on the Screen (The Dashboard)
When you open the "Debit Memos" page, you will see a clean workspace.

### The Action Bar (Top of the page):
* **Search Bar:** Quickly find a Memo by DM Number or Supplier Name.
* **Filter:** Dropdowns to filter by Date Range and Status (Draft, Sent, Acknowledged, Cancelled).
* **Customize View:** Choose which columns to show or hide.
* **Export Button:** Downloads the list to Excel or CSV.
* **Print Button:** Generates a print-ready view to send to the supplier.
* **Create New DM Button:** Prominent button to issue a new debit memo.

### The Data List (The main table):
* DM Date
* DM Number
* Supplier Name
* Reference Document (Bill/RR)
* Total Amount
* Status (Draft, Sent, Acknowledged, Cancelled)

## What Information We Need to Capture (The Data Fields)
The form is divided into Header and Line Items.

**Data Inheritance:** A Debit Memo is typically linked to a specific Vendor Bill or Receiving Report. The items and prices are auto-filled from the source document to ensure accurate chargebacks.

### Section 1: Header Details
| Field Name | Description | UI Component | Required? | Inheritance |
| :--- | :--- | :--- | :--- | :--- |
| DM Number | Unique, auto-generated system number. | Auto-computed | Yes | None |
| DM Date | Date the memo is issued. | Date Picker | Yes | None |
| Supplier | The vendor being charged back. | Dropdown linked to Master Data. UPON SELECTION, system MUST instantly auto-fill: Address, TIN, Credit Terms, and Tax Type. | Yes | Auto-filled from source doc |
| Reference Doc | The RR or Bill this memo applies to. | Search/Dropdown | Optional | None |
| Status | Current state of the memo. | Auto-computed | Yes | None |
| Reason | Explanation for the deduction. | Text Area | Yes | None |

### Section 2: Line Items
| Field Name | Description | UI Component | Required? | Inheritance |
| :--- | :--- | :--- | :--- | :--- |
| Item | The Item to deduct. | Dropdown linked to Items. UPON SELECTION, instantly auto-fills Description, UOM, Unit Price/Cost, and Tax Code. | Yes | Auto-filled from Reference Doc |
| Description | Details of the deduction. | Text Area | Optional | Auto-filled from Reference Doc |
| Quantity | Affected quantity (e.g., returned qty). | Number Input | Optional | None |
| Unit Price | Amount per unit. | Number Input | Optional | Auto-filled from Reference Doc |
| Total Amount | Total chargeback amount. | Auto-computed | Yes | None |

## The Supabase Database Architecture (For Lovable)

**Critical Database Rule for Lovable:**
- Ensure a composite unique constraint on `company_id` and `dm_number` if a `company_id` is used, otherwise `dm_number` must be unique.

**Critical UX/GL Rule:** The system MUST auto-fetch the Item's default Revenue, Expense, COGS, and Inventory GL accounts in the background to auto-generate the Journal Entry. Encoders MUST NEVER manually select a GL account for standard items.

### Table 1: `supplier_debit_memos` (Header Table)
| Column Name | Data Type | Rules | What it stores |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System ID for the Debit Memo. |
| `dm_number` | Text | Required, Unique | Auto-generated number (e.g., "DM-2023-0001"). |
| `dm_date` | Date | Required | Date issued. |
| `supplier_id` | UUID | Required, Foreign Key | Links to `suppliers.id`. |
| `reference_doc_id` | UUID | Nullable | Links to RR or Bill ID. |
| `reference_doc_type`| Text | Nullable | `receiving_report` or `vendor_bill`. |
| `status` | Text | Required | `draft`, `sent`, `acknowledged`, `cancelled`. |
| `total_amount` | Numeric | Required | Total value of the memo. |
| `reason` | Text | Required | Explanation for chargeback. |
| `created_by` | UUID | Required, Foreign Key | Links to `users.id`. |
| `updated_by` | UUID | Nullable, Foreign Key | Links to `users.id`. |
| `created_at` | Timestamp | Auto | Record creation timestamp. |
| `updated_at` | Timestamp | Auto | Record update timestamp. |

### Table 2: `supplier_debit_memo_lines` (Line Items Table)
| Column Name | Data Type | Rules | What it stores |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System ID for the line item. |
| `dm_id` | UUID | Required, Foreign Key | Links to `supplier_debit_memos.id`. |
| `item_id` | UUID | Required, Foreign Key | Links to `items.id`. |
| `description` | Text | Nullable | Line detail. |
| `quantity` | Numeric | Nullable | Affected quantity. |
| `unit_price` | Numeric | Nullable | Amount per unit. |
| `total_amount` | Numeric | Required | Computed line amount. |
| `created_by` | UUID | Required, Foreign Key | Links to `users.id`. |
| `updated_by` | UUID | Nullable, Foreign Key | Links to `users.id`. |
| `created_at` | Timestamp | Auto | Record creation timestamp. |
| `updated_at` | Timestamp | Auto | Record update timestamp. |
