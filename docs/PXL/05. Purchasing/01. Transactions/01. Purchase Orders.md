# PXL ERP Blueprint: Purchase Orders

## Module Overview
The Purchase Order (PO) module initiates the formal buying process in the Philippines. It is a legally binding document sent to a supplier confirming the intent to purchase goods or services at agreed-upon prices and terms. While it does not generate an immediate accounting entry, it is critical for internal control, budget tracking, and matching against future Receiving Reports and Vendor Bills to prevent overbilling or unauthorized purchases.

## What You Will See on the Screen (The Dashboard)
When you open the "Purchase Orders" page, you will see a clean workspace.

### The Action Bar (Top of the page):
* **Search Bar:** Quickly find a PO by PO Number or Supplier Name.
* **Filter:** Dropdowns to filter by Date Range, Status (Draft, Approved, Partially Received, Fully Received, Cancelled).
* **Customize View:** Choose which columns to show or hide.
* **Export Button:** Downloads the list to Excel or CSV.
* **Print Button:** Generates a print-ready view of the PO list.
* **Create New PO Button:** Prominent button to create a new Purchase Order.
* **Import Button:** Bulk upload POs from an Excel/CSV file.

### The Data List (The main table):
* PO Date
* PO Number
* Supplier Name
* Expected Delivery Date
* Total Amount
* Status (Draft, Approved, Partially Received, Fully Received, Cancelled)

## What Information We Need to Capture (The Data Fields)
When creating or editing a PO, the form is divided into Header and Line Items. 

**Data Inheritance:** A PO can be created from scratch or auto-filled from an approved Purchase Requisition (PR). If created from a PR, the Supplier, Items, and Quantities are inherited.

### Section 1: Header Details
| Field Name | Description | UI Component | Required? | Inheritance |
| :--- | :--- | :--- | :--- | :--- |
| PO Number | Unique, auto-generated system number. | Auto-computed | Yes | None |
| PO Date | Date the PO is created. | Date Picker | Yes | None |
| Supplier | The vendor we are purchasing from. | Dropdown linked to Master Data. UPON SELECTION, system MUST instantly auto-fill: Address, TIN, Credit Terms, and Tax Type. | Yes | Auto-filled from PR if applicable |
| Supplier Address | Address of the selected supplier. | Text | Yes | Auto-filled from Supplier master |
| Delivery Address | Where the goods should be delivered. | Text Area | Yes | Default to Company branch |
| Expected Delivery Date| Target delivery date. | Date Picker | Yes | None |
| Payment Terms | Agreed terms (e.g., 30 Days). | Dropdown linked to Payment Terms | Yes | Auto-filled from Supplier master |
| Tax Type | Supplier tax classification. | Dropdown linked to Tax Types | Yes | Auto-filled from Supplier master |
| Reference PR | Link to Purchase Requisition if applicable. | Dropdown/Search | Optional | None |
| Status | Current state of the PO. | Auto-computed | Yes | None |
| Notes/Remarks | Internal or supplier-facing notes. | Text Area | Optional | None |

### Section 2: Line Items
| Field Name | Description | UI Component | Required? | Inheritance |
| :--- | :--- | :--- | :--- | :--- |
| Item/Service | The product or service being ordered. | Dropdown linked to Items. UPON SELECTION, instantly auto-fills Description, UOM, Unit Price/Cost, and Tax Code. | Yes | Auto-filled from PR if applicable |

## The Supabase Database Architecture (For Lovable)

**Critical Database Rule for Lovable:**
- Ensure a composite unique constraint on `company_id` and `po_number` if a `company_id` is used, otherwise `po_number` must be unique.

**Critical UX/GL Rule:** The system MUST auto-fetch the Item's default Revenue, Expense, COGS, and Inventory GL accounts in the background to auto-generate the Journal Entry. Encoders MUST NEVER manually select a GL account for standard items.

### Table 1: `purchase_orders` (Header Table)
| Column Name | Data Type | Rules | What it stores |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System ID for the PO. |
| `po_number` | Text | Required, Unique | Auto-generated number (e.g., "PO-2023-0001"). |
| `po_date` | Date | Required | Date of creation. |
| `supplier_id` | UUID | Required, Foreign Key | Links to `suppliers.id`. |
| `delivery_address`| Text | Required | Delivery location. |
| `expected_date` | Date | Required | Target delivery date. |
| `payment_terms_id` | UUID | Required, Foreign Key | Links to `payment_terms.id`. |
| `tax_type_id` | UUID | Required, Foreign Key | Links to `tax_types.id`. |
| `reference_pr_id` | UUID | Nullable, Foreign Key | Links to `purchase_requisitions.id`. |
| `status` | Text | Required | `draft`, `approved`, `partially_received`, `fully_received`, `cancelled`. |
| `notes` | Text | Nullable | Additional instructions. |
| `total_amount` | Numeric | Required | Sum of line items. |
| `created_by` | UUID | Required, Foreign Key | Links to `users.id`. |
| `updated_by` | UUID | Nullable, Foreign Key | Links to `users.id`. |
| `created_at` | Timestamp | Auto | Record creation timestamp. |
| `updated_at` | Timestamp | Auto | Record update timestamp. |

### Table 2: `purchase_order_lines` (Line Items Table)
| Column Name | Data Type | Rules | What it stores |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System ID for the line item. |
| `po_id` | UUID | Required, Foreign Key | Links to `purchase_orders.id`. |
| `item_id` | UUID | Required, Foreign Key | Links to `items.id`. |
| `description` | Text | Nullable | Specific item details. |
| `quantity` | Numeric | Required | Quantity ordered. |
| `uom_id` | UUID | Required, Foreign Key | Links to `uoms.id`. |
| `unit_price` | Numeric | Required | Cost per unit. |
| `tax_code_id` | UUID | Required, Foreign Key | Links to `tax_codes.id`. |
| `total_amount` | Numeric | Required | Calculated amount (Qty * Price). |
| `created_by` | UUID | Required, Foreign Key | Links to `users.id`. |
| `updated_by` | UUID | Nullable, Foreign Key | Links to `users.id`. |
| `created_at` | Timestamp | Auto | Record creation timestamp. |
| `updated_at` | Timestamp | Auto | Record update timestamp. |
