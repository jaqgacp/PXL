# PXL ERP Blueprint: Supplier Ledger

## Module Overview
The Supplier Ledger (often called the Subsidiary Ledger for Accounts Payable) is an essential financial report in Philippine accounting. It provides a detailed, chronological record of all transactions—Vendor Bills, Payment Vouchers, Vendor Credits, and Debit Memos—associated with a specific supplier. Accountants rely on this view to reconcile balances with supplier statements of account (SOA) and to audit the history of payables before processing final checks. Because this is a reporting module, there is no direct data entry; it securely aggregates data from transaction tables.

## What You Will See on the Screen (The Dashboard)
When you open the "Supplier Ledger" page, you will see a robust reporting interface designed for quick analysis and reconciliation.

### The Action Bar (Top of the page):
* **Supplier Selector:** A mandatory dropdown/search bar to select the specific Supplier to view.
* **Date Range Filter:** Pick a specific period (e.g., This Month, Last Quarter, Custom Date).
* **Currency Filter:** (If applicable) Filter by base currency or foreign currency transactions.
* **Export Button:** Downloads the ledger to Excel or CSV for offline reconciliation.
* **Print Button:** Generates a formal, print-ready PDF of the subsidiary ledger.

### The Data List (The main report table):
The table reads like a standard accounting ledger with a running balance.
* Date
* Document Type (e.g., Vendor Bill, Payment Voucher, Vendor Credit)
* Document Number (Reference)
* Description/Remarks
* Debit (Payments/Credits reducing AP)
* Credit (Bills increasing AP)
* Running Balance

## What Information We Need to Capture (The Data Fields)
Since this is a report, there is no input form. However, the system dynamically pulls and maps fields from underlying transaction modules.

### Report Columns Map
| Column Name | Source | Description |
| :--- | :--- | :--- |
| Date | `transaction_date` | The date of the document (Bill Date, PV Date, etc.). |
| Document Type| System Defined | E.g., "Vendor Bill", "Payment Voucher". |
| Document No. | `bill_number`, `pv_number` | The system-generated number of the document. |
| Reference | `supplier_invoice_no`, `check_no`| The external reference number for easy tracing. |
| Description | `remarks` | Any notes from the transaction header. |
| Debit | `payment_vouchers`, `vendor_credits`| Amounts that reduce the payable balance. |
| Credit | `vendor_bills` | Amounts that increase the payable balance. |
| Balance | Auto-computed | Previous Balance + Credit - Debit. |

## The Supabase Database Architecture
Because the Supplier Ledger is a report, we do not create a new data entry table. Instead, we create a secure **SQL View** that unifies the transactions affecting Accounts Payable for a given supplier.

### View: `vw_supplier_ledger`
This view securely unions data from `vendor_bills`, `payment_vouchers`, and `vendor_credits` to provide a continuous chronological ledger.

| Column Name | Data Type | Rules | What it stores |
| :--- | :--- | :--- | :--- |
| `supplier_id` | UUID | Foreign Key | Links to `suppliers.id` for filtering. |
| `transaction_date`| Date | Required | The date of the transaction. |
| `document_type` | Text | Required | `vendor_bill`, `payment_voucher`, `vendor_credit`, `debit_memo`. |
| `document_id` | UUID | Required | The UUID of the source document (for clickable links). |
| `document_number`| Text | Required | System reference (e.g., VB-2023-001). |
| `external_ref` | Text | Nullable | Supplier Invoice, Check Number, or CM Number. |
| `description` | Text | Nullable | Remarks or narration. |
| `debit_amount` | Numeric | Default: 0 | Value reducing AP (Payments, Credits). |
| `credit_amount` | Numeric | Default: 0 | Value increasing AP (Bills). |
| `created_at` | Timestamp | Required | The timestamp of the source document creation. |

*Note to Developer:* The running balance logic should ideally be calculated on the frontend or via a window function in the query, ordered by `transaction_date` and `created_at` to ensure chronological accuracy.

**Critical UX/GL Rule:** The system MUST auto-fetch the Item's default Revenue, Expense, COGS, and Inventory GL accounts in the background to auto-generate the Journal Entry. Encoders MUST NEVER manually select a GL account for standard items.
