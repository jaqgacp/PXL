# PXL ERP Blueprint: Payment Monitoring

## Module Overview
The Payment Monitoring module tracks the lifecycle of disbursed funds. In the Philippine corporate setup, issuing a Post-Dated Check (PDC) or executing a bank transfer involves float time. This dashboard monitors Payment Vouchers from the moment they are approved, tracking whether checks have been printed, released to suppliers, and finally cleared by the bank. It provides treasury and accounting teams visibility over actual cash outflows and helps prevent bounced checks.

## What You Will See on the Screen (The Dashboard)
When you open the "Payment Monitoring" page, you will see a workspace focused on workflow tracking.

### The Action Bar (Top of the page):
* **Search Bar:** Search by Check Number, Supplier Name, or PV Number.
* **Filter:** Dropdowns to filter by Payment Account (Bank), Check Date (e.g., PDCs due this week), and Status (Approved, Printed, Released, Cleared, Stale).
* **Batch Actions:** Checkboxes on rows to batch-update status (e.g., "Mark as Cleared", "Mark as Released").
* **Export Button:** Downloads the list to Excel or CSV.
* **Print Button:** Generates a check registry report.

### The Data List (The main table):
* Check Date (or Transfer Date)
* PV Number
* Check / Ref Number
* Supplier Name
* Bank Account
* Amount
* Status (Approved, Printed, Released, Cleared, Stale, Cancelled)

## What Information We Need to Capture (The Data Fields)
This module acts primarily as an update interface for existing Payment Vouchers. However, it requires specific tracking dates for the audit trail.

### Updating the Payment Status
When a user clicks "Update Status" on a payment row, a modal appears requesting confirmation and the relevant date.

| Field Name | Description | UI Component | Required? | Inheritance |
| :--- | :--- | :--- | :--- | :--- |
| Date Released | The date the check was physically handed to the supplier. | Date Picker | Required for 'Released' | None |
| Date Cleared | The date the payment reflected in the bank statement. | Date Picker | Required for 'Cleared' | None |
| Remarks | Notes (e.g., who picked up the check). | Text Area | Optional | None |

## The Supabase Database Architecture
Payment Monitoring does not need a new core table. Instead, it utilizes the existing `payment_vouchers` table, but we will add tracking columns to that table to support this module's workflow.

### Modified Table: `payment_vouchers` (Enhancement)
The existing table from the Payment Vouchers module must be extended to include these tracking timestamps and details.

| Column Name | Data Type | Rules | What it stores |
| :--- | :--- | :--- | :--- |
| `status` | Text | Required | Must support: `approved`, `printed`, `released`, `cleared`, `stale`, `cancelled`. |
| `date_released` | Date | Nullable | Records when the check was given to the vendor. |
| `released_by` | UUID | Nullable, Foreign Key | Links to `users.id`. Who released it. |
| `date_cleared` | Date | Nullable | Records when it cleared the bank. |
| `cleared_by` | UUID | Nullable, Foreign Key | Links to `users.id`. Who reconciled it. |

### View: `vw_payment_monitoring`
A specialized SQL View optimized for this dashboard, filtering out drafts.

| Column Name | Data Type | Rules | What it stores |
| :--- | :--- | :--- | :--- |
| `pv_id` | UUID | Required | Links to `payment_vouchers.id`. |
| `pv_number` | Text | Required | System number. |
| `check_date` | Date | Required | Target date of payment. |
| `supplier_name` | Text | Required | Joined from `suppliers`. |
| `bank_name` | Text | Required | Joined from `chart_of_accounts`. |
| `check_number` | Text | Required | The reference. |
| `amount` | Numeric | Required | Total payment value. |
| `status` | Text | Required | Current state. |
*Note to Developer:* Ensure proper indexing on `check_date` and `status` as these will be the most frequently filtered columns in this view.

**Critical UX/GL Rule:** The system MUST auto-fetch the Item's default Revenue, Expense, COGS, and Inventory GL accounts in the background to auto-generate the Journal Entry. Encoders MUST NEVER manually select a GL account for standard items.
