# PXL ERP Blueprint: Inter-Branch Transfers

## Module Overview
The Inter-Branch Transfers module manages the allocation or movement of funds from one branch/department to another within the same legal entity. For Philippine businesses operating multiple branches, accurately tracking inter-branch cash flows is essential for branch-level profitability reporting. Critically, inter-branch transfers require "Due To / Due From" GL account postings — the source branch creates a liability (Due To the destination) and the destination branch records a receivable (Due From the source). These intercompany accounts must net to zero at the entity level for the consolidated financial statements to be correct.

## Dashboard UI

### The Action Bar
* **Search Bar:** Find a transfer by Document Number or Branch Name.
* **Filter:** Filter by Source Branch, Destination Branch, or Status.
* **Customize View:** Choose which columns to show or hide.
* **Export Button:** Downloads the list to Excel or CSV.
* **Create New Transfer Button:** Opens the form to log a branch-to-branch movement.

### The Data List
Default columns:
* Document Number
* Date
* Source Branch
* Destination Branch
* Amount
* Status (Draft, Approved, Received, Voided)

**Pagination Mandate:** Server-side pagination via Supabase `.range(from, to)`. Default page size: 50 records.

## GL Posting Logic (System-Generated on Approval/Post)

Inter-branch transfers generate journal entries in the context of the company's combined books. The Due To / Due From accounts must be configured in the Chart of Accounts before this module can be used.

**On Source Branch (for each destination line):**
| # | Account | Debit | Credit | Source |
| :--- | :--- | :--- | :--- | :--- |
| 1 | Due To [Destination Branch] | | Allocation Amount | `inter_branch_transfer_lines.due_from_account_id` (mirrored) |
| 2 | Cash / Bank Account (Source) | | (if cash actually moved) | `branches.default_bank_account_id` |

**On Destination Branch:**
| # | Account | Debit | Credit | Source |
| :--- | :--- | :--- | :--- | :--- |
| 1 | Cash / Bank Account (Destination) | Allocation Amount | | `branches.default_bank_account_id` |
| 2 | Due From [Source Branch] | Allocation Amount | | `inter_branch_transfer_lines.due_from_account_id` |

*Reconciliation Rule: The total balance of all "Due To" accounts must equal the total balance of all "Due From" accounts at the entity level. This module's Reconciliation screen (a computed view, no stored data) surfaces any imbalance to the Controller.*

## Supabase Database Architecture

### Table 1: `inter_branch_transfers` (Header)

**Critical Database Rules:**
1. `UNIQUE(company_id, document_number)` — prevents cross-tenant duplicate numbers.
2. `journal_entry_id` is the mandatory GL traceability link. Populated on Post (Principle 7).
3. `posted_at` and `posted_by` are system-set. Never editable.

| Column Name | Data Type | Rules | What it stores |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System ID for the transfer. |
| `company_id` | UUID | Required, Foreign Key | Links to `companies.id`. Tenant isolation anchor. |
| `document_number` | Text | Required | `UNIQUE(company_id, document_number)`. |
| `transfer_date` | Date | Required | Validated against open `accounting_periods`. |
| `fiscal_period_id` | UUID | Required, Foreign Key | Links to `accounting_periods.id`. |
| `source_branch_id` | UUID | Required, Foreign Key | Links to `branches.id`. |
| `due_to_account_id` | UUID | Required, Foreign Key | Links to `chart_of_accounts.id`. The Due To liability account on Source Branch books. |
| `total_amount` | Numeric(15,2) | Required | Sum of all destination line amounts. |
| `particulars` | Text | Required | Reason for the transfer. |
| `journal_entry_id` | UUID | Nullable, Foreign Key | Links to `journal_entries.id`. Populated on Post. GL traceability link. |
| `posted_at` | Timestamp | Nullable | System-set on Post. Never editable. |
| `posted_by` | UUID | Nullable, Foreign Key | Links to `users.id`. |
| `status` | Text | Required | ENUM: `draft`, `approved`, `received`, `voided`. |
| `created_by` | UUID | Required, Foreign Key | Links to `users.id`. |
| `updated_by` | UUID | Nullable, Foreign Key | Links to `users.id`. |
| `created_at` | Timestamp | Auto | Creation timestamp. |
| `updated_at` | Timestamp | Auto | Last update timestamp. |

**Composite Unique Constraint:** `UNIQUE(company_id, document_number)`

## Row-Level Security — Immutability Policy

```sql
-- Block UPDATE on approved, received, or voided transfers
CREATE POLICY "block_update_on_posted_ibt"
ON inter_branch_transfers
FOR UPDATE
USING (status = 'draft');

-- Block DELETE on all inter-branch transfers
CREATE POLICY "block_delete_all_ibt"
ON inter_branch_transfers
FOR DELETE
USING (false);

-- Tenant isolation
CREATE POLICY "tenant_isolation_ibt"
ON inter_branch_transfers
FOR ALL
USING (company_id = (auth.jwt() ->> 'company_id')::UUID);

-- Line items: tenant isolation
CREATE POLICY "tenant_isolation_ibt_lines"
ON inter_branch_transfer_lines
FOR ALL
USING (company_id = (auth.jwt() ->> 'company_id')::UUID);
```
