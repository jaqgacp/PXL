# PXL Transaction Field Source Matrix

Status: Mandatory platform control standard
Version: 1.0
Last updated: 2026-07-15
Reference transaction: Sales Invoice

## 1. Purpose

The Transaction Field Source Matrix eliminates ambiguity about every transaction field.

It answers:

- where the field comes from,
- whether it is stored,
- whether it is a snapshot or live master value,
- when and by whom it can be edited,
- where it appears,
- which business processes consume it,
- whether posting, tax, inventory, reporting, API, import/export, printed documents, or BIR outputs depend on it,
- whether the current implementation is complete and validated.

No transaction may be marked complete, validated, or approved unless its Field Source Matrix is complete and validated.

## 2. Scope

This standard applies to every PXL transaction:

- Sales Invoice
- Sales Order
- Sales Quotation
- Delivery Receipt
- Sales Receipt / Official Receipt
- Credit Memo
- Purchase Order
- Goods Receipt
- Vendor Bill
- Vendor Payment
- Vendor Credit
- Journal Entry
- Inventory transactions
- Banking transactions
- Fixed asset transactions
- Compliance and specialized transactions

The Sales Invoice matrix is the first reference implementation. Future transactions must inherit the matrix requirement, not Sales Invoice-specific business assumptions.

## 3. Source Classifications

Every field has exactly one primary source classification.

| Classification | Meaning | Examples |
| --- | --- | --- |
| `USER_INPUT` | Entered directly by the user during create/edit. | Invoice Date, External Reference, Quantity, Unit Price, Remarks |
| `MASTER_DATA` | Inherited or selected from an authoritative master record. | Customer Code, Customer TIN, Item UOM, Revenue Account, Warehouse |
| `SYSTEM_GENERATED` | Generated by the application and not manually entered. | Document Number, Status, Created At, Internal Record ID |
| `POSTING_ENGINE` | Created or finalized by server-side posting logic. | Journal Entry ID, posted GL lines, VAT ledger rows, inventory ledger entries |
| `COMPUTED` | Derived from other authoritative values. | Net Amount, VAT Amount, Invoice Total, Balance Due, Gross Profit |
| `WORKFLOW_ENGINE` | Created by workflow or approval processing. | Approval Status, Current Approver, Submitted At, Rejection Reason |
| `RELATED_DOCUMENT` | Inherited or derived from linked source/target transactions. | Sales Order Number, Delivery Receipt Number, Applied Receipt Amount |
| `INTEGRATION` | Received from an approved external system or API. | External marketplace reference, integration batch ID |
| `MIGRATION` | Loaded from legacy/onboarding data. | Legacy document number, imported opening invoice |

A field may have secondary influences, but one authoritative primary source must be selected.

## 4. Storage Classifications

Use these values in the `Stored?` column:

| Classification | Meaning |
| --- | --- |
| `YES_TRANSACTION_SNAPSHOT` | Stored on the transaction header for historical correctness. |
| `YES_LINE_SNAPSHOT` | Stored on the transaction line for historical correctness. |
| `YES_LEDGER` | Stored only in accounting, tax, inventory, payment, or other authoritative ledger records. |
| `YES_AUDIT` | Stored only in audit/history records. |
| `YES_RELATED_RECORD` | Stored on a related transaction or application record. |
| `NO_COMPUTED` | Not stored; computed from authoritative values. |
| `NO_PRESENTATION_ONLY` | Presentation-only label or UI grouping. |
| `CONDITIONAL` | Stored only under documented conditions. |

Do not use a simple Yes/No where source behavior is more specific.

## 5. Editability Classifications

Use these values in the `Editable?` column:

| Classification | Meaning |
| --- | --- |
| `DRAFT_ONLY` | Editable only while the transaction is Draft. |
| `BEFORE_SUBMISSION` | Editable before submission, locked after submission. |
| `BEFORE_APPROVAL` | Editable until approval. |
| `BEFORE_POSTING` | Editable until posting. |
| `ROLE_RESTRICTED` | Editable only by an authorized role. |
| `SYSTEM_ONLY` | Never directly editable by a user. |
| `DERIVED_READ_ONLY` | Computed and read-only. |
| `MASTER_DATA_CONTROLLED` | Changed through source master data, not directly on the transaction unless a documented override exists. |
| `CORRECTION_FLOW_ONLY` | Changed after posting only through approved correction, credit, reversal, cancellation, or void flow. |
| `NEVER` | Never editable under any condition. |

Editable fields must also identify exact lifecycle statuses and roles.

## 6. Appearance Destinations

Every field must map to zero or more destinations.

Allowed destination codes:

`FORM`, `VIEW`, `HEADER`, `DOC_PANEL`, `CUSTOMER_PANEL`, `CONTEXT_PANEL`, `LINES`, `FINANCIAL`, `GL`, `TAX`, `VALIDATION`, `WORKFLOW`, `APPROVAL`, `AUDIT`, `RELATED_DOCS`, `RELATED_PARTY`, `ATTACHMENTS`, `ACTIVITY`, `NOTES`, `SYSTEM`, `LIST`, `SEARCH`, `DASHBOARD`, `REPORTS`, `PRINTED`, `PDF`, `EXCEL`, `CSV_IMPORT`, `CSV_EXPORT`, `API_REQUEST`, `API_RESPONSE`, `FINANCIAL_REPORT`, `INVENTORY_REPORT`, `TAX_REPORT`, `BIR_FORM`, `CAS_EXPORT`.

Any destination not listed for a field is treated as `No`.

## 9. Field Status Values

Implementation Status:

`NOT_DEFINED`, `DEFINED`, `SOURCE_MISSING`, `MASTER_DATA_MISSING`, `DATABASE_MISSING`, `UI_MISSING`, `POSTING_MISSING`, `REPORTING_MISSING`, `PARTIALLY_IMPLEMENTED`, `IMPLEMENTED`, `VALIDATED`, `BLOCKED`, `DEPRECATED`.

Validation Status:

`NOT_TESTED`, `DOCUMENT_REVIEWED`, `UI_VALIDATED`, `DATABASE_VALIDATED`, `POSTING_VALIDATED`, `REPORTING_VALIDATED`, `END_TO_END_VALIDATED`, `FAILED`, `BLOCKED`.

Do not mark a field complete solely because it appears on screen.

## 10. Authoritative Source Rules

1. Every field must identify one authoritative source.
2. Computed values must identify authoritative inputs and the formula or service that computes them.
3. Posted values must come from posted ledger rows or posting results, not client-side recomputation.
4. Preview values must be labeled `PREVIEW` and must not be treated as posted truth.
5. Master-data fields must explicitly choose snapshot, live master, or both.
6. Posted historical documents must not silently change because master data changed later.

## 13. Rules for Future Transaction Matrices

Every future transaction must:

1. Complete the Field Source Matrix before coding.
2. Identify authoritative source, storage, editability, appearance, and business use for every field.
3. Identify snapshot versus live master behavior for each master-data field.
4. Identify preview versus authoritative posted result for every posting/ledger field.
5. Identify all dimensions, inheritance rules, overrides, posting propagation, and report consumers.
6. Identify API/import/export/printed/BIR usage.
7. Record implementation and validation status per field.
8. Update the matrix with every field change.

No transaction may move from `DEFINED` to `READY_FOR_IMPLEMENTATION` unless its Field Source Matrix is complete enough to establish authoritative source, storage, editability, appearance, and business use for every required field.

No transaction may move to `VALIDATED` or `APPROVED_REFERENCE` unless the Field Source Matrix is end-to-end validated against implementation evidence.

## 14. Validation Checklist

For each field:

- UI component exists where required.
- Database or authoritative service source exists.
- Master-data source exists where applicable.
- Data type and format match.
- Validation and normalization match the matrix.
- Defaulting behavior matches.
- Editability matches lifecycle and role.
- Storage location matches.
- Snapshot/live behavior matches.
- Posting/ledger use matches.
- Reporting/API/import/export/printed/BIR use matches.
- Audit capture matches.
- Permissions and lifecycle immutability are enforced below the UI.
- Empty and fallback behavior are explicit.

For Sales Invoice specifically, validation must cover:

- draft, submitted, approved, posted unpaid, partially paid, fully paid;
- VAT, non-VAT, zero-rated, VAT-exempt;
- inventory item, service item, and mixed invoice after inventory support exists;
- discounts and expected CWT;
- header dimensions and line-level overrides after dimension support exists;
- warehouse after inventory support exists;
- source Sales Order / Delivery Receipt after conversion support exists;
- receipt application, credit memo, reversal/correction;
- imported/API-created invoice when those channels are enabled.

## 15. Change Control

When a field is added, removed, renamed, reformatted, re-sourced, or reused:

1. Update this matrix.
2. Update the transaction definition.
3. Update database schema or authoritative service contracts if needed.
4. Update API/import/export/printed/report/BIR contracts if affected.
5. Update tests.
6. Assess historical-data migration.
7. Assess posted-document immutability.
8. Assess downstream transactions and reports.
9. Assess audit, security, and permission impact.

No field change may be implemented only in the UI.
