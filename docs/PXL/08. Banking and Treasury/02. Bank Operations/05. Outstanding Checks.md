## Module Overview
The Outstanding Checks module is a specialized monitoring dashboard directly interfacing with the Accounts Payable (AP) ledger. In Philippine practice, checks are often issued and recorded in the books (crediting Cash) but are not immediately encashed by suppliers. This dashboard automatically tracks all AP Check Payments that have been issued but remain "uncleared" in the Bank Reconciliation module, ensuring accurate forecasting of actual available cash.

## Dashboard UI
When you open the "Outstanding Checks" page, you will see a real-time list of floating checks.

### The Action Bar
* **Search Bar:** Find a check by Check Number, Payee Name, or AP Voucher No.
* **Filter:** Filter by Bank Account, Date Issued Range, or Aging (e.g., > 6 months stale checks).
* **Customize View:** Choose which columns to show or hide.
* **Export Button:** Downloads the list to Excel or CSV for audit schedules.
* **Mark as Stale Button:** For checks uncashed beyond 180 days, initiate a reversal entry.

### The Data List
The main table displays uncleared payments. Default columns:
* Check Number
* Date Issued
* Payee
* Bank Account
* Amount
* Aging (Days outstanding)
* Source AP Voucher

## Data Fields

Because this is a dynamic reporting module driven by the AP Ledger and Bank Reconciliation, the data fields are virtually entirely inherited.

### Section 1: Check Details (Read-Only Interface)
| Field Name | Description | Required? | UI Component | Data Inheritance |
| :--- | :--- | :--- | :--- | :--- |
| Check Number | The physical check number. | Yes | Read-only | Fetched from AP Payments |
| Issue Date | Date the check was released. | Yes | Read-only | Fetched from AP Payments |
| Payee | Supplier or recipient name. | Yes | Read-only | Fetched from AP Payments / Masterfile |
| Bank Account | Source funding account. | Yes | Read-only | Fetched from AP Payments |
| Amount | Disbursement value. | Yes | Read-only | Fetched from AP Payments |
| Stale Status | Checks > 180 days old. | Yes | Auto-computed | (Current Date - Issue Date) > 180 |

## Supabase Database Architecture

### Table: `vw_outstanding_checks` (Database View)
Instead of a separate data-entry table, this module utilizes a **SQL View** that queries the existing AP Payments and filters out those that have been cleared in the Bank Reconciliation table.

| Column Name | Data Type | Rules | What it stores (Source) |
| :--- | :--- | :--- | :--- |
| `ap_payment_id` | UUID | Primary Key | `ap_payments.id` |
| `check_number` | Text | Required | `ap_payments.reference_number` |
| `issue_date` | Date | Required | `ap_payments.payment_date` |
| `payee_name` | Text | Required | Fetched via `ap_payments.supplier_id` |
| `bank_account_id` | UUID | Required | `ap_payments.bank_account_id` |
| `amount` | Numeric | Required | `ap_payments.amount` |
| `is_stale` | Boolean | Default: `false` | Computed: `CURRENT_DATE - ap_payments.payment_date > 180` |
| `created_by` | UUID | Required | `ap_payments.created_by` |
| `updated_by` | UUID | Nullable | `ap_payments.updated_by` |
| `created_at` | Timestamp | Required | `ap_payments.created_at` |
| `updated_at` | Timestamp | Required | `ap_payments.updated_at` |

*Note: The view strictly includes records where `ap_payments.is_cleared = false` (or not present as `is_cleared = true` in `bank_reconciliation_lines`). This ensures complete AP/AR ledger integrity without duplicating data.*
