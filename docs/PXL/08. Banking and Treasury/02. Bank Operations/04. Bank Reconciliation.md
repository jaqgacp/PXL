## Module Overview
Bank Reconciliation ensures that the system's cash book balance matches the physical bank statement balance. Crucially, this module directly interfaces with the Accounts Payable (AP) and Accounts Receivable (AR) ledgers. When you mark an AP Check Payment or an AR Deposit as "cleared" in this module, it updates the underlying transaction status. This prevents cash from being falsely reported as available and forms the backbone of accurate Treasury management in the Philippines.

## Dashboard UI
When you open the "Bank Reconciliation" page, you will see your reconciliation history and active statements.

### The Action Bar
* **Search Bar:** Find a reconciliation by Bank Account or Month/Year.
* **Filter:** Filter by Status (Draft, Reconciled).
* **Customize View:** Choose which columns to show or hide.
* **Export Button:** Downloads the reconciliation report (Excel/PDF) for auditor sign-off.
* **Create New Reconciliation Button:** Start a new month's matching process.
* **Import Button:** Upload standard MT940 or CSV bank statements to auto-match transactions.

### The Data List
The main table displays statement periods. Default columns:
* Period Ending
* Bank Account
* Statement Balance
* Book Balance
* Variance
* Status

## Supabase Database Architecture

**Critical Database Rule:**
- `bank_reconciliations` must have a composite unique constraint on `company_id`, `bank_account_id`, and `statement_date`.

### Table 1: `bank_reconciliations` (Header)
| Column Name | Data Type | Rules | What it stores |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System ID for the reconciliation. |
| `company_id` | UUID | Required, Foreign Key | Links to companies table. |
| `bank_account_id` | UUID | Required, Foreign Key | Links to `bank_accounts.id`. |
| `statement_date` | Date | Required | The cut-off date (e.g., end of month). |
| `statement_beginning_balance`| Numeric | Required | Starting balance. |
| `statement_ending_balance` | Numeric | Required | Ending balance per bank. |
| `calculated_book_balance` | Numeric | Required | System balance at cut-off. |
| `variance_amount` | Numeric | Required | Should be 0 when fully reconciled. |
| `status` | Text | Required | `draft`, `reconciled`. |
| `created_by` | UUID | Required, Foreign Key | Links to `users.id`. |
| `updated_by` | UUID | Nullable, Foreign Key | Links to `users.id`. |
| `created_at` | Timestamp | Auto | Date record was created. |
| `updated_at` | Timestamp | Auto | Date record was last edited. |

### Table 2: `bank_reconciliation_lines` (Line Items)
*Note: This table tracks which AP payments, AR deposits, or adjustments were cleared in this specific reconciliation, enforcing the AP/AR interface.*

| Column Name | Data Type | Rules | What it stores |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System ID for the line item. |
| `bank_reconciliation_id`| UUID | Required, Foreign Key | Links to `bank_reconciliations.id`. |
| `transaction_type` | Text | Required | `ap_payment`, `ar_collection`, `bank_adjustment`, `fund_transfer`. |
| `transaction_id` | UUID | Required | Polymorphic ID pointing to the source record in AP or AR ledgers. |
| `cleared_amount` | Numeric | Required | The amount cleared in the bank. |
| `is_cleared` | Boolean | Default: `false` | True when matched. Updating to true cascades a "cleared" status back to the AP/AR ledger. |
| `cleared_date` | Date | Nullable | Exact clearing date. |
| `created_by` | UUID | Required, Foreign Key | Links to `users.id`. |
| `updated_by` | UUID | Nullable, Foreign Key | Links to `users.id`. |
| `created_at` | Timestamp | Auto | Date record was created. |
| `updated_at` | Timestamp | Auto | Date record was last edited. |
