# PXL ERP Blueprint: Auto Reversal Run

## Module Overview
The Auto Reversal Run manages standard accounting accruals that require reversal in the subsequent period. For example, in Philippine accounting, if a company incurs utility expenses in December but the bill won't arrive until January, they must accrue the expense in December via a Journal Entry (Dr. Expense, Cr. Accrued Payable). In January, when the actual bill arrives and is processed through AP, the original December accrual must be reversed to prevent double-counting the expense. This module automates the mass reversal of all JEs flagged as "Auto-Reverse" on the first day of the new period.

## Dashboard UI
A specialized month-end/month-beginning batch execution screen.

### The Action Bar
* **Target Reversal Period:** Select the new month (e.g., if accruing in December, the Target Period is January).
* **Preview Run Button:** Generates a list of all Journal Entries from the previous period flagged for auto-reversal.
* **Execute Reversals Button:** Automatically creates and posts the reversing entries.

### The Data List (Preview Mode)
* **Original JE Number:** Hyperlinked to the source entry.
* **Original Date:** Usually the last day of the prior month.
* **Description:** The memo of the accrual.
* **Amount:** The transaction value.
* **Status:** "Pending Reversal" or "Already Reversed".

## Supabase Database Architecture

This relies heavily on flags within the `journal_entries` table.

### Required Schema Updates
To support this feature, the `journal_entries` table must include two specific columns (which were not detailed in the base JE blueprint but are critical here):

| Column Name | Data Type | Rules | What it stores | created_by | UUID | Required, Foreign Key | User who created the record. |

### Data Flow & Logic
1.  **Identify Targets:** The system queries `journal_entries`:
    ```sql
    SELECT * FROM journal_entries 
    WHERE is_auto_reversing = true 
    AND status = 'posted' 
    AND reversal_je_id IS NULL 
    AND transaction_date < [Reversal Date];
    ```
2.  **Generate Reversals:** For each identified JE, the system creates a new JE:
    *   **Date:** The specified `Reversal Date`.
    *   **Description:** "Auto-Reversal of [Original JE Number]".
    *   **Lines:** Copies the lines of the original JE, but strictly swaps Debits and Credits.
    *   *Validation Check:* Total Dr = Total Cr.
3.  **Commit & Link:** 
    *   Posts the new reversing JE to the GL.
    *   Updates the original JE: `reversal_je_id = [New JE ID]`.
4.  **Audit Log:** Records the execution in the `batch_job_logs` table (Job Type = `auto_reversal_run`).
