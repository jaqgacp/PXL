# PXL ERP Blueprint: Posting Review

## Module Overview
The Posting Review module functions as the accounting department's central inbox for approval workflows. In mid-to-large Philippine enterprises, data entry clerks or junior accountants create transactions (Draft Bills, Draft Invoices, Draft Journal Entries), but only senior accountants or controllers have the authority to "Post" them to the General Ledger. This module aggregates all unposted, draft, or pending-approval documents across all ERP modules into a single, unified checklist, ensuring nothing is left hanging before a period close.

## Dashboard UI
A centralized task list designed for rapid review and batch processing.

### The Action Bar
* **Module Filter:** Dropdown to isolate specific document types (e.g., Show only Draft JEs, Show only Pending Bills).
* **Date Filter:** Filter by transaction date.
* **Batch Post Button:** Select multiple rows and post them simultaneously to the GL.
* **Batch Reject Button:** Send selected rows back to the creator with a rejection note.
* **Export Button:** Export the pending list.

### The Data List
* **Checkbox:** For batch selection.
* **Document Type:** E.g., Journal Entry, AP Bill, AR Invoice.
* **Document Number:** Hyperlinked to open a quick-view modal of the document.
* **Date:** Transaction date.
* **Created By:** The user who prepared the draft.
* **Amount:** Total transaction value.
* **Status:** Draft, Pending Review.
* **Actions:** View Details, Approve & Post, Reject.

## Data Fields

This is a unified view. It does not introduce new database tables for data entry but rather queries the status of existing tables.

### Quick View Modal
When a reviewer clicks "View Details", a modal opens displaying the core data of the document without leaving the page.
* If it's a JE: Shows the Debit/Credit lines to ensure `Total Dr = Total Cr`.
* If it's a Bill: Shows the expense accounts and vendor details.
* **Reviewer Notes:** A text input to add comments upon rejection or approval.

## Supabase Database Architecture

This module relies on a SQL View that `UNION ALL`s the unposted records from various header tables.

### Unified View Architecture
```sql
-- Conceptual View for the Posting Review Dashboard
CREATE OR REPLACE VIEW pending_postings_view AS
SELECT 
    'Journal Entry' AS document_type,
    id AS document_id,
    je_number AS document_number,
    transaction_date AS document_date,
    total_debit AS amount,
    created_by,
    status
FROM journal_entries WHERE status IN ('draft', 'pending_approval')
UNION ALL
SELECT 
    'AP Bill' AS document_type,
    id AS document_id,
    bill_number AS document_number,
    bill_date AS document_date,
    total_amount AS amount,
    created_by,
    status
FROM ap_bills WHERE status IN ('draft', 'pending_approval');
-- (Additional UNIONs for Invoices, Payments, etc.)
```

### Approval Workflow Logic
When a document is "Approved & Posted" from this screen:
1. The system locates the source document using `document_id` and `document_type`.
2. It changes the status of that document to `posted`.
3. It triggers the standard posting function for that module, which writes the lines into `general_ledger_entries`.
4. It logs the `updated_by` and `updated_at` with the reviewer's credentials.
