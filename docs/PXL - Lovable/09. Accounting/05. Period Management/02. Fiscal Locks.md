# PXL ERP Blueprint: Fiscal Locks

## Module Overview
Fiscal Locks provide granular, module-level locking within accounting periods that are still **open** in `accounting_periods`. They allow the Controller to lock specific document types independently — for example, AP Bills locked on the 3rd of the month, AR Collections remaining open until the 5th, and Journal Entries open until the 10th for final adjustments.

**IMPORTANT — Layer Hierarchy:** Fiscal Locks are the **second layer** of the period control system. They only apply when `accounting_periods.is_closed = false`. If a period is hard-closed in `accounting_periods`, Fiscal Locks are irrelevant — the master lock already blocks everything. See `Period Closing.md` for the full precedence rules.

## Dashboard UI

### The Action Bar
* **As-Of Date Selector:** Target date to apply locks up to.
* **Apply Locks Button:** Saves the configuration.
* **Audit Log Button:** Shows full history of lock changes (who changed what, when, from what date to what date).

## Supabase Database Architecture

### Table 1: `fiscal_locks`

**Critical Database Rules:**
1. Composite unique constraint on `(company_id, module_name)` — one lock record per module per company.
2. `lock_date` is the operative field. `NULL` means no fiscal lock is active for that module (period master lock still applies).
3. `company_id` is mandatory for multi-tenant isolation.

| Column Name | Data Type | Rules | Description |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System identifier. |
| `company_id` | UUID | Required, Foreign Key | Links to `companies.id`. Tenant isolation anchor. |
| `module_name` | Text | Required | ENUM: `ap_bills`, `ap_payments`, `ar_invoices`, `ar_collections`, `journal_entries`, `inventory_adjustments`, `bank_transactions`, `fixed_assets`. |
| `lock_date` | Date | Nullable | The effective lock cutoff date. `NULL` = no fiscal lock active for this module. |
| `override_reason` | Text | Nullable | Logged when lock date is moved backwards by a super-admin. |
| `created_by` | UUID | Required, Foreign Key | Links to `users.id`. |
| `updated_by` | UUID | Nullable, Foreign Key | Links to `users.id`. |
| `created_at` | Timestamp | Auto | Creation timestamp. |
| `updated_at` | Timestamp | Auto | Last update timestamp. |

**Composite Unique Constraint:** `UNIQUE(company_id, module_name)`

### Table 2: `fiscal_lock_audit_log`

All changes to Fiscal Locks are immutably logged here for BIR audit trail purposes.

| Column Name | Data Type | Rules | Description |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System identifier. |
| `company_id` | UUID | Required, Foreign Key | Links to `companies.id`. |
| `fiscal_lock_id` | UUID | Required, Foreign Key | Links to `fiscal_locks.id`. |
| `module_name` | Text | Required | The module that was changed. |
| `old_lock_date` | Date | Nullable | The previous lock date before this change. |
| `new_lock_date` | Date | Nullable | The new lock date after this change. |
| `change_reason` | Text | Nullable | The override reason supplied by the user. |
| `changed_by` | UUID | Required, Foreign Key | Links to `users.id`. The user who made the change. |
| `changed_at` | Timestamp | Auto | Exact timestamp of the change. |

### Validation Logic (Application Layer — Layer 2)

This check is executed **only after** `accounting_periods.is_closed` returns `false`.

```sql
-- STEP 2: Module-level fiscal lock check.
SELECT lock_date
FROM fiscal_locks
WHERE company_id = :company_id
  AND module_name = :posting_module;

-- If lock_date IS NOT NULL AND :transaction_date <= lock_date:
--   ABORT. Return error: "The [module_name] module is locked for
--   transactions on or before [lock_date]. Contact your Controller."

-- If lock_date IS NULL OR :transaction_date > lock_date:
--   PASS. Proceed with posting.
```
