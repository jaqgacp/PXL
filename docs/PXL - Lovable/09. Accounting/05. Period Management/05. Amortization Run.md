# PXL ERP Blueprint: Amortization Run

## Module Overview
The Amortization Run is a batch-processing engine utilized during the month-end close. Instead of an accountant manually posting individual journal entries for 50 different prepaid assets (rent, insurance, software, etc.), this module scans all active `amortization_schedules` (defined in the Schedules module). It identifies lines scheduled for the current period, aggregates them, and automatically generates and posts the corresponding Journal Entries to shift the prepaid balances to expense, ensuring fast and accurate period-end compliance.

## Dashboard UI
A wizard-like interface used by the accounting team at the end of the month.

### The Action Bar
* **Target Period:** A Month/Year selector (e.g., "January 2026").
* **Preview Run Button:** Generates a draft view of what will be posted without actually committing to the GL.
* **Execute Run Button:** The final action that creates and posts the JEs.

### The Data List (Preview Mode)
When "Preview Run" is clicked, the system displays a list of the schedules that will be processed.
* **Schedule Name:** The source schedule.
* **Asset Account:** The GL account to be credited (decreased).
* **Expense Account:** The GL account to be debited (increased).
* **Amount:** The calculated amortization for this specific month.
* **Status:** "Ready to Post" or "Error" (e.g., if the destination account is inactive).

## Supabase Database Architecture

The Amortization Run does not need a dedicated persistence table for the run itself, but it heavily manipulates existing tables. 

### Data Flow & Logic
1.  **Identify Targets:** The system queries `amortization_schedule_lines` where `period_date` falls within the Target Period AND `status = 'pending'`.
2.  **Generate JEs:** For each valid line, the system constructs a Journal Entry:
    *   **Header:** Date = `Posting Date`, Description = "Amortization for [Schedule Name] - [Period]".
    *   **Debit Line:** Account = `expense_account_id` from the schedule header, Amount = `amount`.
    *   **Credit Line:** Account = `asset_account_id` from the schedule header, Amount = `amount`.
    *   *Validation Check:* Ensure Debit exactly equals Credit.
3.  **Commit:** The system inserts the records into `journal_entries` and `journal_entry_lines`, and immediately flags them as `posted` (updating the `general_ledger_entries`).
4.  **Update Schedule:** The processed `amortization_schedule_lines` record is updated: `status = 'posted'`, and `journal_entry_id = [The UUID of the newly created JE]`.
