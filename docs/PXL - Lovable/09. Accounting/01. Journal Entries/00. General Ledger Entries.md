# PXL ERP Blueprint: General Ledger Entries (Immutable Ledger)

## Module Overview
The General Ledger Entries table (`general_ledger_entries`) is the ultimate financial truth of the PXL ERP system. Every transaction that affects a company's financial position — whether originating from a Sales Invoice, Purchase Bill, Journal Entry, Asset Depreciation Run, Bank Reconciliation, or any other module — MUST produce one or more rows in this table. It is the single source that all financial reports (Trial Balance, Income Statement, Balance Sheet) query.

**This table is append-only.** No UPDATE or DELETE is ever permitted by any user, including system administrators. Corrections are made via reversing entries only. This immutability is enforced by Row-Level Security at the database level — not by application logic.

**Relationship to Journal Entries:** `journal_entries` is the grouping header (the "voucher"). `general_ledger_entries` are the individual debit/credit lines within that voucher. One `journal_entries` record has many `general_ledger_entries`. The GL Entry is the atomic unit of the double-entry accounting system.

---

## Why a Dedicated GL Entries Table

Without a dedicated `general_ledger_entries` table, financial reports must query many scattered module tables (sales invoices, purchase bills, petty cash vouchers, depreciation runs) with complex JOIN logic. This creates:

1. **Performance problems** — Trial Balance cannot be computed from 12 different module tables at scale
2. **Audit risk** — no single immutable record of what was actually posted
3. **Reporting complexity** — every new module requires changes to financial report queries

With `general_ledger_entries`, every module **posts to the ledger** upon confirmation, and all financial reports become simple aggregate queries on a single table.

## Supabase Database Architecture

### Table: `general_ledger_entries`

**This is the most critical table in the PXL ERP system. Its architecture determines the performance of every financial report.**

**Critical Database Rules:**
1. **INSERT-ONLY.** RLS blocks UPDATE and DELETE for all roles including `super_admin`. There are no exceptions.
2. Every row must belong to exactly one `journal_entries` record via `journal_entry_id`.
3. Every INSERT requires a valid `account_id`, `company_id`, and a non-zero amount in either `debit_amount` or `credit_amount` (not both).
4. Double-entry integrity is enforced at the Journal Entry level: the sum of all `debit_amount` must equal the sum of all `credit_amount` for rows sharing the same `journal_entry_id`.
5. **Table Partitioning:** For billion-row scalability, the table is partitioned by `posting_year` (range partitioning on `EXTRACT(YEAR FROM posting_date)`). Each year's data lives in its own partition (`general_ledger_entries_2024`, `general_ledger_entries_2025`, etc.). Queries filtered by date range hit only the relevant partition(s).

| Column Name | Data Type | Rules | What it stores |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System identifier. |
| `company_id` | UUID | Required, Foreign Key | Links to `companies.id`. Denormalized for RLS and partition pruning. |
| `branch_id` | UUID | Required, Foreign Key | Links to `branches.id`. |
| `journal_entry_id` | UUID | Required, Foreign Key | Links to `journal_entries.id`. The grouping voucher header. |
| `fiscal_period_id` | UUID | Required, Foreign Key | Links to `fiscal_periods.id`. The accounting period this posting belongs to. |
| `posting_date` | Date | Required | The date this line was recognized in the ledger. |
| `posting_year` | Integer | Required | `EXTRACT(YEAR FROM posting_date)`. Used as the partition key. |
| `account_id` | UUID | Required, Foreign Key | Links to `chart_of_accounts.id`. The GL account being debited or credited. |
| `account_code` | Text | Required | Snapshot of `chart_of_accounts.account_code` at time of posting. |
| `account_name` | Text | Required | Snapshot of `chart_of_accounts.account_name` at time of posting. |
| `contact_id` | UUID | Nullable, Foreign Key | Links to `contacts.id` (customer or supplier). Required for control accounts. |
| `debit_amount` | Numeric(15,2) | Required, Default 0 | The debit value. Either this or `credit_amount` must be non-zero, but not both. |
| `credit_amount` | Numeric(15,2) | Required, Default 0 | The credit value. Either this or `debit_amount` must be non-zero, but not both. |
| `currency_code` | Text | Required, Default `PHP` | ISO 4217 currency code of the transaction. |
| `exchange_rate` | Numeric(15,6) | Required, Default 1.000000 | Rate used to convert to functional currency (PHP). 1.0 if transaction currency = PHP. |
| `functional_debit` | Numeric(15,2) | Required, Default 0 | `debit_amount × exchange_rate`. Always in PHP. |
| `functional_credit` | Numeric(15,2) | Required, Default 0 | `credit_amount × exchange_rate`. Always in PHP. |
| `description` | Text | Required | Narration for this GL line. |
| `source_document_type` | Text | Required | Originating module: ENUM matching `journal_entries.source_document_type`. |
| `source_document_id` | UUID | Required | FK to the originating document (Principle 7 drill-down). |
| `source_document_no` | Text | Required | Snapshot of the originating document number. |
| `is_reversal` | Boolean | Required, Default `false` | True if this entry was generated by a reversing journal. |
| `reversal_of_id` | UUID | Nullable, Foreign Key | Self-reference to the original GL entry being reversed. |
| `created_by` | UUID | Required, Foreign Key | Links to `users.id`. Who triggered the posting. |
| `created_at` | Timestamp | Auto | Insertion timestamp. |

**NOTE: No `updated_at` column. This table is insert-only. Once a row is written, it is immutable.**

## Table Partitioning Strategy

```sql
-- Create the parent partitioned table
CREATE TABLE general_ledger_entries (
  id UUID NOT NULL DEFAULT gen_random_uuid(),
  company_id UUID NOT NULL,
  branch_id UUID NOT NULL,
  journal_entry_id UUID NOT NULL,
  fiscal_period_id UUID NOT NULL,
  posting_date DATE NOT NULL,
  posting_year INTEGER NOT NULL GENERATED ALWAYS AS (EXTRACT(YEAR FROM posting_date)::INTEGER) STORED,
  account_id UUID NOT NULL,
  account_code TEXT NOT NULL,
  account_name TEXT NOT NULL,
  contact_id UUID,
  debit_amount NUMERIC(15,2) NOT NULL DEFAULT 0,
  credit_amount NUMERIC(15,2) NOT NULL DEFAULT 0,
  currency_code TEXT NOT NULL DEFAULT 'PHP',
  exchange_rate NUMERIC(15,6) NOT NULL DEFAULT 1.000000,
  functional_debit NUMERIC(15,2) NOT NULL DEFAULT 0,
  functional_credit NUMERIC(15,2) NOT NULL DEFAULT 0,
  description TEXT NOT NULL,
  source_document_type TEXT NOT NULL,
  source_document_id UUID NOT NULL,
  source_document_no TEXT NOT NULL,
  is_reversal BOOLEAN NOT NULL DEFAULT false,
  reversal_of_id UUID,
  created_by UUID NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
) PARTITION BY RANGE (posting_year);

-- Annual partitions (create for each year)
CREATE TABLE general_ledger_entries_2024 PARTITION OF general_ledger_entries
  FOR VALUES FROM (2024) TO (2025);

CREATE TABLE general_ledger_entries_2025 PARTITION OF general_ledger_entries
  FOR VALUES FROM (2025) TO (2026);

CREATE TABLE general_ledger_entries_2026 PARTITION OF general_ledger_entries
  FOR VALUES FROM (2026) TO (2027);

-- Future years: add a new partition each January via a scheduled job
```

## Row-Level Security (Immutability Enforcement)

```sql
-- INSERT-ONLY: Block UPDATE for ALL users including super_admin
CREATE POLICY "immutable_gl_no_update"
ON general_ledger_entries
FOR UPDATE
USING (false);

-- Block DELETE for ALL users including super_admin
CREATE POLICY "immutable_gl_no_delete"
ON general_ledger_entries
FOR DELETE
USING (false);

-- INSERT allowed only via SECURITY DEFINER posting functions (not direct client inserts)
-- Regular users cannot INSERT directly
CREATE POLICY "posting_function_insert_only"
ON general_ledger_entries
FOR INSERT
WITH CHECK (false); -- All inserts must go through SECURITY DEFINER functions

-- Tenant-scoped SELECT
CREATE POLICY "tenant_isolation_gl"
ON general_ledger_entries
FOR SELECT
USING (company_id = (auth.jwt() ->> 'company_id')::UUID);
```

## Critical Indexes

Indexes are defined on the parent table and inherited by all partitions automatically.

```sql
-- Primary financial report query pattern: account + date range
CREATE INDEX idx_gle_company_account_date
  ON general_ledger_entries (company_id, account_id, posting_date);

-- Trial Balance query: aggregate by account for a period
CREATE INDEX idx_gle_company_period
  ON general_ledger_entries (company_id, fiscal_period_id, account_id);

-- Source document drill-down (Principle 7)
CREATE INDEX idx_gle_source_document
  ON general_ledger_entries (source_document_type, source_document_id);

-- Contact-based queries (AR/AP sub-ledger reconciliation)
CREATE INDEX idx_gle_company_contact
  ON general_ledger_entries (company_id, contact_id, account_id, posting_date)
  WHERE contact_id IS NOT NULL;

-- Journal entry grouping
CREATE INDEX idx_gle_journal_entry
  ON general_ledger_entries (journal_entry_id);
```

## Standard Financial Report Queries

### Trial Balance
```sql
SELECT
  account_id,
  account_code,
  account_name,
  SUM(debit_amount) AS total_debit,
  SUM(credit_amount) AS total_credit,
  SUM(debit_amount) - SUM(credit_amount) AS balance
FROM general_ledger_entries
WHERE company_id = :company_id
  AND posting_date BETWEEN :period_start AND :period_end
GROUP BY account_id, account_code, account_name
ORDER BY account_code;
```

### GL Account Inquiry (running balance)
```sql
SELECT
  posting_date,
  description,
  source_document_no,
  debit_amount,
  credit_amount,
  SUM(debit_amount - credit_amount) OVER (
    PARTITION BY account_id
    ORDER BY posting_date, id
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
  ) AS running_balance
FROM general_ledger_entries
WHERE company_id = :company_id
  AND account_id = :account_id
  AND posting_date BETWEEN :start_date AND :end_date
ORDER BY posting_date, id;
```
