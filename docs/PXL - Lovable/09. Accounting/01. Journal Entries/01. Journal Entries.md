# PXL ERP Blueprint: Journal Entries

## Module Overview
The Journal Entries module is the fundamental core of double-entry bookkeeping within the PXL ERP. In the Philippine context, this module captures all manual adjustments, accruals, prepayments, depreciation, and corrections that are not generated by sub-ledgers (AR or AP). It strictly enforces BIR-compliant double-entry accounting by demanding that Total Debits exactly equal Total Credits before posting. It also captures critical audit trail information — reference documents and subsidiary ledger tagging — ensuring the General Ledger can be traced back to original source documents during a BIR CAS audit.

### Line Item Level
**CRITICAL VALIDATION:** The system must enforce `SUM(debit_amount) = SUM(credit_amount)` across all lines. The **Post** button is disabled until this condition is met.

**GL Posting Behavior:** Upon posting, line items are committed to the General Ledger as immutable records. Any correction to a posted JE must be done via a Reversal entry — direct editing of a posted JE is blocked at both the application and database layers (see RLS Policy below).

## Supabase Database Architecture

### Table 1: `journal_entries` (Header)

**Critical Database Rules:**
1. Composite unique constraint on `(company_id, je_number)` — prevents duplicate JE numbers within the same tenant.
2. `status` must be enforced as an ENUM: `draft`, `posted`, `reversed`. No other values permitted.
3. `posted_at` timestamp is set by the system on the Post action and is never editable.

| Column Name | Data Type | Rules | Description |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System identifier. |
| `company_id` | UUID | Required, Foreign Key | Links to `companies.id`. Tenant isolation anchor. |
| `branch_id` | UUID | Nullable, Foreign Key | Links to `branches.id`. |
| `je_number` | Text | Required | Formatted JE number (e.g., `JE-2026-0001`). |
| `date` | Date | Required | Transaction date. Must fall within an open `accounting_periods` record. |
| `fiscal_period_id` | UUID | Required, Foreign Key | Links to `accounting_periods.id`. Set on save; blocks posting to closed periods at DB layer. |
| `description` | Text | Required | General memo for the entry. |
| `reference` | Text | Nullable | Source document reference. |
| `currency_code` | Text | Required, Default `PHP` | Links to `currencies.code`. |
| `total_debit` | Numeric(15,2) | Required, Default 0 | Computed sum of all line debits. Must equal `total_credit` before posting. |
| `total_credit` | Numeric(15,2) | Required, Default 0 | Computed sum of all line credits. Must equal `total_debit` before posting. |
| `status` | Text | Required, Default `draft` | ENUM: `draft`, `posted`, `reversed`. |
| `posted_at` | Timestamp | Nullable | Set by system on Post action. Never editable by any user. |
| `posted_by` | UUID | Nullable, Foreign Key | Links to `users.id`. The user who executed the Post action. |
| `source_document_type` | Text | Nullable | Polymorphic type discriminator. ENUM: `sales_invoice`, `purchase_invoice`, `fund_transfer`, `inter_branch_transfer`, `bank_adjustment`, `petty_cash_voucher`, `pcf_replenishment`, `asset_acquisition`, `asset_disposal`, `asset_impairment`, `asset_transfer`, `depreciation_run`, `amortization_schedule`, `manual`. When `NULL`, the JE is a standalone manual entry. |
| `source_document_id` | UUID | Nullable | Polymorphic FK to the originating document's `id`. Combined with `source_document_type`, enables infinite drill-down from any GL entry back to its source document (Principle 7). |
| `reversal_of_id` | UUID | Nullable, Foreign Key | Self-reference to `journal_entries.id`. Populated when this JE is a reversal of another. |
| `created_by` | UUID | Required, Foreign Key | Links to `users.id`. |
| `updated_by` | UUID | Nullable, Foreign Key | Links to `users.id`. |
| `created_at` | Timestamp | Auto | Creation timestamp. |
| `updated_at` | Timestamp | Auto | Last update timestamp. |

**Composite Unique Constraint:** `UNIQUE(company_id, je_number)`

### Table 2: `journal_entry_lines` (Line Items)

**Critical Database Rules:**
1. `journal_entry_id` is the mandatory foreign key linking every line to its parent header. Without this, lines are orphaned.
2. Both `debit_amount` and `credit_amount` columns are required. A line must have one non-zero and one zero. A DB CHECK constraint enforces `(debit_amount = 0 AND credit_amount > 0) OR (debit_amount > 0 AND credit_amount = 0)`.
3. Neither amount may be negative.

| Column Name | Data Type | Rules | Description |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System identifier. |
| `journal_entry_id` | UUID | Required, Foreign Key | Links to `journal_entries.id` (ON DELETE CASCADE). |
| `company_id` | UUID | Required, Foreign Key | Links to `companies.id`. Denormalized for RLS policy efficiency. |
| `account_id` | UUID | Required, Foreign Key | Links to `chart_of_accounts.id`. Must be a postable account. |
| `debit_amount` | Numeric(15,2) | Required, Default 0.00 | Debit value for this line. |
| `credit_amount` | Numeric(15,2) | Required, Default 0.00 | Credit value for this line. |
| `line_description` | Text | Nullable | Specific memo for this line item. |
| `cost_center_id` | UUID | Nullable, Foreign Key | Links to `cost_centers.id`. |
| `branch_id` | UUID | Nullable, Foreign Key | Links to `branches.id`. |
| `created_by` | UUID | Required, Foreign Key | Links to `users.id`. |
| `updated_by` | UUID | Nullable, Foreign Key | Links to `users.id`. |
| `created_at` | Timestamp | Auto | Creation timestamp. |
| `updated_at` | Timestamp | Auto | Last update timestamp. |

**DB CHECK Constraint:**
```sql
CONSTRAINT chk_debit_credit_exclusive
CHECK (
  (debit_amount > 0 AND credit_amount = 0) OR
  (debit_amount = 0 AND credit_amount > 0)
)
```

## Row-Level Security (RLS) Immutability Policy

**This is a BIR CAS compliance requirement.** Once a Journal Entry is posted, it must be unalterable at the database layer — not just the application layer. The following RLS policies must be implemented in Supabase.

### Policy 1: Block UPDATE on Posted/Reversed Journal Entries
```sql
-- Applied to the journal_entries table
CREATE POLICY "block_update_on_posted_je"
ON journal_entries
FOR UPDATE
USING (status = 'draft');
-- Effect: Only Draft JEs can be updated. Any attempt to UPDATE a posted or
-- reversed JE is silently blocked (returns 0 rows affected) or raises an error
-- depending on application-layer handling. Combine with application error messaging.
```

### Policy 2: Block DELETE on All Journal Entries
```sql
-- Posted JEs cannot be deleted. Voided pre-numbered docs must remain on record.
CREATE POLICY "block_delete_all_je"
ON journal_entries
FOR DELETE
USING (false);
-- Effect: No JE can ever be hard-deleted. Corrections are made via Reversal entries only.
```

### Policy 3: Block UPDATE/DELETE on All Journal Entry Lines
```sql
-- Line items of a posted JE cannot be altered.
CREATE POLICY "block_update_on_posted_je_lines"
ON journal_entry_lines
FOR UPDATE
USING (
  (SELECT status FROM journal_entries WHERE id = journal_entry_id) = 'draft'
);

CREATE POLICY "block_delete_on_posted_je_lines"
ON journal_entry_lines
FOR DELETE
USING (
  (SELECT status FROM journal_entries WHERE id = journal_entry_id) = 'draft'
);
```

### Policy 4: Tenant Isolation
```sql
-- Users can only see JEs belonging to their company.
CREATE POLICY "tenant_isolation_je"
ON journal_entries
FOR ALL
USING (company_id = auth.jwt() ->> 'company_id');
```

## Period Validation — Pre-Flight Check

Before any JE can be saved or posted, the application must execute:
```sql
SELECT ap.is_closed, fl.lock_date
FROM accounting_periods ap
LEFT JOIN fiscal_locks fl
  ON fl.company_id = ap.company_id AND fl.module_name = 'journal_entries'
WHERE ap.company_id = :company_id
  AND :transaction_date BETWEEN ap.start_date AND ap.end_date;
```
**Logic:**
1. If `is_closed = true` → **HARD BLOCK.** Cannot post regardless of role.
2. If `lock_date IS NOT NULL AND :transaction_date <= lock_date` → **MODULE LOCK BLOCK.** Cannot post to this date.
3. If neither → Allow posting.
