# PXL ERP Blueprint: Purchase Invoice

## Module Overview
The Purchase Invoice (also called a Vendor Bill) is the BIR-recognized document recording the company's obligation to a supplier for goods received or services rendered. It generates an AP sub-ledger entry (debit Expense/Asset/Input VAT, credit AP), posts to the GL, and populates the Input VAT Review, Purchase Invoice Register, and BIR SLSP (Summary List of Purchases). It is also the trigger for Expanded Withholding Tax (EWT) computation — a BIR-mandated deduction that must be automatically computed based on supplier and item classification.

## Top Withholding Agent (TWA) — Automatic EWT Enforcement

**This is a BIR-mandated process. It is NOT optional.**

If the company's `compliance_profiles.is_twa = true` AND `compliance_profiles.twa_auto_ewt_enabled = true`, the system **MUST** automatically compute and apply EWT on every Purchase Invoice at the point of line-item entry. The behavior is:

1. When an Item line is entered, the system resolves the applicable EWT code in this priority order:
   - Item-level EWT override (`items.default_ewt_code_id`) if set.
   - Supplier-level default EWT (`suppliers.default_ewt_code_id`) if no item override.
   - If `suppliers.ewt_exempt = true` → No EWT applied for that supplier. Log the exemption basis.
2. EWT amount is computed as: `Line Net Amount × EWT Rate`.
3. The EWT amount is displayed on the invoice as a deduction (reducing the net amount due to the supplier).
4. On posting, EWT is credited to the **EWT Payable** GL account (from `ewt_codes` → `tax_codes.gl_account_id`).
5. The EWT is NOT optional or dismissible by the user without a supervisor override with a logged reason.
6. A BIR Form 2307 record is created for each posted Purchase Invoice with EWT > 0, linking this PI to the `form_2307_issued` tracking table.

## GL Posting Logic (System-Generated on Post)

| # | Account | Debit | Credit | Source |
| :--- | :--- | :--- | :--- | :--- |
| 1 | Expense / Inventory Account (per line) | Net Amount | | `items.purchase_expense_account_id` or `items.inventory_account_id` → category → GL config |
| 2 | Input VAT | Input VAT Amount | | `tax_codes.gl_account_id` for the input VAT code |
| 3 | Accounts Payable (AP Control) | | Gross Amount Payable | `suppliers.ap_account_id` |
| 4 | EWT Payable | | Total EWT Withheld | `ewt_codes` → `tax_codes.gl_account_id` |

*The net effect: AP payable = Gross Amount Payable (before EWT). EWT is a separate liability owed to the BIR, not the supplier. The supplier receives only Net Amount Payable.*

## Supabase Database Architecture

### Table 1: `purchase_invoices` (Header)

**Critical Database Rules:**
1. Composite unique constraint on `(company_id, pi_number)`.
2. Composite unique constraint on `(company_id, supplier_id, supplier_invoice_number)` — prevents duplicate entry of the same supplier invoice.
3. `posted_at` and `posted_by` are system-set. Never editable.
4. `ewt_total` must be computed and stored on posting, not recalculated, to ensure MAP consistency.

| Column Name | Data Type | Rules | Description |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System identifier. |
| `company_id` | UUID | Required, Foreign Key | Links to `companies.id`. |
| `branch_id` | UUID | Required, Foreign Key | Links to `branches.id`. |
| `pi_number` | Text | Required | Internal sequential PI number. |

| `supplier_tin_snapshot` | Text | Required | Supplier TIN at time of posting. Used verbatim on 2307 and MAP. |
| `supplier_address_snapshot` | Text | Required | Supplier address at time of posting. |
| `payment_terms_id` | UUID | Required, Foreign Key | Links to `payment_terms.id`. |

**Composite Unique Constraints:**
- `UNIQUE(company_id, pi_number)`
- `UNIQUE(company_id, supplier_id, supplier_invoice_number)` — duplicate invoice guard.

## Row-Level Security — Immutability Policy

```sql
-- Block UPDATE on posted or cancelled Purchase Invoices
CREATE POLICY "block_update_on_posted_pi"
ON purchase_invoices
FOR UPDATE
USING (status = 'draft' OR status = 'approved');

-- Block DELETE on all Purchase Invoices
CREATE POLICY "block_delete_all_pi"
ON purchase_invoices
FOR DELETE
USING (false);

-- Tenant isolation
CREATE POLICY "tenant_isolation_pi"
ON purchase_invoices
FOR ALL
USING (company_id = auth.jwt() ->> 'company_id');
```
