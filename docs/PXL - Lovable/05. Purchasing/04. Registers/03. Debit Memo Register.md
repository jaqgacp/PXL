# PXL ERP Blueprint: Debit Memo Register

## Module Overview
The Debit Memo Register is a specialized reporting tool that provides a chronological summary of all Debit Memos issued to suppliers. In the Philippines, tracking these deductions is vital because they represent claims against suppliers for returns, damages, or pricing errors. This register ensures that accounting teams do not lose sight of these claims and that they are properly applied against upcoming Vendor Bills or refunded by the supplier, ensuring accurate Accounts Payable balances.

## What You Will See on the Screen (The Dashboard)
When you open the "Debit Memo Register" page, you will see a detailed, read-only reporting table.

### The Action Bar (Top of the page):
* **Date Range Filter:** Select the period to view (e.g., This Month, Last Quarter).
* **Supplier Filter:** Search for memos issued to a specific vendor.
* **Status Filter:** View by status (e.g., Draft, Sent, Acknowledged, Cancelled).
* **Export Button:** Downloads the register to Excel or CSV.
* **Print Button:** Generates a formal, print-ready PDF of the register.

### The Data List (The main report table):
* Date Issued
* DM Number
* Supplier Name
* Reference Document (e.g., Bill or RR)
* Reason
* Total Amount
* Status

## What Information We Need to Capture (The Data Fields)
As a reporting module, there is no data entry. It dynamically pulls data from the `supplier_debit_memos` table.

### Report Columns Map
| Column Name | Source | Description |
| :--- | :--- | :--- |
| Date | `dm_date` | The date the memo was issued. |
| DM Number | `dm_number` | The ERP generated reference number. |
| Supplier | `suppliers.registered_name`| The vendor receiving the chargeback. |
| Ref Document | `reference_doc_id` | The related RR or Bill number. |
| Reason | `reason` | The explanation for the deduction. |
| Total Amount | `total_amount` | The value of the chargeback. |
| Status | `status` | Current state of the memo. |

## The Supabase Database Architecture (For Lovable)
We will create a simple **SQL View** that flattens the Debit Memo data, making it fast and easy to query for the frontend reporting grid.

### View: `vw_debit_memo_register`
This view pulls the header details and joins the necessary supplier and user names.

| Column Name | Data Type | Rules | What it stores |
| :--- | :--- | :--- | :--- |
| `dm_id` | UUID | Required | Links to `supplier_debit_memos.id`. |
| `dm_number` | Text | Required | System reference. |
| `dm_date` | Date | Required | Date issued. |
| `supplier_name` | Text | Required | Joined from `suppliers.registered_name`. |
| `reference_doc` | Text | Nullable | Formatted string of the referenced document. |
| `reason` | Text | Required | Explanation. |
| `total_amount` | Numeric | Required | Value of the memo. |
| `status` | Text | Required | Current status. |
| `created_by_name`| Text | Required | Joined from `users.name`. |
| `created_at` | Timestamp | Required | Timestamp of creation. |

**Critical UX/GL Rule:** The system MUST auto-fetch the Item's default Revenue, Expense, COGS, and Inventory GL accounts in the background to auto-generate the Journal Entry. Encoders MUST NEVER manually select a GL account for standard items.
