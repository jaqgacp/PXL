# PXL ERP Blueprint: Payment Register

## Module Overview
The Payment Register (also known as the Cash Disbursements Journal) is a fundamental accounting report. It provides a chronological listing of all outgoing payments made by the company during a specific period. This includes both Payment Vouchers (payments against Accounts Payable) and Cash Purchases (direct expenses). In the Philippines, this register is crucial for bank reconciliation, cash flow analysis, and preparing the official books of accounts for BIR audits.

## What You Will See on the Screen (The Dashboard)
When you open the "Payment Register" page, you will see a detailed, read-only reporting table.

### The Action Bar (Top of the page):
* **Date Range Filter:** Select the period to view (e.g., This Month, Last Month).
* **Payment Account Filter:** Isolate payments made from a specific Bank or Cash account.
* **Supplier Filter:** Search for payments made to a specific vendor.
* **Payment Method Filter:** View only Checks, Cash, or Bank Transfers.
* **Export Button:** Downloads the register to Excel or CSV.
* **Print Button:** Generates a formal, print-ready PDF of the disbursements.

### The Data List (The main report table):
* Date
* Document Number (PV or CP)
* Payee / Supplier Name
* Payment Method
* Check / Ref Number
* Bank/Cash Account
* Total Amount Paid
* Status (e.g., Released, Cleared)

## What Information We Need to Capture (The Data Fields)
As a reporting module, there is no data entry. It dynamically pulls and unions data from the `payment_vouchers` and `cash_purchases` tables.

### Report Columns Map
| Column Name | Source | Description |
| :--- | :--- | :--- |
| Date | `pv_date` or `transaction_date`| The date the payment was recorded. |
| Doc Type | System Defined | "Payment Voucher" or "Cash Purchase". |
| System No. | `pv_number` or `cp_number` | The ERP generated document number. |
| Payee | `supplier_name` or `payee_name` | Name of the entity receiving the funds. |
| Method | `payment_method` | Check, Cash, Transfer. |
| Reference | `check_number` or `reference_no`| The physical check or transfer tracking number. |
| Bank Account | `chart_of_accounts.name` | The source GL account. |
| Amount Paid | `total_amount` | The total cash outflow. |
| Status | `status` | Current state of the payment. |

## The Supabase Database Architecture (For Lovable)
We will create an optimized **SQL View** that safely combines data from both disbursement modules into a single, chronological ledger.

### View: `vw_payment_register`
This view unions the headers of Payment Vouchers and Cash Purchases.

| Column Name | Data Type | Rules | What it stores |
| :--- | :--- | :--- | :--- |
| `transaction_id`| UUID | Required | Links to `payment_vouchers.id` or `cash_purchases.id`. |
| `document_type` | Text | Required | `payment_voucher` or `cash_purchase`. |
| `document_number`| Text | Required | System reference. |
| `transaction_date`| Date | Required | Date of disbursement. |
| `payee_name` | Text | Required | Name of the supplier or payee. |
| `payment_method`| Text | Required | `check`, `cash`, `transfer`. |
| `reference_number`| Text | Nullable | Check or bank transfer number. |
| `bank_account` | Text | Required | Joined from `chart_of_accounts`. |
| `total_amount` | Numeric | Required | Total cash out. |
| `status` | Text | Required | Status of the document. |
| `created_by_name`| Text | Required | Joined from `users.name`. |
| `created_at` | Timestamp | Required | Timestamp of creation. |

**Critical UX/GL Rule:** The system MUST auto-fetch the Item's default Revenue, Expense, COGS, and Inventory GL accounts in the background to auto-generate the Journal Entry. Encoders MUST NEVER manually select a GL account for standard items.
