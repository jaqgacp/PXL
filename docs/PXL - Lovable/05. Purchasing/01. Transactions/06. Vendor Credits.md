# PXL ERP Blueprint: Vendor Credits

## Module Overview
The Vendor Credits module is used to record credits issued by a supplier to our company. This usually happens when we return goods, when we are overbilled, or when the supplier grants a retroactive discount. In the Philippines, the supplier must issue a Credit Note (Credit Memo) referencing the original Sales Invoice. Logging a Vendor Credit in the ERP reduces our Accounts Payable to that supplier, and the credit can be applied to offset future or existing Vendor Bills.

## What You Will See on the Screen (The Dashboard)
When you open the "Vendor Credits" page, you will see a clean workspace.

### The Action Bar (Top of the page):
* **Search Bar:** Quickly find a Vendor Credit by VC Number or Supplier Name.
* **Filter:** Dropdowns to filter by Date Range and Status (Draft, Open, Applied, Cancelled).
* **Customize View:** Choose which columns to show or hide.
* **Export Button:** Downloads the list to Excel or CSV.
* **Print Button:** Generates a print-ready view of the Vendor Credit.
* **Create New Credit Button:** Prominent button to log a new supplier credit.
* **Import Button:** Bulk upload credits from an Excel/CSV file.

### The Data List (The main table):
* Credit Date
* VC Number
* Supplier Name
* Supplier CM No.
* Total Amount
* Remaining Balance
* Status (Draft, Open, Applied, Cancelled)

## What Information We Need to Capture (The Data Fields)
The form is divided into Header and Line Items.

**Data Inheritance:** A Vendor Credit can be created standalone or from a specific Vendor Bill. If created from a Bill, it inherits the supplier, items, and tax codes.

### Section 1: Header Details
| Field Name | Description | UI Component | Required? | Inheritance |
| :--- | :--- | :--- | :--- | :--- |
| VC Number | Unique, auto-generated system number. | Auto-computed | Yes | None |
| Credit Date | Date of the credit memo. | Date Picker | Yes | None |
| Supplier | The vendor issuing the credit. | Dropdown linked to Master Data. UPON SELECTION, system MUST instantly auto-fill: Address, TIN, Credit Terms, and Tax Type. | Yes | Auto-filled from Bill if applicable |
| Reference Bill | The Vendor Bill this credit applies to. | Search/Dropdown | Optional | None |
| Supplier CM No. | The actual Credit Memo number from supplier. | Text Input | Yes | None |
| Status | Current state of the credit. | Auto-computed | Yes | None |
| Remarks | Reason for the credit. | Text Area | Optional | None |

### Section 2: Line Items
| Field Name | Description | UI Component | Required? | Inheritance |
| :--- | :--- | :--- | :--- | :--- |
| Item | The Item being credited. | Dropdown linked to Items. UPON SELECTION, instantly auto-fills Description, UOM, Unit Price/Cost, and Tax Code. | Yes | Auto-filled from Reference Bill |
| Description | Details of the credit. | Text Area | Optional | Auto-filled from Reference Bill |
| Quantity | Credited quantity. | Number Input | Optional | None |
| Unit Price | Amount per unit. | Number Input | Optional | Auto-filled from Reference Bill |
| Gross Amount | Total credit before tax. | Auto-computed | Yes | None |

## The Supabase Database Architecture (For Lovable)

**Critical Database Rule for Lovable:**
- Ensure a composite unique constraint on `company_id` and `vc_number` if a `company_id` is used, otherwise `vc_number` must be unique.

**Critical UX/GL Rule:** The system MUST auto-fetch the Item's default Revenue, Expense, COGS, and Inventory GL accounts in the background to auto-generate the Journal Entry. Encoders MUST NEVER manually select a GL account for standard items.

### Table 1: `vendor_credits` (Header Table)
| Column Name | Data Type | Rules | What it stores |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System ID for the Vendor Credit. |
| `vc_number` | Text | Required, Unique | Auto-generated number (e.g., "VC-2023-0001"). |
| `credit_date` | Date | Required | Date of the supplier credit memo. |
| `supplier_id` | UUID | Required, Foreign Key | Links to `suppliers.id`. |
| `supplier_cm_no` | Text | Required | Supplier's reference CM number. |
| `reference_bill_id`| UUID | Nullable, Foreign Key | Links to `vendor_bills.id`. |
| `status` | Text | Required | `draft`, `open`, `applied`, `cancelled`. |
| `total_amount` | Numeric | Required | Total credit amount. |
| `remaining_balance`| Numeric | Required | Unapplied portion of the credit. |
| `remarks` | Text | Nullable | Reason for credit. |
| `created_by` | UUID | Required, Foreign Key | Links to `users.id`. |
| `updated_by` | UUID | Nullable, Foreign Key | Links to `users.id`. |
| `created_at` | Timestamp | Auto | Record creation timestamp. |
| `updated_at` | Timestamp | Auto | Record update timestamp. |

### Table 2: `vendor_credit_lines` (Line Items Table)
| Column Name | Data Type | Rules | What it stores |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System ID for the line item. |
| `vc_id` | UUID | Required, Foreign Key | Links to `vendor_credits.id`. |
| `item_id` | UUID | Required, Foreign Key | Links to `items.id`. |
| `description` | Text | Nullable | Line detail. |
| `quantity` | Numeric | Nullable | Credited quantity. |
| `unit_price` | Numeric | Nullable | Amount per unit. |
| `gross_amount` | Numeric | Required | Pre-tax credit amount. |
| `vat_code_id` | UUID | Required, Foreign Key | Links to `tax_codes.id`. |
| `net_amount` | Numeric | Required | Final line amount. |
| `created_by` | UUID | Required, Foreign Key | Links to `users.id`. |
| `updated_by` | UUID | Nullable, Foreign Key | Links to `users.id`. |
| `created_at` | Timestamp | Auto | Record creation timestamp. |
| `updated_at` | Timestamp | Auto | Record update timestamp. |
