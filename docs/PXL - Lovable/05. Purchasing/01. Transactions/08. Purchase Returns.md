# PXL ERP Blueprint: Purchase Returns

## Module Overview
The Purchase Returns module tracks the physical movement of inventory back to a supplier. Whether the items were defective, incorrectly delivered, or simply no longer needed, this document ensures that inventory levels are properly reduced. In the Philippines, a Purchase Return usually accompanies a Delivery Receipt from our end and triggers the request for a Supplier Credit Memo. It is tightly integrated with the Receiving Report to ensure we don't return more than what we actually received.

## What You Will See on the Screen (The Dashboard)
When you open the "Purchase Returns" page, you will see a clean workspace.

### The Action Bar (Top of the page):
* **Search Bar:** Quickly find a Return by PReturn Number or Supplier Name.
* **Filter:** Dropdowns to filter by Date Range, Warehouse Location, and Status (Draft, Shipped, Completed, Cancelled).
* **Customize View:** Choose which columns to show or hide.
* **Export Button:** Downloads the list to Excel or CSV.
* **Print Button:** Generates a print-ready return slip.
* **Create New Return Button:** Prominent button to process an inventory return.

### The Data List (The main table):
* Return Date
* Return Number
* Supplier Name
* Reference RR Number
* Total Quantity Returned
* Status (Draft, Shipped, Completed, Cancelled)

## What Information We Need to Capture (The Data Fields)
The form is divided into Header and Line Items.

**Data Inheritance:** A Purchase Return inherits entirely from a Receiving Report (RR). The system auto-populates the Supplier, items received, and limits the returnable quantity to what was originally accepted in the RR.

### Section 1: Header Details
| Field Name | Description | UI Component | Required? | Inheritance |
| :--- | :--- | :--- | :--- | :--- |
| Return Number | Unique, auto-generated system number. | Auto-computed | Yes | None |
| Return Date | Date the goods are returned. | Date Picker | Yes | None |
| Reference RR | The Receiving Report of the items. | Search/Dropdown | Yes | None |
| Supplier | The vendor receiving the goods. | Dropdown linked to Master Data. UPON SELECTION, system MUST instantly auto-fill: Address, TIN, Credit Terms, and Tax Type. (Auto-filled if from RR) | Yes | Auto-filled from RR |
| Location | Warehouse dispatching the return. | Dropdown | Yes | Auto-filled from RR |
| Status | Current state of the return. | Auto-computed | Yes | None |
| Remarks | Reason for returning. | Text Area | Optional | None |

### Section 2: Line Items
| Field Name | Description | UI Component | Required? | Inheritance |
| :--- | :--- | :--- | :--- | :--- |
| Item | The product being returned. | Dropdown linked to Items. UPON SELECTION, instantly auto-fills Description, UOM, Unit Price/Cost, and Tax Code. | Yes | Auto-filled from RR |
| Max Returnable Qty| The total quantity originally received. | Read-only | Yes | Auto-filled from RR |
| Return Qty | Actual quantity being sent back. | Number Input | Yes | User inputs (must be <= Max) |
| UOM | Unit of measure. | Dropdown linked to UOM master | Yes | Auto-filled from RR |
| Reason Code | Categorized reason for return. | Dropdown | Yes | None |

## The Supabase Database Architecture (For Lovable)

**Critical Database Rule for Lovable:**
- Ensure a composite unique constraint on `company_id` and `return_number` if a `company_id` is used, otherwise `return_number` must be unique.

**Critical UX/GL Rule:** The system MUST auto-fetch the Item's default Revenue, Expense, COGS, and Inventory GL accounts in the background to auto-generate the Journal Entry. Encoders MUST NEVER manually select a GL account for standard items.

### Table 1: `purchase_returns` (Header Table)
| Column Name | Data Type | Rules | What it stores |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System ID for the Return. |
| `return_number` | Text | Required, Unique | Auto-generated number (e.g., "PRT-2023-0001"). |
| `return_date` | Date | Required | Date goods leave warehouse. |
| `rr_id` | UUID | Required, Foreign Key | Links to `receiving_reports.id`. |
| `supplier_id` | UUID | Required, Foreign Key | Links to `suppliers.id`. |
| `location_id` | UUID | Required, Foreign Key | Links to `locations.id`. |
| `status` | Text | Required | `draft`, `shipped`, `completed`, `cancelled`. |
| `remarks` | Text | Nullable | Overall comments. |
| `created_by` | UUID | Required, Foreign Key | Links to `users.id`. |
| `updated_by` | UUID | Nullable, Foreign Key | Links to `users.id`. |
| `created_at` | Timestamp | Auto | Record creation timestamp. |
| `updated_at` | Timestamp | Auto | Record update timestamp. |

### Table 2: `purchase_return_lines` (Line Items Table)
| Column Name | Data Type | Rules | What it stores |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System ID for the line item. |
| `return_id` | UUID | Required, Foreign Key | Links to `purchase_returns.id`. |
| `rr_line_id` | UUID | Required, Foreign Key | Links to `receiving_report_lines.id` for validation. |
| `item_id` | UUID | Required, Foreign Key | Links to `items.id`. |
| `return_qty` | Numeric | Required | Quantity returned. |
| `uom_id` | UUID | Required, Foreign Key | Links to `uoms.id`. |
| `reason` | Text | Required | E.g., `damaged`, `wrong_item`. |
| `created_by` | UUID | Required, Foreign Key | Links to `users.id`. |
| `updated_by` | UUID | Nullable, Foreign Key | Links to `users.id`. |
| `created_at` | Timestamp | Auto | Record creation timestamp. |
| `updated_at` | Timestamp | Auto | Record update timestamp. |
