# PXL ERP Blueprint: Payment Vouchers

## Module Overview
The Payment Voucher (PV) module, also known as Check Voucher or Disbursement Voucher, is used to record payments made against open Vendor Bills. It reduces the Accounts Payable balance and credits the Cash/Bank account. In the Philippines, the PV is an essential document that requires internal approval signatures before a check is released or a bank transfer is executed. It also triggers the actual printing of BIR Form 2307 (Certificate of Creditable Tax Withheld at Source) to be given to the supplier.

## What You Will See on the Screen (The Dashboard)
When you open the "Payment Vouchers" page, you will see a clean workspace.

### The Action Bar (Top of the page):
* **Search Bar:** Quickly find a PV by PV Number, Supplier Name, or Check Number.
* **Filter:** Dropdowns to filter by Date Range, Payment Account, and Status (Draft, Approved, Released, Cleared, Cancelled).
* **Customize View:** Choose which columns to show or hide.
* **Export Button:** Downloads the list to Excel or CSV.
* **Print Button:** Generates a print-ready view of the PV.
* **Create New PV Button:** Prominent button to process a payment.

### The Data List (The main table):
* PV Date
* PV Number
* Supplier Name
* Payment Account
* Check/Ref No.
* Total Amount Paid
* Status (Draft, Approved, Released, Cleared, Cancelled)

## What Information We Need to Capture (The Data Fields)
The form is divided into Header and Line Items (which represent the Bills being paid).

**Data Inheritance:** A Payment Voucher inherits data from selected open Vendor Bills. When a user selects a supplier, the system pulls all open bills for that supplier.

### Section 1: Header Details
| Field Name | Description | UI Component | Required? | Inheritance |
| :--- | :--- | :--- | :--- | :--- |
| PV Number | Unique, auto-generated system number. | Auto-computed | Yes | None |
| PV Date | Date of the voucher. | Date Picker | Yes | None |
| Supplier | The vendor being paid. | Dropdown linked to Master Data. UPON SELECTION, system MUST instantly auto-fill: Address, TIN, Credit Terms, and Tax Type. | Yes | None |
| Payment Method | Check, Cash, Bank Transfer. | Dropdown | Yes | None |
| Payment Account | Bank or Cash account source. | Dropdown | Yes | None |
| Check Number | Number of the issued check. | Text Input | Conditional | None |
| Check Date | Date printed on the check. | Date Picker | Conditional | None |
| Status | Current state of the PV. | Auto-computed | Yes | None |
| Remarks | Additional notes. | Text Area | Optional | None |

### Section 2: Line Items (Bills to Pay)
| Field Name | Description | UI Component | Required? | Inheritance |
| :--- | :--- | :--- | :--- | :--- |
| Bill Number | The Vendor Bill being paid. | Dropdown | Yes | Auto-filled from Supplier's open bills |
| Bill Date | Date of the invoice. | Read-only | Yes | Auto-filled from Vendor Bill |
| Original Amount | Total amount of the bill. | Read-only | Yes | Auto-filled from Vendor Bill |
| Amount Due | Remaining unpaid balance. | Read-only | Yes | Auto-filled from Vendor Bill |
| Discount | Any discount applied at payment. | Number Input | Optional | None |
| Amount Paid | Amount to apply to this bill. | Number Input | Yes | Defaults to Amount Due |
| EWT Deducted | Withholding tax applied. | Read-only | Yes | Auto-filled from Vendor Bill |

## The Supabase Database Architecture (For Lovable)

**Critical Database Rule for Lovable:**
- Ensure a composite unique constraint on `company_id` and `pv_number` if a `company_id` is used, otherwise `pv_number` must be unique.

**Critical UX/GL Rule:** The system MUST auto-fetch the Item's default Revenue, Expense, COGS, and Inventory GL accounts in the background to auto-generate the Journal Entry. Encoders MUST NEVER manually select a GL account for standard items.

### Table 1: `payment_vouchers` (Header Table)
| Column Name | Data Type | Rules | What it stores |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System ID for the PV. |
| `pv_number` | Text | Required, Unique | Auto-generated number (e.g., "PV-2023-0001"). |
| `pv_date` | Date | Required | Date of payment preparation. |
| `supplier_id` | UUID | Required, Foreign Key | Links to `suppliers.id`. |
| `payment_method` | Text | Required | `check`, `cash`, `transfer`. |
| `payment_account_id`| UUID | Required, Foreign Key | Links to `chart_of_accounts.id`. |
| `check_number` | Text | Nullable | Check or reference number. |
| `check_date` | Date | Nullable | Post-date of the check. |
| `status` | Text | Required | `draft`, `approved`, `released`, `cleared`, `cancelled`. |
| `total_amount` | Numeric | Required | Total cash out. |
| `remarks` | Text | Nullable | Extra details. |
| `created_by` | UUID | Required, Foreign Key | Links to `users.id`. |
| `updated_by` | UUID | Nullable, Foreign Key | Links to `users.id`. |
| `created_at` | Timestamp | Auto | Record creation timestamp. |
| `updated_at` | Timestamp | Auto | Record update timestamp. |

### Table 2: `payment_voucher_lines` (Line Items Table)
| Column Name | Data Type | Rules | What it stores |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System ID for the line item. |
| `pv_id` | UUID | Required, Foreign Key | Links to `payment_vouchers.id`. |
| `bill_id` | UUID | Required, Foreign Key | Links to `vendor_bills.id`. |
| `amount_paid` | Numeric | Required | Actual amount paid against the bill. |
| `discount_applied`| Numeric | Default: 0 | Prompt payment discount. |
| `ewt_deducted` | Numeric | Required | EWT amount (for 2307 generation). |
| `created_by` | UUID | Required, Foreign Key | Links to `users.id`. |
| `updated_by` | UUID | Nullable, Foreign Key | Links to `users.id`. |
| `created_at` | Timestamp | Auto | Record creation timestamp. |
| `updated_at` | Timestamp | Auto | Record update timestamp. |
