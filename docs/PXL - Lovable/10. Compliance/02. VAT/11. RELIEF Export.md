# PXL ERP Blueprint: RELIEF Export (Reconciliation of Listings for Enforcement)

## Module Overview
The BIR RELIEF (Reconciliation of Listings for Enforcement) system is a data submission requirement where taxpayers provide their sales and purchases data to the BIR for cross-matching with counterparty returns. RELIEF data is submitted via `.dat` files to the BIR eFPS system. Unlike the SLSP (which is quarterly), RELIEF submissions cover specific periods as mandated by the BIR's enforcement calendar.

The critical architectural requirement is that every column in the RELIEF DAT file specification must map to an actual database field with real counterparty data. A generic `transaction_id` and `amount` schema cannot generate a valid RELIEF DAT file.

**RELIEF differs from SLSP in one critical aspect:** RELIEF includes both Sales and Purchases in a single unified submission, and the BIR's cross-matching algorithm requires exact TIN-to-TIN reconciliation across taxpayers' filings.

**Activation Condition:** Visible when `compliance_profiles.relief_required = true`.

## Dashboard UI

### The Action Bar
* **Search Bar:** Find by period or counterparty TIN.
* **Filter:** By Year, Period, Transaction Type (Sales/Purchases), Status.
* **Generate RELIEF Button:** Auto-pulls all posted transactions for the reporting period.
* **Export DAT Button:** Generates the BIR RELIEF `.dat` file in the exact column format required by the BIR validation system.
* **Export Excel Button:** Human-readable review copy.
* **Validate DAT Button:** Runs the BIR validation logic client-side before submission to catch TIN format errors, missing fields, and amount mismatches.
* **Mark as Filed Button:** Updates status to `filed` and records the BIR acknowledgement number.

### The Data List (Summary View)
* Period
* Total Sales Records
* Total Purchase Records
* Total Sales Amount
* Total Purchase Amount
* Validation Status
* Filing Status

**Pagination Mandate:** Server-side pagination via Supabase `.range(from, to)`. Default page size: 50. Line-item drill-down uses a separate paginated view.

## BIR RELIEF DAT File Format â€” Required Columns

The BIR RELIEF validation software requires the following fields exactly. The system generates the DAT file by querying these columns from `compliance_relief_export_lines`.

| DAT Column # | Column Name | Format | Source |
| :--- | :--- | :--- | :--- |
| 1 | Taxpayer TIN | 9 or 12 digits | `companies.tin` (the reporting company) |
| 2 | Transaction Type | `S` = Sale, `P` = Purchase | Derived from `source_document_type` |
| 3 | Counterparty TIN | 9 or 12 digits | Customer TIN (Sales) or Supplier TIN (Purchases) |
| 4 | Counterparty Registered Name | Text, max 100 chars | Snapshot from customer/supplier master |
| 5 | Counterparty Address | Text, max 200 chars | Snapshot from customer/supplier master |
| 6 | Document Number | SI/OR/PI number | `sales_invoices.si_number` or `purchase_invoices.supplier_invoice_number` |
| 7 | Document Date | MMDDYYYY | Transaction date |
| 8 | Gross Amount | Numeric, 2 decimal places | Net amount exclusive of VAT |
| 9 | Exempt Amount | Numeric | Portion that is VAT-exempt |
| 10 | Zero-Rated Amount | Numeric | Portion that is zero-rated |
| 11 | Taxable Amount | Numeric | Portion subject to 12% VAT |
| 12 | Output / Input VAT | Numeric | VAT amount (Output for Sales, Input for Purchases) |
| 13 | Classification | `AT`=Taxable, `AZ`=Zero-Rated, `AE`=Exempt | Derived from `vat_classification` |

## Supabase Database Architecture

### Table 1: `compliance_relief_export_headers`

**Critical Database Rules:**
1. `UNIQUE(company_id, period_year, period_quarter)` â€” one RELIEF header per quarter per company.
2. Filed headers are immutable.

| Column Name | Data Type | Rules | What it stores |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System ID. |
| `company_id` | UUID | Required, Foreign Key | Links to `companies.id`. |
| `period_year` | Integer | Required | The year covered. |
| `period_quarter` | Integer | Required, CHECK (1â€“4) | The quarter. |
| `total_sales_count` | Integer | Required, Default 0 | Number of sales line items. |
| `total_purchase_count` | Integer | Required, Default 0 | Number of purchase line items. |
| `total_gross_sales` | Numeric(15,2) | Required, Default 0 | Sum of all sales `gross_amount`. |
| `total_gross_purchases` | Numeric(15,2) | Required, Default 0 | Sum of all purchase `gross_amount`. |
| `total_output_vat` | Numeric(15,2) | Required, Default 0 | Total Output VAT on sales. |
| `total_input_vat` | Numeric(15,2) | Required, Default 0 | Total Input VAT on purchases. |
| `bir_acknowledgement_no` | Text | Nullable | BIR acknowledgement upon submission. |
| `date_filed` | Date | Nullable | Actual submission date. |
| `status` | Text | Required, Default `draft` | ENUM: `draft`, `final`, `filed`. |
| `created_by` | UUID | Required, Foreign Key | Links to `users.id`. |
| `updated_by` | UUID | Nullable, Foreign Key | Links to `users.id`. |
| `created_at` | Timestamp | Auto | Creation timestamp. |
| `updated_at` | Timestamp | Auto | Last update timestamp. |

**Composite Unique Constraint:** `UNIQUE(company_id, period_year, period_quarter)`

### Table 2: `compliance_relief_export_lines`
One row per invoice. Every field maps directly to a BIR RELIEF DAT file column.

| Column Name | Data Type | Rules | What it stores |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System ID. |
| `company_id` | UUID | Required, Foreign Key | Links to `companies.id`. Denormalized for RLS. |
| `header_id` | UUID | Required, Foreign Key | Links to `compliance_relief_export_headers.id`. |
| `source_document_type` | Text | Required | ENUM: `sales_invoice`, `purchase_invoice`. Drives `transaction_type` DAT column. |
| `source_document_id` | UUID | Required | FK to the originating document. |
| `transaction_type` | Text | Required | DAT column 2: `S` for Sales, `P` for Purchases. Derived from `source_document_type`. |
| `document_number` | Text | Required | DAT column 6: SI number (sales) or supplier invoice number (purchases). |
| `document_date` | Date | Required | DAT column 7. |
| `counterparty_tin` | Text | Required | DAT column 3. Customer or Supplier TIN. BIR requires â€” cannot be null. |
| `counterparty_name` | Text | Required | DAT column 4. Registered name snapshot. Max 100 chars. |
| `counterparty_address` | Text | Required | DAT column 5. Registered address snapshot. Max 200 chars. |
| `gross_amount` | Numeric(15,2) | Required | DAT column 8. Total invoice amount exclusive of VAT. |
| `exempt_amount` | Numeric(15,2) | Required, Default 0 | DAT column 9. VAT-exempt portion. |
| `zero_rated_amount` | Numeric(15,2) | Required, Default 0 | DAT column 10. Zero-rated portion. |
| `taxable_amount` | Numeric(15,2) | Required, Default 0 | DAT column 11. Regular 12% VAT portion. |
| `vat_amount` | Numeric(15,2) | Required, Default 0 | DAT column 12. Output or Input VAT amount. |
| `vat_classification` | Text | Required | ENUM: `regular`, `zero_rated`, `exempt`. |
| `dat_classification_code` | Text | Required | DAT column 13: `AT`=Taxable, `AZ`=Zero-Rated, `AE`=Exempt. |
| `created_at` | Timestamp | Auto | Creation timestamp. |

## Row-Level Security

```sql
-- Filed headers are immutable
CREATE POLICY "block_update_on_filed_relief"
ON compliance_relief_export_headers
FOR UPDATE
USING (status IN ('draft', 'final'));

-- No deletes
CREATE POLICY "block_delete_relief"
ON compliance_relief_export_headers
FOR DELETE
USING (false);

-- Tenant isolation
CREATE POLICY "tenant_isolation_relief"
ON compliance_relief_export_headers
FOR ALL
USING (company_id = (auth.jwt() ->> 'company_id')::UUID);
```
