# PXL ERP Blueprint: Check Vouchers (AP Payment Module)

## Module Overview
The Check Voucher (CV) module executes supplier payments from bank accounts against outstanding Accounts Payable (Purchase Invoices). It is the last step in the AP Payment Cycle: the supplier delivers goods → the company receives and validates the invoice → the AP liability is created → the Check Voucher releases cash and extinguishes the liability.

**Philippine Compliance Critical:** Every AP payment to a supplier who rendered services subject to EWT must have the corresponding 2307 (Certificate of Creditable Tax Withheld at Source) generated and deducted from the payment. The CV must compute: `Net Payment = Gross Amount - Expanded Withholding Tax (EWT)`.

**GL Impact on Posting:**
```
DR  Accounts Payable (per vendor)       [Gross Invoice Amount]
CR  EWT Payable (per ATC code)          [EWT Amount]
CR  Bank Account (disbursing bank)      [Net Payment = Gross - EWT]
```

---

## AP Payment Cycle

1. **Create CV** — Select the bank account, vendor, and the AP invoices being settled.
2. **System computes** — Gross amount, EWT rate and amount (from vendor's EWT profile), and net cash payment.
3. **Submit for approval** — Routed via `approval_workflows` with `module_type = 'payment'`.
4. **Approval granted** — CV moves to `approved` status.
5. **Post CV** — Posts the GL entry. Bank balance decreases. AP liability is extinguished.
6. **Generate 2307** — System creates a `form_2307_certificates` record for the EWT deducted.
7. **Print and release check** — Treasury prints the check and delivers it to the vendor.

## Supabase Database Architecture

### Table 1: `check_vouchers`

**Critical Database Rules:**
1. `UNIQUE(company_id, cv_number)` — sequential numbering per tenant.
2. Once `status = 'posted'`, no modifications. RLS blocks UPDATE.
3. `journal_entry_id` is set when the CV is posted. Before posting, it is null.
4. Each CV is linked to the disbursing bank via `bank_account_id`.

## GL Posting Specification

When a Check Voucher is posted, the system creates ONE journal entry with the following structure:

```
FOR EACH LINE:
  DR  Accounts Payable [ap_account_id]    payment_amount     (per invoice line)
  CR  EWT Payable [ewt_payable_account_id] ewt_amount        (per ATC code, if ewt_rate > 0)

HEADER LEVEL:
  CR  Bank Account [bank_gl_account_id]   net_payment        (total net cash disbursed)

source_document_type = 'check_voucher'
source_document_id   = check_vouchers.id
```

**Example — Paying a ₱100,000 professional services invoice with 5% EWT:**
```
DR  Accounts Payable (Trade)         100,000.00
CR  EWT Payable — Professional Fees    5,000.00  (ATC: WC158)
CR  BPI Checking Account              95,000.00
```

---

## 2307 Certificate Auto-Generation

On posting, for each line where `ewt_rate > 0`, the system automatically creates one `form_2307_certificates` record:
- `supplier_id` → from CV header
- `payee_tin` → from CV header snapshot
- `gross_income` → `payment_amount`
- `ewt_rate` → from CV line
- `ewt_amount` → from CV line
- `atc_code` → from CV line
- `source_document_type` = `check_voucher`
- `source_document_id` = CV line ID
- `status` = `pending` (pending physical issuance)

## Row-Level Security

```sql
-- Block UPDATE on posted CVs
CREATE POLICY "block_update_on_posted_cv"
ON check_vouchers
FOR UPDATE
USING (status NOT IN ('posted', 'cancelled'));

-- No deletes
CREATE POLICY "block_delete_cv"
ON check_vouchers
FOR DELETE
USING (false);

-- Tenant isolation
CREATE POLICY "tenant_isolation_cv"
ON check_vouchers
FOR ALL
USING (company_id = (auth.jwt() ->> 'company_id')::UUID);
```
