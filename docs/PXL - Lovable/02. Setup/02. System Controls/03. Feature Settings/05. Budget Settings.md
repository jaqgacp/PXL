# PXL ERP Blueprint: Feature Settings — Budget Settings

## Module Overview
The Budget Settings page defines the global rules that govern how the Budgeting module operates across the company. These settings determine how budgets are structured (by department, branch, or cost center), how strictly the system enforces budget limits on transactions, and how budget vs. actual performance is monitored and reported. The core decision in this module is the **Budget Control Mode** — whether the system acts as a hard financial gatekeeper or a soft advisory tool.

**Philippine Business Context:** Budget control is a critical internal management tool for Philippine SMEs and corporations. Many businesses operate with annual budgets approved by the Board of Directors or owners, and department heads are held accountable to their budget allocations. Depending on the company's governance policy, the system can either **strictly prevent** spending beyond the approved budget (protecting financial discipline) or **allow spending with a warning** (preserving operational flexibility while maintaining visibility).

## What You Will See on the Screen (The Dashboard)
When you open the "Budget Settings" page, you will see a single, organized settings form — not a list. This page has one configuration record per company.

### The Action Bar (Top of the page):
* **Company Selector:** A dropdown to select which company's budget settings you are configuring.
* **Save Settings Button:** A prominent button to save all changes on the page.
* **Reset to Defaults Button:** A secondary button to revert all settings to the system defaults.

## What Information We Need to Capture (The Data Fields)
The settings form is divided into logical sections.

### Section 1: Budget Control Mode
| Field Name | Description | Required? |
| :--- | :--- | :--- |
| Company | Dropdown linked to active companies. These settings apply to the selected entity. | Yes |
| Budget Control Mode | Dropdown with two options: **Strict (Hard Block)** or **Advisory (Soft Warning)**. This is the most critical setting in this module — see the detailed description below. | Yes |

**Budget Control Mode — Detailed Explanation:**

* **Strict (Hard Block):** When a Purchase Order (PO), Purchase Request (PR), or Vendor Bill is created and the total amount would cause the department's or cost center's budget to be exceeded, the system **blocks the transaction entirely**. The user cannot save or confirm the document. A clear error message is shown: *"This transaction exceeds the approved budget for [Department/Cost Center]. The remaining budget balance is PHP [amount]. Please obtain a budget amendment before proceeding."* This mode is recommended for companies with strict financial governance.

* **Advisory (Soft Warning):** When a transaction would exceed the budget, the system **displays a prominent warning banner** but **allows the user to proceed** and save the document. The system logs the over-budget transaction for management review. Budget consumption reports will highlight all over-budget line items. This mode is recommended for companies that prioritize operational flexibility over hard controls, or where budget amendments are handled informally.

### Section 2: Budget Structure
| Field Name | Description | Required? |
| :--- | :--- | :--- |
| Budget Period | Dropdown: **Annual** (one budget for the full fiscal year — most common), Quarterly (separate budget per quarter), Monthly (separate budget per month). | Yes |
| Budget Dimension | Multi-select: **Department** (track budget by organizational department), **Branch** (track budget by physical location), **Cost Center** (track budget by cost center code), **GL Account** (track budget at the account level). At least one dimension must be selected. | Yes |
| Allow Budget Reallocation | Toggle (Yes / No). If YES, users with the appropriate permission can move budget from one department/cost center to another within the same period, without needing a formal budget amendment. If NO, all reallocations must go through the Approval Matrix. | Yes |
| Require Budget Amendment Approval | Toggle (Yes / No). If YES, any increase to an approved budget line item requires an approval before it becomes effective. | Yes |

### Section 3: Monitoring & Alerts
| Field Name | Description | Required? |
| :--- | :--- | :--- |
| Enable Budget Utilization Alerts | Toggle (Yes / No). If YES, the system sends automated alerts to the department head or budget owner when budget consumption reaches the defined thresholds below. | Yes |
| Alert at 75% Utilization | Toggle (Yes / No). Sends a first warning alert when 75% of the budget for a department or cost center has been consumed. | Conditional |
| Alert at 90% Utilization | Toggle (Yes / No). Sends a critical warning alert when 90% of the budget has been consumed. | Conditional |
| Alert at 100% Utilization | Toggle (Yes / No). Sends a final alert when the budget is fully consumed. In Advisory mode, this is an informational alert. In Strict mode, the hard block will already be in effect before this point is reached. | Conditional |
| Include Committed Amounts in Budget Check | Toggle (Yes / No). If YES, the budget check includes the amounts on **confirmed but unpaid Purchase Orders** (committed spending) in addition to actual posted expenses. This gives a more accurate picture of remaining available budget and is the recommended setting. | Yes |

### Section 4: GL Integration
| Field Name | Description | Required? |
| :--- | :--- | :--- |
| Budget vs. Actual Report Basis | Dropdown: **Accrual Basis** (compares budget against posted expense journal entries), **Cash Basis** (compares budget against actual cash payments made). Accrual Basis is aligned with PFRS and recommended for most companies. | Yes |
| Default Budget Variance Account | Dropdown linked to the Chart of Accounts. Used in management reports to highlight over/under-budget variances. This is a reporting account and does not affect GL postings. | Optional |

## The Supabase Database Architecture (For Lovable)
This module requires one settings table and one main budget allocation table.

### Table 1: `budget_settings` (The Configuration Table)
**Critical Database Rule for Lovable:** Apply a UNIQUE constraint on `company_id` to ensure only one settings record exists per company.

| Column Name | Data Type | Rules | What it stores |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System ID. |
| `company_id` | UUID | Required, Foreign Key, Unique | Links to `companies.id`. |
| `budget_control_mode` | Text | Required, Default: `advisory` | `strict` or `advisory`. |
| `budget_period` | Text | Required, Default: `annual` | `annual`, `quarterly`, `monthly`. |
| `budget_by_department` | Boolean | Default: `true` | Whether budgets are tracked by department. |
| `budget_by_branch` | Boolean | Default: `false` | Whether budgets are tracked by branch. |
| `budget_by_cost_center` | Boolean | Default: `false` | Whether budgets are tracked by cost center. |
| `budget_by_gl_account` | Boolean | Default: `true` | Whether budgets are tracked at the GL account level. |
| `allow_budget_reallocation` | Boolean | Default: `false` | Whether users can reallocate budget informally. |
| `require_amendment_approval` | Boolean | Default: `true` | Whether budget amendments need approval. |
| `enable_utilization_alerts` | Boolean | Default: `true` | Whether budget utilization alerts are active. |
| `alert_at_75_pct` | Boolean | Default: `true` | Whether to send an alert at 75% consumption. |
| `alert_at_90_pct` | Boolean | Default: `true` | Whether to send an alert at 90% consumption. |
| `alert_at_100_pct` | Boolean | Default: `true` | Whether to send an alert at 100% consumption. |
| `include_committed_amounts` | Boolean | Default: `true` | Whether confirmed POs are included in the budget check. |
| `report_basis` | Text | Required, Default: `accrual` | `accrual` or `cash`. |
| `default_variance_gl_id` | UUID | Nullable, Foreign Key | Links to `gl_accounts.id`. Budget variance reporting account. |
| `created_by` | UUID | Required, Foreign Key | Links to `users.id`. User who created this record. |
| `updated_by` | UUID | Nullable, Foreign Key | Links to `users.id`. User who last edited this record. |
| `created_at` | Timestamp | Auto | Date record was created. |
| `updated_at` | Timestamp | Auto | Date record was last edited. |

### Table 2: `budget_allocations` (The Budget Data Table)
This table stores the actual budget amounts per dimension (department, branch, cost center) per GL account per period. This is used for comparison against actual expenses when generating Budget vs. Actual reports.

| Column Name | Data Type | Rules | What it stores |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System ID for the budget line. |
| `company_id` | UUID | Required, Foreign Key | Links to `companies.id`. |
| `fiscal_year` | Integer | Required | The fiscal year this budget applies to (e.g., 2026). |
| `period_number` | Integer | Nullable | Quarter or month number if budget period is not annual (e.g., 1–4 for quarterly, 1–12 for monthly). |
| `gl_account_id` | UUID | Required, Foreign Key | Links to `gl_accounts.id`. The specific expense account being budgeted. |
| `department_id` | UUID | Nullable, Foreign Key | Links to `departments.id`. |
| `branch_id` | UUID | Nullable, Foreign Key | Links to `branches.id`. |
| `cost_center_id` | UUID | Nullable, Foreign Key | Links to `cost_centers.id`. |
| `budgeted_amount` | Numeric(18,2) | Required | The approved budget amount for this line. |
| `is_approved` | Boolean | Default: `false` | Whether this budget line has been formally approved. |
| `approved_by` | UUID | Nullable, Foreign Key | Links to the `users` table. Who approved this budget line. |
| `approved_at` | Timestamp | Nullable | When the budget was approved. |
| `created_by` | UUID | Required, Foreign Key | Links to `users.id`. User who created this record. |
| `updated_by` | UUID | Nullable, Foreign Key | Links to `users.id`. User who last edited this record. |
| `created_at` | Timestamp | Auto | Date record was created. |
| `updated_at` | Timestamp | Auto | Date record was last edited. |
| `updated_at` | Timestamp | Auto | Date record was last edited. |
