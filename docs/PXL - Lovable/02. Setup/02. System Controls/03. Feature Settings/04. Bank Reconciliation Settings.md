# PXL ERP Blueprint: Feature Settings — Bank Reconciliation Settings

## Module Overview
The Bank Reconciliation Settings page defines the global rules that govern how the Bank Reconciliation module operates for each company. Bank reconciliation is the process of matching the company's book records (GL entries) against the bank's official statement to identify outstanding items, bank errors, and unrecorded transactions. These settings control auto-matching behavior, tolerance limits for variance handling, and the workflow controls required to lock a completed reconciliation.

**Philippine Practice Note:** In the Philippines, bank reconciliation is prepared monthly and is a standard requirement during BIR tax mapping (tax audit) and external audits. The reconciliation output must clearly show: outstanding checks, deposits in transit, bank charges not yet recorded in the books, and book errors. PXL ERP's reconciliation module is designed to produce this standard Philippine bank reconciliation format automatically.

## What You Will See on the Screen (The Dashboard)
When you open the "Bank Reconciliation Settings" page, you will see a single, organized settings form — not a list. This page has one configuration record per company.

### The Action Bar (Top of the page):
* **Company Selector:** A dropdown to select which company's bank reconciliation settings you are configuring.
* **Save Settings Button:** A prominent button to save all changes on the page.
* **Reset to Defaults Button:** A secondary button to revert all settings to the system defaults.

## What Information We Need to Capture (The Data Fields)
The settings form is divided into logical sections.

### Section 1: Auto-Matching Configuration
| Field Name | Description | Required? |
| :--- | :--- | :--- |
| Company | Dropdown linked to active companies. These settings apply to the selected entity. | Yes |
| Enable Auto-Matching | Toggle (Yes / No). If YES, the system will automatically attempt to match imported bank statement lines against GL entries based on the matching criteria below. Unmatched items are flagged for manual review. | Yes |
| Auto-Match By | Multi-select checkboxes (visible if Auto-Matching = YES): **Amount** (exact peso match), **Reference Number** (check number or bank reference), **Date** (transaction date). All selected criteria must match for auto-matching to succeed. | Conditional |
| Matching Date Tolerance (Days) | The number of days before or after the bank statement date within which the system will search for a matching GL entry. For example, a tolerance of 3 days means a payment recorded in the books on March 28 can be matched to a bank entry dated March 30. Useful for handling check clearing delays. | Yes |
| Amount Tolerance (PHP) | The maximum peso variance between a bank statement line and a GL entry that the system will still consider a match. For example, a tolerance of PHP 1.00 handles minor rounding differences between the bank and the books. Recommended default: **PHP 0.00** (exact match). | Yes |
| Auto-Match Confidence Threshold (%) | The minimum matching confidence score (based on how many criteria matched) for the system to auto-confirm a match without requiring user review. Matches below this threshold are flagged as "Suggested" and require manual confirmation. | Yes |

### Section 2: Reconciliation Period Controls
| Field Name | Description | Required? |
| :--- | :--- | :--- |
| Reconciliation Period | Dropdown: Monthly (Standard and Recommended), Weekly. Determines the period for which each bank reconciliation statement is prepared. | Yes |
| Allow Backdated Reconciliation | Toggle (Yes / No). If YES, users can open and modify reconciliation for a prior period. If NO, only the current open reconciliation period can be edited. Recommended: **NO** for audit integrity. | Yes |
| Require Approval to Lock Reconciliation | Toggle (Yes / No). If YES, a completed bank reconciliation must be reviewed and approved by an authorized user before it is locked and the period is closed. | Yes |
| Lock Reconciliation After Approval | Toggle (Yes / No). If YES, once a bank reconciliation is approved, it is permanently locked — no further changes are allowed, and the matched items cannot be unmatched. | Yes |

### Section 3: Outstanding Item Controls
| Field Name | Description | Required? |
| :--- | :--- | :--- |
| Outstanding Check Age Alert (Days) | The number of days after which an uncleared check is flagged as "Stale" in the Outstanding Checks report. In the Philippines, checks are typically stale after 180 days. | Yes |
| Stale Check GL Account | Dropdown linked to the Chart of Accounts. The account to which stale check liabilities are reclassified when a check is formally written off (e.g., "2100-099 Other Payables - Stale Checks"). | Yes |
| Deposit in Transit Age Alert (Days) | The number of days after which an uncleared deposit is flagged for investigation in the reconciliation report. | Yes |

### Section 4: Bank Feed & Statement Import
| Field Name | Description | Required? |
| :--- | :--- | :--- |
| Supported Import Format | Dropdown: CSV, Excel (.xlsx), OFX, MT940. Determines which file format the system accepts for bank statement uploads. | Yes |
| Default Date Format in Import File | Dropdown: MM/DD/YYYY, DD/MM/YYYY, YYYY-MM-DD. Must match the date format used by the company's bank in its exported statement files. | Yes |

## The Supabase Database Architecture (For Lovable)
This module requires one settings table. It stores one record per company.

### Table 1: `bank_reconciliation_settings` (The Main Table)
**Critical Database Rule for Lovable:** Apply a UNIQUE constraint on `company_id` to ensure only one settings record exists per company.

**Application Logic for Lovable:** When the auto-matching engine runs, it must apply the `matching_date_tolerance_days` and `amount_tolerance_php` values as parameters. The matching confidence score should be computed as a percentage of matched criteria (e.g., if 2 out of 3 criteria match, confidence = 67%). Only matches at or above `auto_match_confidence_threshold_pct` should be auto-confirmed; all others should be presented as "Suggested Matches" for user review.

| Column Name | Data Type | Rules | What it stores |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System ID. |
| `company_id` | UUID | Required, Foreign Key, Unique | Links to `companies.id`. |
| `enable_auto_matching` | Boolean | Default: `true` | Whether auto-matching is active. |
| `auto_match_by_amount` | Boolean | Default: `true` | Whether amount is used as a matching criterion. |
| `auto_match_by_reference` | Boolean | Default: `true` | Whether reference/check number is used as a criterion. |
| `auto_match_by_date` | Boolean | Default: `true` | Whether date is used as a matching criterion. |
| `matching_date_tolerance_days` | Integer | Required, Default: `3` | Days before/after bank date to search for GL match. |
| `amount_tolerance_php` | Numeric(18,2) | Required, Default: `0.00` | Maximum peso variance for an auto-match. |
| `auto_match_confidence_threshold_pct` | Numeric(5,2) | Required, Default: `100.00` | Minimum confidence % to auto-confirm a match. |
| `reconciliation_period` | Text | Required, Default: `monthly` | `monthly` or `weekly`. |
| `allow_backdated_reconciliation` | Boolean | Default: `false` | Whether prior period reconciliations can be edited. |
| `require_approval_to_lock` | Boolean | Default: `true` | Whether approval is needed to lock a reconciliation. |
| `lock_after_approval` | Boolean | Default: `true` | Whether locking is permanent after approval. |
| `stale_check_age_days` | Integer | Required, Default: `180` | Days after which an uncleared check is flagged stale. |
| `stale_check_gl_id` | UUID | Nullable, Foreign Key | Links to `gl_accounts.id`. Stale check reclassification account. |
| `deposit_in_transit_age_days` | Integer | Required, Default: `30` | Days after which an uncleared deposit is flagged. |
| `import_format` | Text | Required, Default: `csv` | `csv`, `xlsx`, `ofx`, `mt940`. |
| `import_date_format` | Text | Required, Default: `MM/DD/YYYY` | Date format used in the bank statement import file. |
| `created_by` | UUID | Required, Foreign Key | Links to `users.id`. User who created this record. |
| `updated_by` | UUID | Nullable, Foreign Key | Links to `users.id`. User who last edited this record. |
| `created_at` | Timestamp | Auto | Date record was created. |
| `updated_at` | Timestamp | Auto | Date record was last edited. |
| `updated_at` | Timestamp | Auto | Date record was last edited. |
