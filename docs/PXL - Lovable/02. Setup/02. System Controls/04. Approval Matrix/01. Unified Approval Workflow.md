# PXL ERP Blueprint: Unified Approval Workflow

## Module Overview
The Unified Approval Workflow is the single system-wide mechanism by which the PXL ERP enforces segregation-of-duties and authorization controls. Instead of maintaining four separate approval tables (one per module), the system uses ONE `approval_workflows` table with a `module_type` discriminator column. This eliminates code duplication, enables cross-module approval reporting, and means that every new module automatically inherits the approval infrastructure without requiring a new database table.

**All approval workflow behavior — trigger conditions, step sequencing, approver assignment, and approval instance tracking — is consolidated here.**

---

## Modules Using This System

| Module Type | Document Examples | When Approval Triggers |
| :--- | :--- | :--- |
| `sales` | Quotations, Sales Orders, Credit Notes | Amount > threshold, Discount > threshold |
| `purchasing` | Purchase Requisitions, Purchase Orders | Amount > threshold, Vendor not in approved list |
| `payment` | Check Vouchers, Petty Cash Vouchers, Fund Transfers | Amount > threshold |
| `journal` | Manual Journal Entries | Any manual JE, or JEs above amount threshold |
| `master_data` | Customer Credit Limit changes, Vendor edits | Any modification to master data records |
| `asset` | Asset Disposals, Impairments, Transfers | Any disposal or impairment event |
| `credit_memo` | Customer Credit Memos, Debit Memos | Amount > threshold |

## Data Fields

| :--- | :--- | :--- |

| Field Name | Description | Required? |
| :--- | :--- | :--- |

## Supabase Database Architecture

### Table 1: `approval_workflows`

**Critical Database Rules:**
1. `UNIQUE(company_id, module_type, document_type, trigger_condition_type, threshold_value)` — prevents creating two identical workflows for the same module, document, and trigger.
2. Only one active workflow can apply to a given `module_type` + `document_type` combination at any time. The system checks `is_active = true` when applying workflows.
3. This single table replaces: `sales_approval_workflows`, `purchasing_approval_workflows`, `payment_approval_workflows`, `journal_approval_workflows`, and `master_data_approval_workflows`. Those tables must NOT be created.

| Column Name | Data Type | Rules | What it stores |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System ID. |
| `company_id` | UUID | Required, Foreign Key | Links to `companies.id`. |
| `workflow_name` | Text | Required | Descriptive name of the rule. |
| `module_type` | Text | Required | ENUM: `sales`, `purchasing`, `payment`, `journal`, `master_data`, `asset`, `credit_memo`. |
| `document_type` | Text | Required | The specific document within the module (e.g., `purchase_order`, `sales_order`, `check_voucher`, `journal_entry`, `asset_disposal`). |
| `trigger_condition_type` | Text | Required | ENUM: `always`, `amount_exceeds`, `discount_pct_exceeds`, `credit_limit_exceeded`. |
| `threshold_value` | Numeric(15,2) | Nullable | Required when `trigger_condition_type = 'amount_exceeds'` or `'discount_pct_exceeds'`. |
| `is_active` | Boolean | Required, Default: `true` | Whether this workflow is currently enforced. |
| `created_by` | UUID | Required, Foreign Key | Links to `users.id`. |
| `updated_by` | UUID | Nullable, Foreign Key | Links to `users.id`. |
| `created_at` | Timestamp | Auto | Creation timestamp. |
| `updated_at` | Timestamp | Auto | Last update timestamp. |

**Composite Unique Constraint:** `UNIQUE(company_id, module_type, document_type, trigger_condition_type, threshold_value)`

---

### Table 2: `approval_workflow_steps`
Defines who must approve, in what order, within a given workflow.

| Column Name | Data Type | Rules | What it stores |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System ID. |
| `company_id` | UUID | Required, Foreign Key | Denormalized for RLS. |
| `workflow_id` | UUID | Required, Foreign Key | Links to `approval_workflows.id`. |
| `step_sequence` | Integer | Required | Numeric order: 1, 2, 3... |
| `approver_type` | Text | Required | ENUM: `user`, `role`, `dept_head`. |
| `approver_user_id` | UUID | Nullable, Foreign Key | Links to `users.id` when `approver_type = 'user'`. |
| `approver_role_id` | UUID | Nullable, Foreign Key | Links to `roles.id` when `approver_type = 'role'`. |
| `action_required` | Text | Required | ENUM: `approve` (blocking), `review` (informational). |
| `escalation_hours` | Integer | Nullable | Hours before auto-escalation or reminder. |
| `created_at` | Timestamp | Auto | Creation timestamp. |

**Unique Constraint:** `UNIQUE(workflow_id, step_sequence)` — no two steps can share the same sequence in the same workflow.

---

### Table 3: `approval_instances`
This is the runtime table. Every time a document is submitted for approval, one row is created per approval step. This is what powers the "Pending Approvals" inbox for each approver.

**Critical Database Rules:**
1. `approval_instances` is INSERT-ONLY for the workflow engine. Once a row is created, its `status` is the only field that can be updated (by the approver action).
2. This table is the source of truth for WHERE a document is in its approval lifecycle.

| Column Name | Data Type | Rules | What it stores |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System ID. |
| `company_id` | UUID | Required, Foreign Key | Links to `companies.id`. Denormalized for RLS. |
| `workflow_id` | UUID | Required, Foreign Key | Links to `approval_workflows.id`. |
| `workflow_step_id` | UUID | Required, Foreign Key | Links to `approval_workflow_steps.id`. |
| `source_document_type` | Text | Required | The module-type of the document being approved (e.g., `purchase_order`, `check_voucher`). |
| `source_document_id` | UUID | Required | The UUID of the specific document awaiting approval. |
| `source_document_no` | Text | Required | Human-readable document number (snapshot). |
| `source_document_amount` | Numeric(15,2) | Nullable | Amount snapshot at the time of submission (for audit trail). |
| `step_sequence` | Integer | Required | Snapshot of the step number. |
| `required_approver_type` | Text | Required | Snapshot of `approver_type`. |
| `required_approver_id` | UUID | Nullable | Snapshot of the user or role required to approve. |
| `actual_approver_id` | UUID | Nullable, Foreign Key | Links to `users.id` who actually acted. |
| `status` | Text | Required, Default `pending` | ENUM: `pending`, `approved`, `rejected`, `escalated`, `bypassed`. |
| `remarks` | Text | Nullable | Approver's comment or rejection reason. |
| `acted_at` | Timestamp | Nullable | When the approver took action. |
| `submitted_at` | Timestamp | Required | When the document was submitted for this step. |
| `escalated_at` | Timestamp | Nullable | When the system auto-escalated this step. |
| `created_by` | UUID | Required, Foreign Key | System user who created the approval instance. |
| `created_at` | Timestamp | Auto | Creation timestamp. |

**Indexes:**
```sql
-- Fast lookup for "My Pending Approvals" inbox
CREATE INDEX idx_approval_instances_approver
  ON approval_instances (company_id, required_approver_id, status)
  WHERE status = 'pending';

-- Fast lookup for "Where is document X in its approval chain?"
CREATE INDEX idx_approval_instances_document
  ON approval_instances (source_document_type, source_document_id);

CREATE INDEX idx_approval_instances_company_status
  ON approval_instances (company_id, status, submitted_at DESC);
```

## Row-Level Security

```sql
-- Approver can only see their own pending instances or the steps they already acted on
CREATE POLICY "approver_can_view_own_instances"
ON approval_instances
FOR SELECT
USING (
  company_id = (auth.jwt() ->> 'company_id')::UUID
  AND (
    required_approver_id = (auth.jwt() ->> 'sub')::UUID
    OR actual_approver_id = (auth.jwt() ->> 'sub')::UUID
    -- Or the user is the original document owner (checked via source_document_id)
  )
);

-- Only the workflow engine (SECURITY DEFINER function) can INSERT approval instances
CREATE POLICY "workflow_engine_insert_only"
ON approval_instances
FOR INSERT
WITH CHECK (false); -- Blocked for all regular users; engine uses SECURITY DEFINER

-- Approvers can UPDATE status on their pending steps only
CREATE POLICY "approver_can_update_own_pending"
ON approval_instances
FOR UPDATE
USING (
  company_id = (auth.jwt() ->> 'company_id')::UUID
  AND required_approver_id = (auth.jwt() ->> 'sub')::UUID
  AND status = 'pending'
);

-- No deletes — approval history is permanent
CREATE POLICY "block_delete_approval_instances"
ON approval_instances
FOR DELETE
USING (false);

-- Tenant isolation on workflow definitions
CREATE POLICY "tenant_isolation_workflows"
ON approval_workflows
FOR ALL
USING (company_id = (auth.jwt() ->> 'company_id')::UUID);
```
