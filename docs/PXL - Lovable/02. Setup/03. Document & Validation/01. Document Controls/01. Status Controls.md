# PXL ERP Blueprint: Status Controls

## Module Overview
Status Controls govern the lifecycle states of business documents (e.g., Draft, Pending Approval, Approved, Posted, Closed). In Philippine ERP setups, controlling document statuses is crucial for audit trails and ensuring that only verified documents hit the General Ledger. This module allows system administrators to define which user roles can transition a document from one state to another.

## What You Will See on the Screen (The Dashboard)
The dashboard displays the allowable status transitions per document type.

### The Action Bar (Top of the page):
* **Search Bar:** Search by Document Type or Status.
* **Filter:** Filter by Source Status or Target Status.
* **Export Button:** Download transitions to CSV.
* **Create Transition Rule Button:** Define a new status transition rule.

### The Data List (The main table):
* Document Type
* Current Status
* Allowed Next Status
* Permitted Roles
* Requires Reason? (Yes/No)

---

## What Information We Need to Capture (The Data Fields)

### Section 1: Transition Rules (Header-less configuration list)
| Field Name | Description | Required? |
| :--- | :--- | :--- |
| Document Type | E.g., Sales Order, Payment Voucher. | Yes |
| Source Status | The current state (e.g., Draft). | Yes |
| Target Status | The allowed next state (e.g., Pending Approval). | Yes |
| Permitted Roles | Multi-select: Which roles can execute this change. | Yes |
| Requires Remarks | Must the user enter a reason for this transition? | Yes |

## The Supabase Database Architecture (For Lovable)

**Critical Database Rule for Lovable:** Ensure appropriate unique constraints are applied to prevent duplicate entries.

### Table 1: `document_status_transitions` (Header)
| Column Name | Data Type | Rules | What it stores |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System ID. |
| `document_type` | Text | Required | E.g., `sales_order`. |
| `source_status` | Text | Required | E.g., `draft`. |
| `target_status` | Text | Required | E.g., `approved`. |
| `requires_remarks`| Boolean | Default: `false` | Must user provide a reason? |
| `created_by` | UUID | Required, Foreign Key | Links to `users.id`. |
| `updated_by` | UUID | Nullable, Foreign Key | Links to `users.id`. |
| `created_at` | Timestamp | Auto | Creation timestamp. |
| `updated_at` | Timestamp | Auto | Update timestamp. |

### Table 2: `document_status_roles` (Line Items)
| Column Name | Data Type | Rules | What it stores |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System ID. |
| `transition_id` | UUID | Required, Foreign Key | Links to `document_status_transitions.id`. |
| `role_id` | UUID | Required, Foreign Key | Links to `roles.id`. |
| `created_by` | UUID | Required, Foreign Key | Links to `users.id`. |
| `updated_by` | UUID | Nullable, Foreign Key | Links to `users.id`. |
| `created_at` | Timestamp | Auto | Record creation date. |
| `updated_at` | Timestamp | Auto | Record edit date. |
