# PXL ERP Blueprint: Void Controls

## Module Overview
Void Controls manage the cancellation of non-posted documents. In local auditing, it's important to track why a physical form (like a pre-printed check or SI) was voided before it ever affected the GL. This module enforces reason codes and tracking for voided pre-numbered documents.

## What You Will See on the Screen (The Dashboard)
The interface allows configuration of voiding rules and reason codes.

### The Action Bar (Top of the page):
* **Search Bar:** Search by Reason Code.
* **Filter:** Active/Inactive.
* **Export Button:** Export list.
* **Add Reason Code Button:** Add a standard reason for voiding.

### The Data List (The main table):
* Reason Code
* Description
* Requires Manager Approval (Yes/No)
* Status

---

## What Information We Need to Capture (The Data Fields)

### Section 1: Void Reasons Setup
| Field Name | Description | Required? |
| :--- | :--- | :--- |
| Reason Code | Short code (e.g., ERR_ENTRY). | Yes |
| Description | Full text (e.g., Data Entry Error). | Yes |
| Requires Approval | Must a manager sign off to void? | Yes |

## The Supabase Database Architecture (For Lovable)

**Critical Database Rule for Lovable:** Ensure appropriate unique constraints are applied to prevent duplicate entries.

### Table 1: `void_reason_codes`
| Column Name | Data Type | Rules | What it stores |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System ID. |
| `code` | Text | Required, Unique | Short identifier. |
| `description` | Text | Required | Full explanation. |
| `requires_approval` | Boolean | Default: `false` | If supervisor check is needed. |
| `is_active` | Boolean | Default: `true` | Status toggle. |
| `created_by` | UUID | Required, Foreign Key | Links to `users.id`. |
| `updated_by` | UUID | Nullable, Foreign Key | Links to `users.id`. |
| `created_at` | Timestamp | Auto | Creation date. |
| `updated_at` | Timestamp | Auto | Update date. |
