# PXL ERP Blueprint: Department Setup

## Module Overview
The Department Setup allows you to define the organizational departments within your company. Departments are used throughout the system for cost tracking, budgeting, approval routing, and segmented financial reporting. Each department can be scoped to a specific company and branch, enabling detailed breakdowns of income and expenses by organizational unit. Department codes are referenced in journal entries, purchase orders, and approval workflows across all modules.

## What You Will See on the Screen (The Dashboard)
When you open the "Department Setup" page, you will see a clean and organized workspace.

### The Action Bar (Top of the page):
* **Search Bar:** Allows you to find a department by typing its Code or Name.
* **Company Filter:** A dropdown to filter departments by parent company.
* **Branch Filter:** A secondary dropdown to further filter by branch within the selected company.
* **Filter:** A dropdown to show either "Active" or "Inactive" departments.
* **Customize View:** A button that lets you choose which columns to show or hide on your screen.
* **Import Button:** Allows bulk creation of departments by uploading a pre-formatted CSV or Excel file. Useful for importing an organization chart during initial system setup.
* **Export Button:** Downloads the current filtered list to an Excel or CSV file for reporting or offline review.
* **Print Button:** Generates a print-ready view of the department list for hardcopy documentation.
* **Create New Department Button:** A prominent button to add a new department.

### The Data List (The main table - must be customizable):
You will see a list of all your created departments. The default columns displayed are:
* Department Code
* Department Name
* Company (Parent)
* Branch (If blank, indicates a company-wide Shared Service)
* Parent Department
* Department Head
* Status (A green badge for Active, gray for Inactive)

At the end of every row, there will be buttons to "Edit" or "View" the complete details of that department.

## What Information We Need to Capture (The Data Fields)
When you click "Create New Department" or "Edit," a form will open. The form is divided into logical sections to make data entry easy. Here is the exact data the system will save.

### Section 1: Department Identity
| Field Name | Description | Required? |
| :--- | :--- | :--- |
| Company | Dropdown linked to active companies. The parent entity that owns this department. | Yes |
| Branch | Dropdown linked to active branches under the selected company. Leave blank if the department applies to all branches (e.g., a company-wide Finance department). | Optional |
| Department Code | A short identifier (e.g., "ACCTG", "HR", "OPS"). Auto-uppercase. | Yes |
| Department Name | The full descriptive name (e.g., "Accounting", "Human Resources"). | Yes |
| Parent Department | Dropdown of existing departments under the same company. Used to build a department hierarchy (e.g., "Finance" is the parent of "Accounting"). | Optional |

### Section 2: Department Details & Approvals
| Field Name | Description | Required? |
| :--- | :--- | :--- |
| Department Head Name | The text name of the person leading this department. | Optional |
| Department Head User | Dropdown linking to the System Users. Crucial for automatically routing Purchase Orders and Leave Requests for approval. | Optional |
| Description | A brief description of the department's primary function and responsibilities. | Optional |

## The Supabase Database Architecture (For Lovable)
This module requires one main table. It supports self-referencing for parent-child department hierarchies and links to both `companies` and `branches`.

### Table 1: `departments` (The Main Table)
This table will store all the department records for the organization.

**Critical Database Rule for Lovable:** 
1. Ensure a composite unique constraint is placed on `company_id` and `department_code`. 
2. The `parent_department_id` must have a foreign key constraint referencing `id` in the same `departments` table.

| Column Name | Data Type | Rules | What it stores |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | System ID for the department. |
| `company_id` | UUID | Required, Foreign Key | Links to `companies.id`. |
| `branch_id` | UUID | Nullable, Foreign Key | Links to `branches.id`. Null means it applies to all branches. |
| `department_code` | Text | Required | Short identifier (e.g., "ACCTG"). |
| `department_name` | Text | Required | Full department name. |
| `parent_department_id` | UUID | Nullable, Foreign Key | Self-references `departments.id` for hierarchy. |
| `department_head_name` | Text | Nullable | Name of the department head. |
| `department_head_user_id`| UUID | Nullable, Foreign Key | Links to the system's `users` table for Approval Matrices. |
| `description` | Text | Nullable | Brief description of department function. |
| `is_active` | Boolean | Default: `true` | System status. |
| `created_by` | UUID | Required, Foreign Key | Links to `users.id`. User who created this record. |
| `updated_by` | UUID | Nullable, Foreign Key | Links to `users.id`. User who last edited this record. |
| `created_at` | Timestamp | Auto | Date record was created. |
| `updated_at` | Timestamp | Auto | Date record was last edited. |
| `updated_at` | Timestamp | Auto | Date record was last edited. |
