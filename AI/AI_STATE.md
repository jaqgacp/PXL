## Session 85 (current, shipped locally) — VAT/PT tax-code effective-date + version governance

Migration `20260713000012_tax_code_effective_date_governance.sql` (test 039, TAX-CODE-VERSION-001, 17 assertions) closes **PXL-DA-010**. It extends the ATC release pattern (`20260701000018` + `20260713000002`) to `tax_codes` (the VAT/PT rate holder) and `vat_codes` (classification/mapping): effective-from/to + deprecation/supersession columns; version-aware uniqueness `(code, effective_from)` replacing the global `tax_codes_code_key` (and `(vat_code, effective_from)` on vat_codes); `fn_enforce_tax_code_version_rules` (overlap + successor integrity); `fn_tax_code_used`/`fn_vat_code_used` (tax ledger + the six document-line tables — sales_invoice_lines, credit_memo_lines, debit_memo_lines, vendor_bill_lines, cash_purchase_lines, vendor_credit_lines); `fn_guard_tax_code_history`/`fn_guard_vat_code_history` freezing rate/identity/effective-start once used and blocking deletes of used codes; and `fn_tax_code_version_asof`/`fn_tax_code_is_current` resolvers. No heavy VAT/PT posting RPC was modified: because a used version's rate is now immutable and a statutory rate change creates a NEW successor version (a new vat_code the item/document then selects), every already posted line keeps resolving the frozen rate on the version it referenced — historical VAT/PT reports stay stable. Optional admin successor-management UI on `TaxSetupPage` is a backlog enhancement, not a defect. ## Session 77 (current, shipped locally) — ATC document-date validation + rate versioning

Trusted migration `20260713000002_atc_document_date_versioning.sql` (test 033, ATC-DOCDATE-VERSION-001, 15 assertions) closes **PXL-AUD-035** (TAX-001) and **PXL-AUD-036** (TAX-002), replacing the held-out draft `20260710000004` (which stays excluded). AUD-035: `fn_validate_payment_voucher_line_ewt` and `fn_validate_receipt_line_cwt` each collapse to one signature with a trailing `p_document_date DATE` and evaluate the ATC effective window as of the document date; every caller threads the document date (PV save/line-trigger/ready, OR save/line-trigger/ready, cash sale via the receipt-line trigger, CV header trigger + post). The prior 4-/6-arg overloads are dropped so un-updated call sites bind to the single new function and fall back to CURRENT_DATE. AUD-036: global `UNIQUE (code)` replaced by version-aware `uq_atc_code_version (code, tax_category, effective_from)`; `fn_enforce_atc_version_rules` rejects overlapping active windows and validates successor linkage; `fn_atc_version_asof(code, category, as_of)` resolves the in-force version; `fn_guard_atc_code_history` also freezes `effective_from` once used. Master-data default triggers intentionally keep CURRENT_DATE. Frontend ATC pickers still resolve by row (not yet by document date through `fn_atc_version_asof`) — a UI enhancement, not a posting defect.

Migration `20260713000003_settlement_total_line_authority.sql` (test 034, SETTLEMENT-TOTAL-AUTHORITY-001, 8 assertions) then closed **PXL-AUD-038** and **PXL-AUD-048**: `fn_save_payment_voucher`/`fn_save_receipt` derive header `total_amount`/`total_ewt`/`total_cwt` from the persisted lines (client header ignored), so GL cash/AP/AR mirror the subledger and tax ledger; `fn_validate_payment_voucher_ewt_ready`/`fn_validate_receipt_cwt_ready` also reject any stored header cash total diverging from `SUM(line payment_amount)` before a JE is written. Because header withholding totals now equal the line sums exactly, the 0.02 tolerance can no longer leave a residual GL-to-ledger variance blocking SAWT/QAP/DAT exports. Built on the `20260713000002` bodies (document-date threading preserved). Cash sales are unaffected (own JE, separate path). `receipts.total_amount` semantics split for cash sales remains **AUD-046** (open).

Migration `20260713000004_cm_vc_aware_overapply_guards.sql` (test 035, CM-VC-OVERAPPLY-001, 6 assertions) then closed **PXL-AUD-039**: the `fn_save_receipt`/`fn_save_payment_voucher` over-apply guards now net applied credit memos (AR) and non-reversed vendor-credit applications on open/applied vendor credits (AP) from the invoice/bill outstanding, using the same formula as `fn_ar_aging_asof`/`fn_ap_aging_asof` (scalar subqueries so the two credit sources do not fan out).
